    2. GitHub Copilot App とは（提供概要）

    GitHub Copilot アプリは、エージェント駆動の開発をワークフローに直接持ち込むネイティブなデスクトップ体験。Issue や Pull Request という実際の仕事からセッションを開始し、それらを並列に走らせ、GitHub コンテキストを内蔵した1つのワークスペースで管理できる。

    • 入手先: gh.io/app（または github.com/features/ai/githubapp）からダウンロード。
    • 対象: 有料の GitHub Copilot サブスクリプション（Pro / Pro Plus / Max / Business / Enterprise）を持っていれば利用可能。
    • CLI との連携: GitHub Copilot CLI を使っていれば、そのセッションが自動でアプリにも表示され、両者を行き来できる。
    • 新モデル: 配信当日から MAI Code One Flash モデルのアプリ・CLI へのロールアウトを開始（個人サブスクライバー向け）。
    「もしまだ持っていないなら、gh.io/app に行ってダウンロードしてほしい。有料の Copilot サブスクがあれば使える」（Pierce）

    3. AI パートナーシップの進化と「新しい課題」

    Pierce は、自身が AI 領域で働き始めた 2024 年当時を振り返る。当時はエージェントがなく、Copilot edits やゴーストテキストがある程度で、Next Edit Suggestions すらまだ無かった。その後エージェントが登場し（GPT-4.1 世代）、モデルの能力向上とともに人間の関わり方も大きく変わった。

    「retain code from agent」という指標

    GitHub が追跡している指標に retain code from agent（エージェントが生成したコードのうち、実際にコミットへ採用される割合＝人がエージェントをどれだけ信頼しているかの尺度）がある。

    0%→約50%
    GPT-4.1 世代の retain code（それ以前は 0%）
    約90〜95%
    最新 Opus / GPT 世代の retain code
    1タスク → 複数
    並列実行へと変わった働き方

    協働スタイルの変化と、そこから生まれた課題

    エージェント登場初期は、モデルがまだ万全でなかったため「チャットを凝視し、すべての diff を承認し、一つひとつレビューする」関わり方だった。しかしモデル能力の向上とともに信頼が高まり、多くの人が「まず走らせる（必要なら plan だけ立てて autopilot に投げる）」スタイルへ移行。すると次の誘惑「複数タスクを同時に走らせたい」が生まれ、同時にいくつもの新しい課題が浮上した。

    • 5つのことを同時進行させると頭が混乱する（高速に出力できても認知負荷が上がる）。
    • これまでは「1度に1タスク」だったのが、常に「次に何を拾うか」を判断し続ける必要が出てくる。
    • コーディングエージェントが生成したコードを、実際にコードベースへ着地（マージ）させるという別問題が残る。

    こうした「AI との協働の変化」が生んだ問題を解くための新しいツールとして生まれたのが GitHub Copilot アプリ、という位置づけになる。

    flowchart TB A["2024年: エージェント以前
    ghost text / Copilot edits"] --> B["エージェント登場
    GPT-4.1 世代・retain code 約50%"] B --> C["全 diff を逐一承認
    人間がすべてレビュー"] C --> D["モデル進化

    4. アプリ全体ツアー（ホーム画面・モード・プロジェクト）

    Pierce はまずホーム画面から紹介する。ホーム画面は「あらゆるプロンプトを起動するホームベース」。

    3つのモード

    モード使いどころ
    Planまず計画を立てたいとき
    Interactiveモデルとより同期的（対話的）に協働したいとき

    モデルピッカーとプロジェクト

    5. My Work・Automations・Quick Chats

    My Work ― GitHub コンテキストに根ざした起点

    市場には並列タスクを走らせる「セッションマネージャー」が数多くあるが、GitHub Copilot アプリの強みは GitHub コンテキストに深く根ざしていること。Pierce は1日の始まりを「My Work」から始める。

    • 自分にアサインされた Issue が見える。独自ビュー（例: 自分のアサイン一覧）を作成でき、プリセットのフィルタもある。
    • ここで初期トリアージを行い、Issue をクリックしてその場でセッションを起動（例: VS Code リポジトリに対して新規セッション）。

    Automations ― 繰り返しの雑務を自動化

    「Automations」は、日々の反復的なトイル（toil）を自動化する仕組み。すべてはプロンプトであり、MCP・スキル・プロジェクトコンテキストにアクセスでき、ローカルでもクラウドでも実行できる。Pierce の例:

    自動化の例内容
    Issue トリアージGitHub App リポジトリのホットな Issue を巡回してトリアージ
    テレメトリ分析Azure MCP 経由で Azure Data Explorer にクエリ（turn analysis、onboarding analysis、KPI のビジネスレポート など）
    Today会議（Work IQ 経由）・アサインされた Issue・受信メッセージなどから「今日フォーカスすべきこと」を準備
    エンタープライズフィードバックアカウントチームからの企業向けフィードバックをトリアージ
    メッセージ要約Slack / Teams のメッセージを要約（トークン消費が大きいので手動実行）

    Cassidy の上司 Ashley Willis は「常時40個ほどの Automations」を走らせているといい、使い方は人によって大きく異なる。

    Quick Chats ― コーディング以外の活用

    特定のプロジェクトにスコープされない用途は「Quick Chats」で。Pierce はアプリをコーディング以外にも「何でも」使う。例として、GitHub Copilot アプリのリポジトリと github.com 上のドキュメント／マーケティングページをもとに、社内セールスチーム向けの PowerPoint プレゼンを実際に生成していた。

    6. Chief of Staff モデル（1セッションが複数を駆動）

    「チャットが増えすぎて管理に疲れる」という共通の課題に対し、GitHub Copilot アプリの目玉のひとつが 1つのセッションで他の5つのセッションを駆動する使い方（Pierce は "chief of staff" / "CEO/ chief of staff model" と呼ぶ）。人間は高レベルの関わりに留まれる。

    デモでは「前日のローンチで浮上した上位5件の Issue」に対して、次のように指示した。

    「これらの Issue それぞれについて、plan モード・GPT-5.5・high reasoning effort で5つのセッションを起動して」（音声モードで発話）
    • 音声モード: 設定でローカルモデルをダウンロードして使用。音声はマシン外に出ない。Mac では option＋スペースで発話（他 OS でも動作）。
    • 「アプリがアプリを駆動する」形で5つのセッションが生成され、各々が Git work tree で隔離実行される。
    • 駆動元セッションに「plan がレビュー可能になったら報告して」と頼めば、他セッションの進捗を監視・報告してくれる（入れ子のツリー表示）。
    • ノイズが不要なら折りたたんでおけばよく、セッション完了時には通知音が鳴る。会議中でも音で気づける。
    sequenceDiagram participant U as 開発者 participant O as オーケストレーター
    セッション participant S as 子セッション 1〜5
    （Git work tree で分離） U->>O: 上位5件の Issue を
    plan / GPT-5.5 / high で起動して（音声） O->>S: 5つのセッションを生成・並列実行 S-->>O: 各セッションの進捗 U->>O: plan が完成したら報告して O-->>U: 完了通知＋レビュー依頼（＋通知音）

    7. レビューの再定義と Agent Merge

    アウトカム重視のレビュー

    Pierce は「AI との協働で変わったこと」として、まず diff をレビューしないスタイルを挙げる。コードを見ないわけではないが、第一のレビュー対象は diff ではなく成果（アウトカム）だという。

    「まず知りたいのは『で、ちゃんとやったの?（did you do the thing?）』ということ」（Pierce）
    • 1クリック実行: run スクリプトを設定していればアプリからそのままアプリを起動できる。ブラウザも使える（例: coloring book アプリを起動して確認）。
    • Pick and polish: 実行中のアプリ上で要素を指定し「なぜ 2026 年なのに 2024 と表示される?」のように成果物に対して直接フィードバックできる。
    • diff へのコメント: Word 文書へのコメントのように diff 上をクリックしてコメントし、そのまま Copilot に送って修正作業を開始させられる。
    • VS Code 連携: ワンクリックで同じセッションを VS Code で開ける。しかもチャットの継続性があり、アプリでのやり取りが VS Code 側にも流れてくる。CLI・アプリ・VS Code が相互に連携する。

    PR レビューのボトルネックと Agent Merge

    AI は「作業を大量に始めること」を促すが、終わらせること（＝ PR レビュー）はもともとチームのボトルネック。しかも Pierce のようなエンジニアではない PM までコードベースに貢献するようになり、PR はかつてないほど増える。

    アプリは GitHub コンテキストに深く根ざしているため、PR を開いて GitHub Actions の状況やパフォーマンス上の指摘を確認し、各コメントを個別に Copilot に修正させることができる。CI 失敗があれば「直す?」ボタンが出る。しかし「CI が落ちた」「レビューコメントに対応して」と人間がボタンを押し続けるのは新たなお守り作業になってしまう。

    これを解くのが Agent Merge。有効化すると、その PR を継続的にプルし、レビューコメント・CI 失敗・ブロッキングアクションを常時チェックして先回りで対処する。

    • 人間／Copilot いずれのレビューコメントに対しても、妥当でないと判断すれば盲従せず反論する。
    • CI 失敗も「flaky なので再実行すればよい」のか「本当の不具合」なのかを見て対処する。
    • マージされるまでプルし続ける（このリポジトリでは最終的な承認だけは人間が行う設定）。

    PR を継続プル"} C --> D["レビューコメント確認"] C --> E["CI 失敗の確認"] C --> F["ブロッキング
    アクション確認"] D --> G["対応 / 妥当でなければ反論"] E --> G F --> G G --> C C --> H["マージされるまで継続
    （最終承認は人間）"]

    8. Canvases（エージェントネイティブ UI）

    「チャットが増えすぎて頭で管理しきれない」問題への探求として、アプリには Canvases という概念がある。Canvas とは、エージェントが生成する UI であり、その UI がそのままアプリを操作（駆動）できるもの。

    • デモでは「GitHub Copilot triage board canvas を開いて」と指示してトリアージボードを生成。ボード上のカード（例: サブエージェントの hang 問題）から「start session」を押すと、その UI から実際にセッションが起動し、アプリが駆動される。
    • Cassidy が Microsoft Build のキーノートで見せた「PR レビュー用 Canvas」も例に挙がる。サムズアップ／ダウンで、アップなら PR 承認＆マージのコメントをセッションへ返し、ダウンなら何もしない、という UI。
    • 用途は自由。ローンチ直前に Pierce が作った「launch board」は、GitHub Issue だけでなく社内 Slack / Teams のフィードバックも1か所に集約していた。
    • Trello ボードのような Canvas も作れ、カードを列から列へドラッグすると、実際にアプリ側の作業が前進する。
    • フルスクリーンにして「1日中その中で暮らす」使い方も可能。組み込みで markdown・browser があり、さらに数種が近日追加予定。
    「Canvas は、チャットの先にある『エージェントネイティブな UI』がどんなものかを探る良い試み。チャットを頭の中で管理し続けることから解放してくれる」（Pierce）

    9. クラウドセッションとクロスデバイス（Enrique デモ）

    Enrique は「セッションや Automation を走らせたまま夕食に行きたいとき、ノート PC を閉じてよいのか?」という日常の悩みから始める。これを解くのが GA でシップした クラウド機能。

    • クラウドセッションはエージェントループをクラウド上で実行するため、ノート PC を閉じても・別デバイスに移っても動き続ける（Web を別 PC で開く、ソファでスマホを使う、など）。
    • デモ題材は「is it Friday?」アプリ。金曜でないのに「Friday」と表示するバグを、クラウドセッションで「金曜なら CSS で紙吹雪をアニメーション」と指示して修正。
    • クラウドセッション接続時、上部に 「Open on remote」リンクと QR コードが表示される。QR を GitHub スマホアプリでスキャンすればスマホから、Open on remote なら github.com からセッションを継続できる。
    • 別途走らせていた「ダークモード追加」のクラウドセッションを github.com 側で継続し、「新しいブランチを作ってコミットし PR を送って」と指示するデモも（配信中は一部フレーキーな挙動あり）。

    Automations でトリガー実行 ＋ 今後の予定

    Enrique は Automation のデモも実施。リポジトリに Issue を作成すると、「issue created で常に issue を直す」Automation がそれを検知して着手。これらのクラウド Automation は github.com 側（agents → automations）からも確認できる。生成される PR はドラフトモードで作られ、レビュー・完了・マージへ進める。

    今後の拡張として2点が予告された。

    • より多くのトリガー種別のサポート。
    • 時間ベーストリガーの柔軟化。現在は daily / hourly / weekly のみだが、曜日・時刻・複数回、さらに cron 式による指定まで可能にする。「数日中」レベルで近い。

    URL ペーストで即レビュー（Cmd+K）

    Enrique が毎日使う機能として、PR / Issue の URL を Cmd+K（Windows は Ctrl+K）でペーストすると、そのブランチをローカルに引いて PR に紐づいた新規ローカルセッションを作れる。

    • Issue でも PR でも動作し、自分が所有していないリポジトリでもよい（OSS のレビュー依頼などに便利）。
    • Burke は「クローンせずにリポジトリで作業を始められる」点を特に評価。クローンの手間（フォルダ作成・移動・クローン・ツールを開く…）が抽象化され、「やりたいなら、ただ始めればいい」体験になる。クラウドならローカルに載せる必要すらない。

    10. 信頼と YOLO モード

    「allow all（全許可）＝ yellow / YOLO モード」をどこまで使うかについて、4人がそれぞれの立場を語った。

    人物スタンス
    Enriqueアプリ開発を毎日フル YOLO。問題が起きたことは一度もない（保証はしないが）。
    Pierce基本フル YOLO。ただしインフラを変更しうるツール（Azure MCP など）を渡すときだけ慎重。コードは Git で復元できるので、危険なツールを与えなければ怖いことは基本起きない。
    Cassidyまだ YOLO は怖い。ただしモデルとガードレールの向上で信頼は高まりつつある。「正しく行われているか確認したい」というコントロール欲求がある。
    Burke企業は「allow all」に慎重で当然。しかし YOLO / yellow モードなしでは AI 開発は「まったく楽しくない」。AI で開発するなら自律的に動けることが必要。
    • 「allow all」を押し続けると、人はプロンプトを読まなくなる（「本番 DB を消してよい?」→「はい、許可」になりかねない）。VS Code でも見られた人間の性。
    • だからこそクラウドセッションはサンドボックス化されていて安全。クレジットカードデータを扱う企業の CIO に「エージェントを信じて」と言うのは難しいが、クラウドなら外部に害を及ぼさない。
    • Pierce いわく、複数タスクを同時に走らせて自分を解放するには、常時監視を諦めてある程度 YOLO を受け入れる必要がある。
    • Pierce が過去のクラウドエージェントで感じていた不満は「関与しなさすぎ（issue を割り当てて翌日見に行くだけ）」だったが、今回のクラウドセッションは結果がすぐストリーミングで返り、ローカル並みに速く、ループの中に居られる点が大きな解放。加えてデバイス横断・ローカルをサーバー化しない・ローカルのディスクを work tree で埋めない、といった利点がある。

    12. Microsoft Build の舞台裏とエンドツーエンドまとめ

    Microsoft Build の舞台裏

    • オープニングキーノートは Cassidy が登壇、Burke が裏で各種オペレーションを担当、Pierce も会場に。リハーサル中にアプリを使いながら Pierce が「ここ直せるね」とその場で修正を入れ、驚異的な速さでキーノートを磨き上げた。
    • GitHub Copilot App チームはキーノート当日にリリースをシップ。Cassidy のサムズアップ／ダウンのデモ案も、その日のうちにアプリへ実装された。

    エンドツーエンドのワークフロー

    Pierce がまとめた「どの作業を拾い、どう作り、どう着地させるか」の全体像。

    flowchart TB A["My Work / Automations で
    1日の作業を把握"] --> B["Issue からセッション起動"] B --> C["Plan モードで計画づくり"] C --> D["Autopilot で実行"] D --> E["アウトカム重視レビュー
    1クリック実行 / pick and polish"] E --> F["必要なら diff にコメント
    VS Code で継続も可"] F --> G["PR 作成"] G --> H["Agent Merge を有効化"] H --> I["コードベースに着地"] C -.高レベル運用.-> J["Chief of Staff
    1セッションが複数を駆動"] J -.-> D

    これに加えて、繰り返しのトイルを担う Automations、コード以外の用途（テレメトリ分析・フィードバック分析・PowerPoint 作成・Slack/Teams メッセージ要約）、そしてチャット管理から解放する Canvases が、アプリを「開発のホームベース」たらしめている。

    クロージング

    「自分の仕事をアプリから出ずに全部できるようにする」ことが開発チームの目標。まだ足りない部分もあるが、要望した機能が「すでにある／すぐ来る」ほどの速さで進化しているのが印象的だった、と4人は締めくくる。アプリは gh.io/app（github.com/features/ai/githubapp）から、GitHub Copilot サブスクリプションで利用できる。
