    2. ソフトウェアは「LEGO ブロックの組み立て」

    1 年前の最初の AI Dev で示した比喩を再掲。ソフトウェア、ツール、フレームワーク、API は LEGO ブロックのようなもので、種類が増えるほど組み合わせが 指数的（組合せ的） に爆発し、面白いものが作れる。

    flowchart LR W["白ブロックのみ"] --> X["作れる物 少"] M["色・形が増える"] --> Y["組合せ 指数爆発"] Y --> Z["AI エージェントが
    高速に組み立て"]

    この 1 年で AI コーディングエージェントが急速に進化し、ブロックを素早く組み合わせて魅力的なソフトを作れるようになった。ブロックには 2 種類ある。

    区分例
    AI ブロックプロンプト可能な LLM、RAG、エージェント的ワークフロー など
    非 AI ブロックUI コンポーネント、DB（永続化）、認証/認可マネージャ など

    AI コーディングのおかげで、両方のブロックが過去に例のない速度で毎日増えている。AI エージェントが自分の代わりにそれらを組み合わせてくれる、という点が特に強力。

    3. AI コーディング 80% と 100% は別物

    スタートアップ、アカデミア、大企業のチームと話すと「AI コーディングをどれだけ使うべきか」が必ず議論になる。よくあるアンケート：「あなたのコードのうち AI が書く割合は？」。

    AI 比率典型コメント体感スループット
    0%「全部自分で書く」―（推奨せず、徐々に引き上げを）
    80%「うちは AI で 80%」残り 20% を人が書く／レビューする時間が支配的で、体感は思ったほど速くない
    ほぼ 100%Andrew Ng 本人。フロンティアチームの傾向同じ時間で圧倒的に多くを片付ける
    100% AI コーディングのチームは、短時間で大量を片付ける。残り「最後の 20%」を人がやるかどうかで体感は大きく違う。

    4. コードレビューもエージェントに任せる流れ

    「AI に書かせるが、全行を人間がレビューする」という運用をしているチームは多い。だが Andrew Ng と彼のチームは、それをやると 自分自身がボトルネック になると感じている。

    ただし宗教的に「人間が書くな」とは言っていない。例外も明確にある。

    NASA で宇宙船用の 50 行を書くなら手で書け。それで構わない。

    とはいえフロンティア企業は「ほぼ 100% AI 記述」へ寄っており、書く側もレビュー側も、80% 止まりとは別世界の加速がある。

    5. 第二次効果 ― PM ボトルネックの出現

    コーディングが AI に置き換わると、興味深い 二次的効果 が生まれる。昨年 7 月の The Batch ニュースレターで「プロダクトマネジメント・ボトルネック」として書いた内容。

    flowchart LR A["アイデア"] --> B["プロトタイプ作成
    (エンジニアリング)"] B --> C["ユーザに当てる
    フィードバック取得 (PM)"] C --> D["改善 (エンジニアリング)"] D --> C style B fill:#f97316,color:#fff style D fill:#f97316,color:#fff

    この図の「B：プロトタイプ作成」が AI コーディングで 10〜100 倍速くなった結果、新しいボトルネックは「何を作るか」を決める PM の仕事 側に移った。

    PM:エンジニア比率の崩壊

    1 : 8
    従来の典型(PM:エンジニア)
    1 : 2
    最近のチームでよく見る比率
    1 : 1
    さらに進んだチーム
    1 人
    究極形：両役割を 1 人に統合

    究極的には「PM 1 人 + エンジニア 1 人」よりも、その 2 人を 1 人に潰して両方やる方が速い。プロダクトを形にできるエンジニア、あるいはコードが書ける PM が爆速で動ける。速さ自体が大きな競争優位になる。

    6. 小さなジェネラリスト・チームへの収束

    もちろん「1 人チーム」で全てができるわけではない。AI ネイティブな組織で増えているのは 少人数のジェネラリスト・チーム。

    エンジニア

    PM 仕事をいくらかこなす。

    PM

    エンジニアリングをいくらかこなす。

    全員

    ほぼ全領域を把握。小チームのまま高速に動ける。

    シリコンバレーの一部では「AI による雇用大崩壊」が流行りの話題だが、Andrew Ng はそれを実際には見ていない。むしろ このプロフィールのエンジニアの未来は明るい。

    7. AI エンジニア採用で見ているもの

    Andrew Ng のチームが AI エンジニアを採るときに重視する観点。

    #観点内容
    1コーディングエージェントを実効的に使えるClaude Code、Gemini の AI、Codex、open code など何でもよいから使いこなせる
    2最新のビルディングブロックを把握している数が多く変化も速いので把握自体が難題。だから差がつく
    3ジェネラリスト・スキル基本的なプロダクトマネジメントなど、領域横断の力

    まとめると「どう作るか」と「何を作るか」の両方ができる人。このミックスを持つエンジニアへの需要は巨大で、自分のチームでも十分には見つけられていない。

    8. デザイン・法務・マーケ・営業のボトルネック

    話を PM ボトルネックの先へもう一歩進める。ソフトウェアエンジニアリングが 10〜100 倍速くなると、他のあらゆる工程が比較で遅く見える。

    ボトルネック症状対処の方向
    デザインデザイナー → エンジニアの受け渡しが律速デザイナーが Figma を介さずコードで実装する形まで進めるとさらに速い
    法務・コンプライアンス1 日で書いたコードに対して 1 か月の法務待ちAI で下書きを作り、本物の弁護士にローンチ前サインオフ
    マーケティングシップが速すぎて、何が出たかをマーケが追えないエンジニアが AI で「基本的なマーケ作業」をある程度カバー
    営業 他同様に追いつけない少人数ジェネラリスト・チームで巻き取る

    救いになるのが 少人数ジェネラリスト・チーム。マーケの専門家でないエンジニアでも、AI を上手にプロンプトできれば「ベース部分のマーケ作業」はこなせる。AI 単独だと変なマーケ判断もするが、人が回せばかなり通用する。

    2 人チームでソフトウェア・プロダクト・デザイン・法務・マーケのすべてが必要なら、その 2 人は定義上全領域に何らかのスキルを持つことになる。AI ネイティブな賢いチームは、こうした他職能を AI で補えるので非常に速く動く。

    Andrew Ng 本人の運用例：法務に出す前段のドラフトは AI に作らせ、ローンチ前に本物の弁護士にサインオフしてもらう。「弁護士になれ」とは誰にも言っていないが、AI を「下書き屋」として使うのは現実的。

    補足： 時間の都合で詳細は省略されたが、「複数の AI ネイティブ・チームをどう協働させるか」も触れられた。10 人で全部はできないが、100 人組織を「コミュニケーションを絞り、明確な API 境界を持つ AI ネイティブ小チームの集合」として設計すると、単一チームを超えてスケールできる。

    9. 「AI 雇用大崩壊」は起きていない

    ソフトウェアを作るために必要な仕事のうち、コードを書く作業は一部にすぎない。それが急速に AI 化されているのは事実だが、AI による全面的な雇用大崩壊は近い将来には起きていない、と Andrew Ng は明言する。

    主張根拠／引用
    大衆紙とビジネスメディアの精度差大衆紙は刺激的なストーリーを語りがちだが、ビジネスメディアは「正確であること」で対価を得ているため精度が高い
    論調の変化ここ 1〜2 か月、ビジネスメディアは「AI 雇用大崩壊は来ていない／後ろ倒し」というトーン
    連邦準備銀行の調査フィラデルフィア連邦準備銀行による調査：人員削減を必要とした CEO はごく少数

    ただし、雇用不安を感じている人が多いこと自体は否定しない。それには対処が必要であり、コロナ期の過剰採用やゼロ金利時代の採用過多からの揺り戻しでレイオフされた人が多いことも理解している。

    雇用環境が完璧だとは言わない。良くない面も多い。ただ、一部で語られているような「雇用大崩壊」は起きていない。

    だからこそ、私たち全員が「ものを作る力」をさらに高めることに投資し続けたい。

    10. 並行スキル開発というフレームワーク

    私たち一人ひとりが「より優れたビルダー」になっていく上で、もっとも難しく、もっとも重要なのが ビルディングブロックの習熟。ここで Andrew Ng は Parallel Skill Development（並行スキル開発） というフレームを提示する。

    flowchart TB subgraph Agent["エージェント側スキル"] A1["コーディングエージェントの
    高機能化"] A2["Agent Skills (Anthropic 等)"] end subgraph Human["人間側スキル"] H1["教育・トレーニング"] H2["エージェントを
    正しく駆動する補完スキル"] end Agent -. パートナー .-> Human Human -. パートナー .-> Agent Agent --> Build["新しいものを共同で作る"] Human --> Build

    エージェントが進化するなら、人間側はそれを 適切に駆動する補完スキル を身につけ、両者を組ませて新しいものを作る、という考え方。残り時間で、両軸に対応する具体プロダクトを 2 つ紹介する。

    • エージェント側を助けるもの → Context Hub
    • 人間側を助けるもの → 終盤で発表する Code Dream

    11. Context Hub ― エージェントに最新情報を渡す

    ビルディングブロックがあまりに速く増えているため、コーディングエージェントが持つ 知識カットオフ（1 年前、ときに 1 か月前）が問題になる。

    • 古い／非推奨 API を使ってしまう
    • 最新ツールを知らない
    • 結果としてハルシネーション（嘘の API 呼び出し）が起きる

    具体例：OpenAI API の呼び方

    現時点でも Auto Code に OpenAI API を呼ばせると、旧来の Chat Completions API を選び、古いモデルを叩こうとする。インターネット上の学習データの大半が旧 API ベースだから。

    素のエージェントContext Hub あり
    API旧 Chat Completions（非推奨）新 Responses API
    モデル古いモデルを選択最新（GPT‑5.5 等）まで認識
    ハルシネーション頻発大幅に減少

    仕組み

    Context Hub は Vivek Prasad、Sanyam Hota、Andrew Ng の 3 名で作ったツール。役割は AI エージェントに 最新ドキュメント を渡すこと。

    sequenceDiagram participant U as 開発者 participant A as コーディングエージェント participant H as Context Hub participant D as OpenAI ドキュメント U->>A: 「OpenAI を呼ぶコードを書いて」 A->>H: 最新ドキュメントを要求 H->>D: 取得 D-->>H: GPT-5.5 / Responses API 仕様 H-->>A: 最新仕様を反映したマークダウン A-->>U: 新 Responses API を使う正しいコード

    実演イメージ：

    • chop search openai でパッケージ一覧が返る
    • chop get openai/chat 等で OpenAI の最新ドキュメントを取得（GPT‑5.5、Responses API 等にも対応）
    • 出力は「600 行くらいのマークダウン」になることもあり、人が読むには長いが、本来の用途は コーディングエージェントに食わせる こと（人も読める）

    Andrew Ng 自身も、OpenAI、Gemini、各種データベース系サービスの呼び出しでエージェントがハルシネーションする経験は数多くしている。最新コンテキストを差し込めるかどうか が体験を大きく変える、というのが要点。

    12. Code Dream 発表 ― 会話で学ぶ AI コーディング
