    Loop Engineering（ループエンジニアリング）— AIコーディングの新しいメタ

    「エージェントにプロンプトを打つ人」をやめ、「エージェントにプロンプトを打つループ（システム）を設計する」へ。Addy Osmani、Cobus Greyling、MindStudio、LangChain（Sydney Runkle）、Matt Van Horn の5記事を横断し、Loop Engineering の定義・構成要素・実行基盤・落とし穴を1本に統合。

    3 行サマリ

    • Loop Engineering とは「自分自身（プロンプトを打つ人）を置き換える」こと。エージェントに1ターンずつ指示する代わりに、仕事を見つけ・割り当て・検証し・記録し・次を決める「ループ（小さな制御システム）」を一度だけ設計し、ループにエージェントを突かせる。Peter Steinberger・Boris Cherny（Anthropic）らが提唱し、2026年6月の AI コーディング界で最も語られた概念。
    • ループは「reason → act → observe → repeat」（ReAct）が核。明確なゴール（停止条件）、ツール群、コンテキスト管理、終了ロジック、エラー処理が必要。実務では Automation（心拍）・Worktree（並列分離）・Skill（知識）・Connector/MCP（連携）・Sub-agent（作る人と検証する人の分離）＋外部メモリの6要素で構成される（Addy Osmani）。LangChain は Agent → 検証 → イベント駆動 → Hill Climbing の4階層で整理。
    • ループは万能ではない。検証・理解・判断は依然あなたの責任。無人で動くループは無人でミスもする。コスト（トークン）は急増し、comprehension debt（理解の負債）と cognitive surrender（思考の放棄）のリスクが高まる。「press go する人」ではなく「エンジニアであり続ける人」としてループを設計せよ。

    1. Loop Engineering とは何か — プロンプトからループへ

    Addy Osmani は端的にこう定義する。

    「Loop engineering とは、エージェントにプロンプトを打つ人としての自分自身を置き換えること。代わりに、それを行うシステムを設計する。ここでのループは再帰的なゴール（recursive goal）——目的を定義すれば AI が完了まで反復する——と考えられる。」

    この転換のきっかけとなった発言が、AI コーディング界で繰り返し引用された。

    • Peter Steinberger:「もうコーディングエージェントにプロンプトを打つべきではない。エージェントにプロンプトを打つループを設計すべきだ。」
    • Boris Cherny（Anthropic / Claude Code 責任者）:「私はもう Claude にプロンプトを打たない。Claude にプロンプトを打ち、何をすべきか考えるループを走らせている。私の仕事はループを書くことだ。」

    Matt Van Horn の記事「WTF Is a Loop? Peter Steinberger vs. Boris Cherny」は、この現象をこう皮肉る。「今週の AI コーディングで最も繰り返された一文はわずか6語だが、それを口にする人のほとんどが定義できない」。つまり「ループを設計せよ」という言葉だけが先行し、その実体が曖昧なまま広まった——本まとめはその実体を5記事から再構成する。

    従来との違い。過去2年は「良いプロンプトを書き、十分なコンテキストを与える」→「返答を読む」→「次を打つ」の繰り返しで、エージェントは道具、あなたはそれを常に手で握っていた。Loop Engineering ではその握る手をシステムに置き換える。Osmani の言う harness engineering（1体のエージェントが動く環境を作る）の一つ上の階に位置し、ハーネスがタイマーで動き、小さなヘルパーを生成し、自分自身に餌を与える状態を指す。

    2. ループの核 — reason → act → observe → repeat（ReAct）

    MindStudio の記事は、ループの本質を最もシンプルに説明する。ループとは、モデルが行動し、環境からフィードバックを受け、それを使って次の一手を決める、終了条件が満たされるまで繰り返す循環である。これは「A→B→C」と直線的に進むチェーン（chain）とは根本的に異なる。ループは動的で、B が失敗したら修正して再試行し、その後で初めて先に進む。

    ReAct パターン — ループ工学の源流

    現代のエージェントループの多くは、Princeton と Google の研究で提唱された ReAct（Reason + Act）パターンに遡る。推論ステップと行動ステップを交互に織り込む考え方で、コード生成では次のようになる。

    flowchart LR A["① ゴールを理解"] --> B["② コードを書く"] B --> C["③ 実行して出力/エラーを観察"] C --> D["④ 何が悪かったか推論"] D --> E["⑤ 修正して再実行"] E --> F{"テスト通過 / 完了?"} F -->|No| C F -->|Yes| G["完了"]

    LangChain も同じ核を「エージェントとは、タスク完了までループの中でツールを呼ぶモデルにすぎない」と表現する。なぜコーディングで特に重要か——コーディングは本質的に反復的で、熟練エンジニアでも一発で完璧には書かず、実行し、エラーを見て、直し、また実行する。この循環を飛ばして一度きり生成する AI は、実行時エラーを捉えられず、環境固有の問題に適応できず、生成物が実際に動くか検証できない。ループはこのフィードバックギャップを閉じる。

    LangChain（Sydney Runkle）が引く Swyx の言葉では、これは 「loopcraft（ループを積み重ねる技芸）」とも呼ばれる。

    3. よく設計されたループの解剖（MindStudio）

    すべてのループが等価ではない。設計が悪いとトークンを浪費し、永久に走り、進捗を幻覚する。良いループは効率的で、正しく終了し、信頼できる出力を生む。MindStudio は5つの構成要素を挙げる。

    要素内容
    ① 明確なゴール / タスク定義「完了」がどう見えるかをループが知る必要がある。評価可能なほど具体的で、テスト可能なサブタスクに分解され、過剰に手を広げない範囲に。「アプリを良くする」は無限ループを生み、「全ユニットテストを通す」は本当の終了条件になる。
    ② エージェントが使えるツール群コード実行（stdout/stderr）、ファイルシステム、ターミナル、検索・ドキュメント参照、テストランナー。ツールの質がループの有効性を直接決める。自分のコードを実行できなければ、ループはただの当て推量。
    ③ コンテキスト管理反復ごとにコンテキスト（書いたコード・遭遇したエラー・下した判断）が増える。過去反復の要約、試行と結果の構造化ログ、不要コンテキストの剪定で、トークン上限と「試したことを忘れる」を防ぐ。
    ④ 終了ロジック成功条件（テスト通過・期待一致・ユーザー承認）、失敗条件（最大反復到達・進捗なき反復エラー・ツール呼び出し失敗）、エスカレーション経路（人間や別エージェントへ引き継ぎ）。停止条件は後付けでなく一級の設計要件。
    ⑤ エラー処理と回復回復可能なエラー（構文ミス・import 漏れ）と硬いブロッカー（認証情報欠如）を区別し、エラー種別に応じて戦略を変え、同じ失敗の無限反復を避ける。同じエラー後に同じ行動を繰り返すのは学習ではなく「空回り」。

    4. ループの構成要素（Addy Osmani）— 5つの部品＋メモリ

    Osmani は、ループに必要なのは「5つの部品＋記憶する場所1つ」だと言う。驚くべきは、これがもはやツール固有の話ではなく、製品（Codex アプリ／Claude Code）に同じ形で出荷されていること。「どちらのツールか」を議論するのをやめ、どちらに座っていても機能するループを設計できる。

    部品役割Codex アプリClaude Code
    Automation（自動化）スケジュールで発見・トリアージAutomations タブ（project/prompt/cadence/environment）、結果は Triage inbox、/goalscheduled tasks・cron、/loop・/goal、hooks、GitHub Actions
    Worktree（作業ツリー）並列エージェントの衝突回避スレッドごとに組み込み worktreegit worktree、--worktree、subagent に isolation: worktree
    Skill（スキル）プロジェクト知識を明文化Agent Skills（SKILL.md）、$nameか暗黙呼び出しAgent Skills（SKILL.md）
    Plugin / Connector既存ツールへ接続Connectors（MCP）＋配布用 pluginsMCP servers＋plugins
    Sub-agent（サブエージェント）発案する者と検証する者を分離.codex/agents/ に TOML 定義.claude/agents/ の task subagents、agent teams
    State / Memory（6番目）完了済みと次を記録Markdown または Connector 経由 LinearMarkdown（AGENTS.md・progress files）または MCP 経由 Linear

    各部品の要点

    • Automation はループの「心拍」。これがあって初めて「一度きりの実行」が「実際のループ」になる。OpenAI は社内で日次の issue トリアージ、CI 失敗の要約、コミットブリーフィング、先週仕込まれたバグ探しなどに使う。in-session の primitive として、/loop は一定間隔で再実行、/goal は自分で書いた条件が真になるまで継続し、毎ターン後に別の小さなモデルが完了を判定する（コードを書いたエージェントが自分で採点しない）。
    • Worktree は並列が混沌に変わるのを防ぐ。2体が同じファイルを書くのは、2人のエンジニアが相談なしに同じ行をコミットするのと同じ頭痛。git worktree は別ブランチの独立作業ディレクトリで履歴を共有し、片方の編集がもう片方に物理的に触れられない。ただし「機械的衝突」は消えてもあなたのレビュー帯域が天井になる（orchestration tax）。
    • Skill は毎セッション説明し直す手間をなくす。エージェントは毎回コールドスタートし、意図の穴を自信満々の推測で埋める（intent debt）。スキルは規約・ビルド手順・「あの事故以来こうはしない」を外部に書き出したもの。スキルなしではループは毎サイクル、プロジェクトをゼロから再導出する。
    • Connector はループを実環境に触れさせる。ファイルシステムしか見えないループは小さい。MCP ベースの connector で issue トラッカー、DB、staging API、Slack に到達でき、「修正はこうです」と言うだけのエージェントが、自分で PR を開き Linear チケットを紐づけ CI green でチャンネルに通知するループに変わる。
    • Sub-agent は「作る者」と「検証する者」を分ける。コードを書いたモデルは自分の宿題を甘く採点しすぎる。別の指示（時に別モデル）の第2エージェントが、第1が言いくるめた問題を捕まえる。定番は「探索する／実装する／仕様に対し検証する」の分担。Claude Code の /goal は内部的にこれをやっており、作る者と検証する者の分離を停止条件そのものに適用している。サブエージェントはトークンを多く消費するので、第2の意見に金を払う価値がある所で使う。

    5. 1つのループの実例（Addy Osmani）

    部品を組み合わせると、1つのスレッドが小さな管制盤に変わる。Osmani が繰り返し使う形。

    flowchart TB A["毎朝 Automation がリポジトリで起動"] --> B["triage スキルが昨日のCI失敗・
    open issue・最近のコミットを読む"] B --> C["発見を Markdown / Linear に書き出す"] C --> D{"対応する価値あり?"} D -->|Yes| E["独立 worktree を開く"] E --> F["sub-agent① が修正を起草"] F --> G["sub-agent② が
    スキルと既存テストに対しレビュー"] G --> H["Connector が PR を開き
    チケットを更新"] D -->|処理不能| I["triage inbox に届き
    人間が対応"] H --> J["State ファイルに
    試行・通過・未解決を記録"] J -.->|翌朝 続きから| A

    「あなたがそこでやったことを見てほしい。一度だけ設計した。これらのステップを1つもプロンプトしていない。これが Steinberger の主張を現実にしたもので、部品が同じだから Codex でも Claude Code でも同じループになる」。なお6番目のメモリ（State）は「単純すぎて重要に思えない」が、長時間動くエージェントが依存する同じトリックである——モデルは実行間ですべてを忘れるので、記憶はコンテキストではなくディスクに置く。「エージェントは忘れるが、リポジトリは忘れない」。

    6. LangChain の4階層モデル（Sydney Runkle）

    LangChain は、ループを「積み重ねるスタック」として4階層で整理する。エージェントの価値は良いモデル単体ではなく、タスク群に適合するよう注意深く設計されたハーネスから生まれる、という立場。社内のドキュメント改善エージェントを例に各層を説明する。

    階層仕組み目的LangChain の道具
    1. Agent loopモデルがタスク完了までツールを繰り返し呼ぶ作業の自動化create_agent、任意の対応モデル
    2. Verification loop（検証）出力をルーブリックで採点し、不足なら feedback を付けて再試行品質・正しさの担保RubricMiddleware、after_agent hook
    3. Event driven loop（イベント駆動）イベント（新文書・スケジュール・webhook）が発火しエージェントが走るスケールした自動作業LangSmith Deployment（cron/webhook）、Fleet channels
    4. Hill climbing loop（坂登り）本番トレースを分析エージェントが解析し、ハーネス構成を改善ハーネス自体の改善LangSmith Engine
    • 検証ループは grader を加える。決定的（リンク解決・CI 通過・diff が要求範囲か）でも、エージェント的（LLM-as-a-judge）でもよい。トレードオフはレイテンシとコスト増だが「速度より品質が重要な大半の本番では割に合う」。
    • イベント駆動ループでエージェントは手動起動する対象ではなく、より大きなシステムの中で継続的に走るコンポーネントになる（例: openclaw の「heartbeat」で常時オンの能動アシスタントに）。
    • Hill climbing ループは最重要。最初の3層が「作業」を自動化するのに対し、これは「改善」を自動化する。各実行が残すトレース（モデルの行動、呼んだツール、grader の feedback）を分析エージェントが解析し、戻り矢印が先頭に戻るだけでなく、内側に手を伸ばしてエージェントループ自体を更新する。プロンプト/ツール調整が最も簡単だが、open-weight モデルならトレースや eval 結果を訓練信号とする RL ファインチューニングにも繋げられる。
    Steipete、Boris、Andrej（Karpathy）らは皆同じ結論に至った——エージェントの潜在力は、その周りに作るループにある。価値が複利で増えるのは、エージェントを自社のエコシステムに埋め込み、自分の基準に応じて継続的に改善する loop 3 と 4 である。

    7. ループはどこで動くか（Cobus Greyling のプレイブック）

    Cobus Greyling は「会話を Grok Build と Claude Code が支配しているが、ループは製品ロゴではなく小さな制御システム。primitive は移植可能で、どのランタイムを選ぶかは『ループがどれだけ長く走るか・誰に奉仕するか・何に触れてよいか』で決まる」と整理する。ループは「プロンプトではなく、記憶・検証・境界を持つ反復プロセス」であり、住む場所（home）——次の実行をスケジュールし、状態を永続化し、並列作業を隔離し、実ツールに到達する場所——が要る。

    4つの手（プレイブックの Tier 分け）

    Tier適する対象内容
    Tier A: ターミナル・ハーネス・ループ個人 / 小チームGrok・Claude Code が優れた「ハーネス＋ループ」面。/loop・skill・sub-agent・worktree・MCP connector が一カ所に揃う。リポジトリにスコープされ、人間が近くにいる開発者ループに最速。
    Tier B: プラットフォーム・ランタイム・ループ本番 / マルチテナント製品やチームに奉仕し、再起動を生き延び、監査証跡が要り、自分が書いていないコードを走らせる場合。深夜3時に誰もターミナルを開いていない時に走り、プロセスクラッシュを耐え、人間承認のため3日間一時停止する——これがランタイム領域。LangChain が構築してきた所。
    Tier C: エディタ / 軽量な代替エディタ固定 / 軽量用途すでにエディタに固定されているか、フルのエージェントプラットフォームを採用せず cron と webhook が欲しい場合。

    「すべてのエージェントにコンピュータを」— LangChain の本番解

    Greyling は LangChain の「Give your agent its own computer（エージェントに自分のコンピュータを与えよ）」というフレーミングを評価する。洞察は単純——推論だけではループは閉じない。提案しかできないエージェントはループを走らせておらず、コードを実行し、結果を観察し、直し、再実行できるエージェントこそループを走らせている。本番での違いは「コンピュータがどこにあるか」。

    • LangSmith Sandboxes：エージェントごとにハードウェア隔離された microVM（ファイルシステム・シェル・パッケージ・ネットワーク・永続状態）。ノート PC 上の共有コンテナではない。
    • Durable execution：チェックポイント化されたグラフ状態で、20分の実行がクラッシュやデプロイを生き延びる。
    • Agent Server 上の Cron：対話セッションと同じトレース・認証でスケジュール実行。
    • Human-in-the-loop interrupt：ループが一時停止してリソースを解放し、数日後に正確なチェックポイントから再開。

    cron の区別も盗む価値がある——Stateful cron（毎回同じ thread_id、昨日を覚える：夜間リサーチ、履歴付き監視）と Stateless cron（毎回新スレッド：バッチトリアージ、単発スイープ）。これは「STATE.md に追記する」vs「リポジトリを読んで終了する」に直接対応する。

    8. コストと安全のデフォルト

    Loop Engineering の最大の実務リスクはコストの急増。複数記事が口を揃えて警告する。

    トリアージは安く

    シグナルへの1モデルパスで済ませ、cron の毎ティックで実装者＋検証者を回さない（Greyling）。

    サブエージェントは条件付きで

    state が「対応すべき」と言う時だけ生成。5分ループで毎回フル maker/checker は朝食前にプランを焼き切る。

    反復に上限を

    1項目あたり3回試行したら人間にエスカレーション（Greyling）。

    ツール呼び出し予算

    反復ごとに上限を設け、進捗なく使い切ったら失敗シグナルとして別戦略へ（MindStudio）。

    Osmani も「token rich か token poor かで使用パターンが大きく変わるので、トークンコストに絶対に注意」と釘を刺す。MindStudio は「構造化フィードバック（エラー時に生のダンプでなく、原因コード＋意図＋反復エラーか新規かのフラグを付ける）」「すべてログし、頻繁に要約」「ハッピーパスでなく失敗ケースでループをテスト」を推奨する。

    9. ループでもなくならないもの — 検証・理解・判断

    Osmani の最重要パート。「ループは仕事を変えるが、あなたを仕事から消しはしない。そして3つの問題はループが良くなるほど、楽にではなく深刻になる」。

    問題内容
    検証は依然あなたの責任無人で動くループは無人でミスもするループ。maker と verifier を分けるのは「完了」という主張に意味を持たせるためだが、それでも「done」は主張であって証明ではない。あなたの仕事は、動くと確認したコードを出荷すること。
    理解は放置すれば腐る（comprehension debt）ループが自分の書いていないコードを速く出荷するほど、「存在するもの」と「自分が理解しているもの」の差が広がる。滑らかなループはこの差を速く広げる——ループが作ったものを読まない限り。
    快適な姿勢こそ危険（cognitive surrender）ループが自走すると、意見を持つのをやめて返ってきたものをそのまま受け取りたくなる。ループ設計は、判断を伴えば解毒剤、思考を避けるために使えば促進剤。同じ行動が正反対の結果を生む。
    「同じループを2人が作っても、正反対の結果になりうる。一方は深く理解した仕事を速く進めるために使い、もう一方は仕事を理解しないために使う。ループはその違いを知らない。あなたは知っている。だからループ設計はプロンプトエンジニアリングより難しい——簡単ではない。Cherny の主張は仕事が楽になったということではなく、レバレッジの点が移動したということだ。」

    Greyling も同じ結論を共有する。「速度は『存在するもの』と『理解しているもの』の差を覆い隠す。ループが修正を出荷しても、それがどう動くか知っているとは限らない。トレースは何が起きたかを教えるが、誰かがそれを読まねばならない」。そして Satya Nadella の言葉を引く——「人間の判断とトークン資本が共に複利で増える学習ループを早く作る企業は、模倣困難な優位を築く」。LangChain も「自動化は人間をループから外すことではない」とし、機微な操作（金融取引・DB 操作）には live な人間レビューが不可欠だと明言する。

    10. 結論 — ループを作れ、しかしエンジニアであり続けよ

    5記事に共通する結論は明快。Loop Engineering は仕事の進め方の進化のプレビューだが、万能の置き換えではなくレバレッジである。

    • ループを設定せよ。ただしエージェントに直接プロンプトを打つことも依然有効。バランスが肝心（Osmani）。
    • 自分でコードをレビューせず、自動ループに完全依存すれば、製品品質は落ち、下降スパイラルに陥る（Osmani）。
    • 価値が複利で増えるのは loop 3・4——エコシステムに埋め込み、基準に応じて継続改善するループ（LangChain）。
    • ループは制御システムであり、ランタイムは「長さ・対象・権限」で選ぶ（Greyling）。
    • インフラ（リトライ・レート制限・ツールオーケストレーション）はプラットフォームに任せ、ループのロジックは推論に集中させてよい（MindStudio）。
    「ループを作れ。だが『go を押す人』ではなく『エンジニアであり続けるつもりの人』として作れ。」——Addy Osmani
