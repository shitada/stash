    「今日は、オープンソースモデルと Fireworks を活用して、アプリケーション固有で本番グレードの、推論最適化された AI をどう作るか。そして、それらのモデルをどう Foundry 上で推論提供するかを話します。」 — Vivek Chauhan（00:00 付近）

    2. AI 市場における3つの基本原則

    Fireworks は、現在の AI ランドスケープに「3つの基本的な真実（three fundamental truths）」があると考えている。これがセッション全体の前提となる。

    #原則含意
    1 オープンウェイトとクローズドの差が消滅 大半のエンタープライズ用途で、品質基準を保ったままオープンウェイトモデルに切り替え可能。例: メール要約に Claude Opus 4.7 を使う代わりに、より小さなオープンウェイトモデルで同等品質を達成しコスト削減。
    2 真の差別化は自社データの組み込みから 競合と同じクローズドモデルを使う限り差別化は生まれない。ファインチューニングで自社のデータモード（data mode）を AI に埋め込むことが鍵。
    3 リーダー企業は「フライホイール」を回す 先進企業は推論最適化やファインチューニング単発にとどまらず、AI スタックを継続的に改善する好循環（flywheel）を見出している。
    「自分の AI スタックを本当に差別化したいなら、ファインチューニングの活用が極めて重要です。」 — Vivek Chauhan（01:27 付近）

    3. Fireworks AI の規模と推論エンジン

    Fireworks の足元の規模は大きい。1日あたり 30兆トークン以上を処理しており、これは Anthropic（Claude）や OpenAI といったフロンティアのクローズドラボに次ぐ規模だという。グローバルで 10,000社以上のエンタープライズ顧客が、オープンウェイトモデルを使った本番アプリケーションを構築している。

    30兆+
    1日あたり処理トークン数
    10,000+
    エンタープライズ顧客数
    ~84,000
    1モデルの推論最適化の組み合わせ数

    具体的な顧客名も挙げられた。Cursor は Fireworks の RL ロールアウト機能を使って Composer 2／Composer 2.5 モデルを構築。ほかに Vercel、Jasper、Uber、DoorDash などが、ファインチューニングやトレーニングによる品質向上、速度・スループット・Time to First Token（最初のトークンまでの時間）といったアプリ固有の性能要件のために Fireworks を利用している。

    なぜ自前の推論エンジンが要るのか

    Vivek は「LLM 推論は非常に難しく残酷な組み合わせ最適化問題だ」と表現する。speculative decoding（投機的デコーディング）や量子化モード、利用可能な多様な GPU SKU、並列化戦略などの掛け合わせは、1つのモデルを提供するだけで 84,000〜85,000 通りの順列に達する。ほとんどの企業にとって、この複雑性を社内で抱えるのは合理的でない。

    Fireworks は CUDA カーネル、量子化、speculative decoding、適応的キャッシュ（adaptive caching）といった新しい最適化トリックを継続的に自社スタックへ取り込み、顧客は「Time to First Token を 500 ミリ秒未満に」「エンドツーエンドのレイテンシを 2 秒未満に」といったアプリ固有の要件だけを指定すればよい、とした。これらの推論エンジン機能が、後述のとおり Foundry 上でも利用可能になる。

    4. 顧客の段階に合わせる ― 3つのトレーニングサーフェス

    Vivek が「このスライドが一番のお気に入り」と述べたのが、企業に AI スタックを「所有」してもらうという思想だ。企業ごと、さらには同じ企業内のチームごとに AI ジャーニーの段階は異なる。Fireworks は、その段階に応じて3つの異なる抽象度（コントロールの粒度）を提供する。

    サーフェス主な対象特徴
    Training Agent プロダクトマネージャー / アプリビルダー データとビジネス用途を渡すと、ファインチューニング済みモデルを返す。ML の知識が不要。
    Managed Training ML エンジニア 使う学習手法とデータセットが既に決まっている場合に、データを移すだけで学習ジョブを開始。バックエンドのインフラ調整は不要。
    Training API 上級 ML チーム / 研究者 プラットフォームの生のプリミティブを公開。カスタム損失関数・報酬関数、最先端アルゴリズム、最大コンテキスト、フルパラメータファインチューニング。

    重要なのは、どのサーフェスを使っても「ワンクリックデプロイ」できること。別の場所で学習し、別の場所で推論し、ダウンロードや再フォーマットを行う、といった手間が要らない。これがイテレーション速度に直結する。「ワンクリックでデプロイし、eval を回し、本番に出す」という流れだ。

    flowchart TD Start[AI ジャーニーの段階で選ぶ] --> A{ML の専門性は?} A -->|低い / PdM・アプリ開発| TA[Training Agent
    自然言語で指示] A -->|中 / ML エンジニア| MT[Managed Training
    手法とデータを選ぶ] A -->|高い / 研究者| API[Training API
    生プリミティブを操作] TA --> Deploy[ワンクリックデプロイ] MT --> Deploy API --> Deploy Deploy --> Eval[eval 実行] --> Prod[本番投入]

    5. フライホイール ― 継続的な学習ループ

    原則3で触れた「フライホイール」は、本セッションの中核概念だ。先進企業は Fireworks 上で次のループを高速に回している。すなわち 学習 → デプロイ → モニタリング → トレース取得 → 再学習。Fireworks が GPU オーケストレーション、弾力性（elasticity）、不良ノードや CSP の問題といった「複雑な部分」を引き受けるため、利用者は重みの配置・ホットロード・チェックポインティングといった苦労から解放される。

    Vivek は「社内 DIY だと数週間かかるところを、Fireworks なら 10 回のイテレーションを回せるだろう」と述べた。

    flowchart LR Train[学習
    SFT / DPO / RL] --> Deploy[ワンクリック
    デプロイ] Deploy --> Monitor[本番モニタリング] Monitor --> Trace[トレース取得] Trace --> Retrain[再学習] Retrain --> Train

    数値忠実性（Numerical Fidelity）の保証

    もう一つ強調されたのが「数値忠実性」だ。大規模モデル、特に浮動小数点精度を使う MoE（Mixture of Experts）モデルを学習すると、数値が大きくドリフトしうる。「学習したもの」と「実際に提供されるもの」が一致しない危険がある。Fireworks はこの一致を保証し、「eval で得た結果が本番プラットフォームのエンドユーザー向け性能と一致する」ことを担保するという。さらに、学習と推論が1つのプラットフォームに統合されているため、ベンダースプロール（vendor sprawl、複数ベンダーの乱立）が起きない点もメリットとされた。

    6. デモ① ― トレーニングエージェント

    Vivek は会場に「eval を書いたりデータを再フォーマットするのが好きな人は?」と問いかけ、誰も手を挙げないのを確認したうえで「ではこのエージェントはあなたのためのものです」と切り出した。トレーニングエージェントの発想はシンプルで、平易な英語で「これが私のデータセット、これがビジネスゴール、ファインチューニングを手伝って」と伝えるだけでよい。

    エージェントが代行すること

    • データの再フォーマット
    • eval（評価）の作成
    • 用途に合うモデルの探索・選定
    • ハイパーパラメータ探索（多数のノブを手作業で試す苦痛を肩代わり）

    進め方（プラン承認フロー）

    1. エージェントがビジネス用途を受け取り、モデル選定・eval 作成・データ整形を含むプランを提示。学習イテレーションにかかるコストも事前提示する。
    2. 承認すると、データセットの一部（small cut）でハイパーパラメータのグリッドサーチを行い、探索空間を絞り込む。
    3. 「10 通り試したうち、有望なのはこの2つ。理由と eval 結果はこれ。フルデータセットで本実行してよいか?」と再確認。
    4. 承認後、フルデータセットで学習を実行し、内部の挙動をステップごとに可視化、最後に総合レポートを提示。

    この機能はすべてヘッドレス（headless）でも利用可能で、skill file として提供される。GitHub Copilot、Claude、Cursor など自分のコーディングエージェント／ハーネスから直接モデルを学習させたい場合、skill file を置いて API キーを追加するだけでよい。教師ありファインチューニング（SFT）、選好最適化（preference optimization）、分類タスク向けモデル作成などがあらかじめパッケージ化されている。

    7. デモ② ― マネージドトレーニングと推論デプロイ

    ML の詳細を自分で握りたい層向けに、クリック操作のワークフロー（Managed Training プロダクト）も用意されている。教師ありファインチューニング、強化ファインチューニング（RFT）、選好最適化から手法を選べる。

    学習ジョブの構成

    • モデルを選択（Fireworks はオープンウェイトモデルの推論で「day 0」提供を志向しており、ファインチューニング対応も最速級）。
    • データセットを直接アップロード。セキュアトレーニング（BYOB）として自社クラウドからデータをストリーミングすることも可能。
    • エポック数・学習率・バッチサイズなどの生パラメータを公開。スマートなデフォルトも用意。
    • RFT の場合は grader（採点器） をアップロード、または GitHub リポジトリで接続し、報酬付与に用いる。top K・top P なども調整可能。
    • Weights & Biases 連携に対応。API キーを差し込めば詳細なテレメトリで内部挙動をモニタリングできる。
    • 学習中はプラットフォーム上で loss / reward 関数をリアルタイム表示。挙動が想定外なら一時停止して報酬関数や eval を見直せる。

    コンパウンドトレーニング（積み重ね学習）

    SFT したモデルの上に DPO を、その上にさらに強化学習を…という形で、好きなだけイテレーション／パスを重ねられる。これがフライホイールでいう「継続的な学習」の具体形だ。

    推論デプロイとカスタムシェイプ

    学習後はファインチューニング済みモデルのオンデマンドデプロイに対応する。注目点は file optimizer による「カスタムシェイプ」で、同じ GPU 台数でも、スタックの組み方次第で性能が大きく変わる。同一モデルを「最小シェイプ」でも「フル精度シェイプ」でも、また「速度最適化」「スループット最適化」でも提供できる。トラフィックの増減に合わせたオートスケーリングも備える。

    8. 上級者向け ― Training API

    Training API は、プラットフォームの「生の力（raw power）」を上級 ML ユーザーへ開放するための製品だ。インフラ部分は Fireworks が裏で引き受けつつ、ノブはすべて公開する。カスタム損失関数を書け、最新アルゴリズム（オンポリシー系など、いわゆる「on policy」「SYSPO」等として話題のもの）を使える。

    • フルパラメータファインチューニングに対応。
    • 公開 GitHub リポジトリの クックブック（cookbook） に多数のレシピがあり、ゼロから書かずにレシピのノブを変えるだけで始められる。
    • 最大の特長は自分のハーネス内から直接モデルを学習できること。AI は「より良く仕事をする方法」を、自社の環境・ハーネスの中で、どのツールにアクセスでき、どうオーケストレーションするかを知ったうえで学ぶ必要がある、という思想に基づく。
    • 用意された例を使えば、ラップトップから 1 兆パラメータ規模のモデルを学習することも可能（インフラはすべて Fireworks 側が処理）。

    9. 大型発表 ― Fireworks が Microsoft Foundry で一般提供（GA）

    前日の Satya Nadella 基調講演に絡む発表として、Fireworks の Microsoft Foundry 上での一般提供（GA）が紹介された。Vivek が示した推論デプロイ機能はすべて Foundry 上で利用可能になる（ただし学習側は対象外。Fireworks で学習し、ダウンロードして Foundry のカスタムモデルデプロイへ移す形）。

    「Fireworks のすべての良さと、Foundry のすべての良さを、シンプルな形で ― ワンエンドポイント、ワンコントラクト、ワンジャーニー、ワンプラットフォームで使えるようにする。」 — 発表ビデオより（16:18 付近）

    10. なぜ Foundry 上の Fireworks なのか ― 4つの理由

    ここから Jet（Jetashree Ravi）が、モデルカタログから本番までの流れと、顧客が Foundry 上の Fireworks を選ぶ理由を説明した。理由は主に4つに整理される。

    1. パフォーマンス

    CUDA カーネルレベルからの最適化で、GPU 上にモデルが載った状態で各ユースケースに合わせて性能を引き出す。

    2. モデルの優位性

    多くのモデルを day 0 で提供。Deep Seek V4 や Kimi 2.6 などの最先端 OSS を出た当日にエージェントへ組み込める。

    3. コントロールと拡張性

    Fireworks と Azure Foundry カタログの統合により、多くのカスタムモデルを Foundry 上にデプロイできる。

    4. エコノミクス

    クローズドソースはスケール時に急激に高価化。同等品質でより安い代替を、Azure クレジットを活かして使える。

    11. 3つの統合モード

    顧客が Fireworks を Foundry に統合する際の主なモードは3つある。本番要件に応じて選ぶ。

    モード課金 / 構成向いている場面提供モデル例
    Serverless（従量課金） デプロイ作成不要。複数ユーザー共有のトークン単位課金。共有インフラのためレート制限あり。 まず手早く試したい / 共有で十分な用途 GLM 5.1、Kimi 2.6、GPT 120B など
    PTU（Provisioned Throughput Unit） GPU を予約して専用デプロイ。PTU でゲートされる。 本番ワークロードが明確で、専有が必要なとき 用途に応じた予約デプロイ
    Bring Your Own Weights Fireworks で学習 → 重みをダウンロード → Azure にアップロードしてデプロイ。Azure クレジットを使用。 クローズドに対抗して自社学習した独自モデルを使うとき 自社の独自・プロプライエタリモデル

    12. 顧客事例（Foundry × Fireworks）

    統合に対する顧客の声として3社が紹介された。

    企業求めたもの結果
    UiPath より速く・よりコスト効率の良いモデル。さらにオープンソースモデルでクローズドの Sonic 4.6 の品質を超えること。 Fireworks × Azure Foundry のモデルで実現可能に。
    Bolt New 本番用途でスループットとレイテンシの両面でよくスケールすること。 Azure エコシステムと Fireworks モデルで解決策を発見。
    Motive 反復タスクを高ボリュームワークロードで実行すること。 Fireworks エンジンで最適化したモデルを Azure Foundry にデプロイし、高ボリュームの反復処理を提供。

    13. デモ③ ― Foundry でのデプロイとエージェント統合

    カタログからの Serverless デプロイ

    Azure Foundry の discover ページからモデルカタログへ進む。Fireworks で最適化されたモデルは多くが FW プレフィックスを持つ。従量課金で使うなら、モデルを選んで deploy をクリックし、レート制限などのカスタム設定（共有インフラのため必ず何らかのレート制限がある）を調整。デプロイ後はチャット UI で対話でき、API でエージェントやアプリに組み込める。

    カスタムモデルのアップロードとデプロイ

    独自モデルを扱いたい場合は、Azure の build セクション → モデルページの custom model セクション → add ボタンから自分のモデルを追加し、ベースアーキテクチャを選ぶ。デフォルト未対応のアーキテクチャなら Azure と Fireworks に連絡すれば有効化を支援するという。提示されるコマンドラインでモデルをアップロードし、custom model を選んで deploy。専有デプロイなのでレート制限はなく、PTU（プロビジョンドスループット）数を自分で決められる。

    Azure エージェントへの組み込み

    build ページの agents から、エージェントに名前を付けて deploy。背後で使うモデルを Foundry 上の Fireworks モデルやカスタムモデルに差し替えられる。Web 検索ツールや自前で定義したツール／関数とも統合でき、エージェントは選んだ Fireworks モデルとともにツールを使いこなす。

    14. 事例研究 ― Harvey AI と Legal Agent Benchmark（LAB）

    ゲストの Nico Groupin は Harvey AI の応用研究責任者。Harvey は法務・専門サービス向けの生成 AI プラットフォームで、世界中の数千の法律事務所が利用する。ミッションは、弁護士の仕事をより効率的・高品質にし、雑務（busy work）を減らすこと。

    3週間前、Harvey は Legal Agent Benchmark（略称 LAB）を公開した。エージェントがアプリ層で広がる中で「自分の業界・領域で品質をどう考えるか」という問いに答えるためのベンチマークで、法務タスクにおける Long Horizon（長期ホライズン）エージェントの性能を測る。LAB は4つの構成要素からなる。

    15. Advisor Agents ― ハイブリッドで品質とコストを両立

    Nico が「最も興奮している」と語り、当日朝に記事も公開したのが Advisor Agents の取り組みだ。これは、オープンウェイトモデル（例: GLM 5.1）が駆動する worker エージェントが、より重い（多くはクローズドのフロンティア、例: Opus 4.7）advisor モデルを必要に応じて呼び出す、というシステムである。推論時のルーティング（inference time routing）によって、品質とコストのパレートフロンティアを効果的に攻略できるという。

    sequenceDiagram participant U as 依頼（法務タスク） participant W as Worker Agent
    (GLM 5.1 / オープンウェイト) participant A as Advisor Model
    (Opus 4.7 / クローズド) U->>W: タスク投入 W->>W: 大半の作業を低コストで実行 W->>A: 難所だけ助言を要求（必要時のみ） A-->>W: 高品質な助言を返す W-->>U: 成果物を生成

    クローズドのフロンティアは性能の最前線を与える一方、コストとレイテンシのトレードオフが大きい。オープンウェイトははるかに安く速く、セキュリティ面の保証も大きいが性能で劣りうる。ハイブリッドシステムは、より低いコストとレイテンシで品質を維持するどころか、多くの場合で品質を上回ったという。Vivek はこの結果を、あるタスクで最大 2.4 倍安く、かつ all-task メトリクスで約 30% 良いと要約した。

    「多くの企業が、あらゆるタスクにフロンティアのクローズドモデルを使っている。これは、バンカーを狙うにもアリを狙うにも空母を持ち出すようなものだ。そんな必要はない。」 — Vivek Chauhan（33:45 付近）

    16. Harvey のオープンモデル戦略 ― 4つの考慮事項

    「なぜ Harvey にとってオープンモデルが重要か」という問いに対し、Nico はパートナーシップには2側面（マネージド推論経由でのオープンウェイト消費と、ポストトレーニングによる改善）があるとし、ポストトレーニングを魅力的にする4つの考慮事項を挙げた。

    ① ドメイン専門性（最初に挙げた理由）

    多くの人は「まずコストとレイテンシだろう」と思うが、Nico が最初に挙げたのはドメイン専門性だった。評価を通じて見えてきたのは「jagged intelligence（ギザギザの知能）」― Andrej Karpathy が命名した、モデルがある事には非常に強い一方で別の事には惨めに失敗し、その理由を解釈しづらい、という観察だ。法務でも同様で、あるモデルは企業 M&A や財務デューデリのような取引業務に強いのに、判例リサーチや文章作成中心の訴訟業務には弱い、といったことが起きる。ポストトレーニングで、こうした凹凸をならし、全実務領域でモデルを底上げできる。

    ② コストとレイテンシ

    2番目がコストとレイテンシ。Vivek の言い回しを引いて「必要がないならフェラーリをレーストラックから出す必要はない」と表現。利用は二峰性（bimodal）の分布になりつつあり、極めて高ステークスで複雑な仕事にはプレミアムを払ってフロンティアのクローズドを使う一方、オープンウェイトが完全に適する単純な仕事も多い。この2モード間のルーティングがますます重要になる。

    ③ セキュリティとガバナンス

    業界全体が重視し始めているのがセキュリティとガバナンス。最高ステークスの仕事では、顧客は自社 VPC 内でのセキュアなデプロイを求める。オープンウェイトはそれを可能にし、さらにモデルの完全な推論トレースにアクセスできるため、エージェントの挙動に対する解釈可能性・監査可能性の作業を大きく高められる。これはコンプライアンス上、時とともに重要性を増している。

    ④ エンドツーエンドのイテレーション速度

    マネージド推論側は「プラグアンドプレイ」で、イテレーション不要にすぐオープンソースモデルを使い始められる。マネージド学習インフラによりイテレーションループがよりタイトに・速く・効率的になる。Harvey には「all timelines are compressed（あらゆるタイムラインは圧縮されている）」という社内の言い回しがあり、これは研究にも当てはまる。ポストトレーニング実験を並列・低タッチで多数走らせられることが、研究チームの速度を押し上げる。

    17. まとめ ― Foundry の価値とファウンダーへの助言

    Nico は「エンタープライズグレードのインフラ、とりわけ AI インフラといえば、すぐに Microsoft Foundry を思い浮かべる」と述べた。法務はナレッジワークの中でも最高ステークスで、最高グレードのインフラ・セキュリティ・ガバナンス・コンプライアンスを要する。だからこそ Harvey は一貫して Azure 顧客であり続けたという。

    Fireworks との組み合わせは「両得（best of both worlds）」だ。すでに活用しているエンタープライズグレードのインフラの上に、オープンウェイト向けのマネージド推論、学習インフラ、そしてそれらのシームレスなデプロイを重ねられる。自社インフラチームの足かせにならずに済む、と評価した。

    「Harvey に参加した頃、私たちは Airbnb に5〜6人。革新的な AI 課題どころか、その日 Wi-Fi がつながるかを心配していた。3年前、こうした AI インフラの多くは存在しなかった。今は、すべてを自分たちで解決しなくてよいと分かっていて、どれだけ夜よく眠れるか。」 — Nico Groupin（39:10 付近）

    ファウンダー／エンタープライズへの助言

    • 今日 AI ネイティブ企業を立ち上げて初めて AI スタックを作るなら、あるいは大企業として AI スタックをエージェントネイティブに作り直すなら、利用可能な「追い風（tailwinds）」に乗るべき。
    • 具体的にはマネージドインフラを使うことを推奨。「all timelines are compressed」だからこそ、はるかに速く動ける。
    • すでに Azure を使っていて、新規ユースケースや既存ユースケースの再構築を考えているなら、Azure 上の Fireworks モデルを試すとよい。Azure 上のエージェントでも、自作のエージェントでも、複数の統合モードで組み込める。
