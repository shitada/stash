    3 行サマリ

    • Microsoft AI が独自開発した MAI-Thinking-1（35B active / 1T total の MoE）は、30T トークンをゼロから事前学習し、第三者モデルからの蒸留を一切行わずに、SWE-Bench Pro 52.8%／AIME 2025 97.0%／LiveCodeBench v6 87.7% という同サイズ帯トップ級のスコアを達成した推論モデル。
    • その背後にある考え方が 「Hill-Climbing Machine（丘登り機械）」。データ・アーキテクチャ・分散学習基盤（YOLO）・強化学習基盤（Rocket）・評価・安全評価を 一体化した経験的最適化ループとして運用し、長期にわたる log-linear な性能改善を持続させる仕組みである。
    • RL は 「学ばせる、模倣させない」を原則とし、STEM／Agentic／Helpfulness & Safety の 3 つの専門モデルを GRPO の改良 + Self-Distillation で訓練したのち SFT で統合し、最終的に軽量 RL を 1 度かけて MAI-Thinking-1 を完成させる構成。MAIA-200 推論ハードでは GB200 比 40% 以上スループット改善。

    0. はじめに ― 3 つの設計原則と全体像

    MAI-Thinking-1 は Microsoft AI（MAI）チームが、社内クラスタでゼロから事前学習した最初の推論モデルである。STEM 推論とコーディングの両方で同サイズ帯のフロンティアモデルと競争力があり、特に SWE-Bench Pro 52.8%、AIME 2025 97.0%、AIME 2026 94.5%、LiveCodeBench v6 87.7% といったベンチマークで、Sonnet 4.6 と広範囲で互角〜上回る結果を示している。

    0-1. 3 つの設計原則

    ① Capabilities should be learned, not inherited

    能力は 「学ばせる」 もので、蒸留で受け継ぐと長期 climb に必要な操作性・頑健性が欠ける。

    ② Simplicity is sustainable

    シンプルでスケーラブルなレシピ・クリーンで信頼できるデータ・透明な基盤が、ゼロからの climb を支える。

    ③ Scientific rigor avoids shortcuts

    すべての意思決定は scaling ladder・ablation・evaluation でテスト可能でなければならない。

    1. 事前学習（Pre-training）

    1-1. モデルアーキテクチャ

    MAI-Base-1 はデコーダ専用 Transformer。5 層の local attention に 1 層の global attention を交互配置（Gemma 3 流）、dense FFN と高 sparsity な MoE を交互に配置するインターリーブ構成を採用する。

    • Tokenizer：o200k_base（vocab 200,019）。社内 tokenizer よりわずかに精度劣るが、社内ツール連携を優先。
    • Attention：local は RoPE（window 512、base 10,000）、global は位置エンコーディングなし。GQA（KV 8 head／128 dim）。FlashAttention-4 + Ulysses-style context parallelism。
    • MoE：LatentMoE 設計（NVIDIA 2025）。8/512 expert を softmax gating で選択。圧縮潜在空間で all-to-all 通信、ルーティング判断は元の表現に基づく。
    • Load balancing：global-batch 集約による GShard 流ロス。capacity を取らない fully dropless MoE を採用（possible memory blowup を別の static-memory dropless モードで吸収）。
    • 正規化：input/output 双方の residual add 直前で RMSNorm。Q/K に RMSNorm 適用。bias は使わず embedding は tied。

    家族全体の構成は L12（365M active）から L78（35.6B active）までを scaling ladder として用意し、活性化パラメータあたり同 TPP（tokens per parameter）で訓練して効率比較を行う。

    1-2. アブレーション方法論 ― Scaling Ladder と Efficiency Gain

    小規模で見えた改善が大規模でも生き残るかを担保するため、scaling ladder と Efficiency Gain（EG） を中心に意思決定する。EG はベースラインの scaling law $L=AC^{-\alpha}+E$ を fit し、候補が同じ loss に到達するのに必要なベースラインコストの比として定義する：

    $$\text{EG} = \frac{f^{-1}(L')}{C'}$$

    • FLOPs を C にした EGFLOPs はアーキテクチャ自体の優劣を見るのに使用。
    • wall-clock 時間を C にした EGTime は既存ハード上での実効性を見るのに使用。
    • 例：MoE-every-layer に shared expert を加えると EGFLOPs = 1.03 だが EGTime = 0.82 となるため、現行レシピである interleaved 構成のほうが優れている。
    • Sparsity を 256 → 512 → 1024 expert に上げると EG が一貫して向上することを確認。MAI-Base-1 は品質と効率のバランスから top-8/512 を採用。

    1-3. 評価方法論 ― NLL ベースの社内ベンチ

    日々の開発判断には、汎用ベンチではなく NLL（負の対数尤度）に基づく社内ベンチマーク約 40 種 を使用する。理由は、(a) 自動生成や judge を必要とする評価は高コスト、(b) MMLU や MATH のような評価は format 変化や MCQ 解釈能力の創発に敏感、(c) NLL は teacher-forcing なので minor error が積み上がらず信号品質が高い、の 3 点。

    カテゴリは Code / STEM / Math / General Knowledge / Multilingual の 5 群で、集約スコアは次の重み付き和で定義：

    $$\text{Target} = 0.5 \cdot \text{Code} + 0.175 \cdot \text{STEM} + 0.175 \cdot \text{Math} + 0.1 \cdot \text{General} + 0.05 \cdot \text{Multilingual}$$

    公開評価はリーク対策として huggingface.co とそのミラーを学習データから除外し、すべての訓練ソースで 20-gram 80% 類似度の fuzzy dedup を適用。それでも漏れる例があるため、Web 上に存在しない private benchmark を独自に構築する。

    1-4. データソース・ライセンス・処理

    • 方針：LLM 生成の合成データは事前学習に使わない。収集データ中の AI 生成コンテンツも検出して除外。社内 PII/Safety フィルタを必ず通す。
    • 独自 Web クローラ：robots.txt と各種 HTML/meta 制御を尊重。USTR Notorious Markets リスト掲載サイトは除外。
    • 第三者データセット：商用契約で取得、適切なオーナーシップ／利用権を diligence。
    • ユーザーデータ：明示的 opt-in や合意条件下のみ使用。

    HTML 抽出戦略

    単一抽出器ではなく、ソース構造に応じて 4 つを使い分け：

    • Source-specific structured parser（HTML/XML のスキーマ対応）
    • Hand-crafted extractor（BeautifulSoup 等で重要ドメインを丁寧に処理）
    • LLM / agent-based 処理（normalization・semantic filtering。LLM は 原文の保持／削除のみ可。新規合成は禁止）
    • Raw 訓練（Wikipedia の wikitext は約 3 倍冗長でも raw のままが品質高い）

    重複除去（Deduplication）

    • Boilerplate 除去：ヘッダ・ナビ・フッタなどを line-occurrence 統計で除去。
    • Exact dup：MD5 ベース。
    • Fuzzy dup：MinHash-LSH、類似度 0.8。
    • Templated page：頻出 token に skeletonize して fuzzy dedup。
    • Semantic dup：Qwen3-Embedding-0.6B でクラスタリングし、各クラスタから一定数のみ保持。
    • Cross-dataset：global drop-order に基づき、複数データセットで重複した場合に高優先度のみ残す。

    数百のデータソースをどう重み付けるかは、(i) 効用の定義、(ii) 巨大な探索空間、(iii) スケール依存性、(iv) データセット間相互作用、(v) multi-epoch 効果、(vi) 計算コストの 6 課題を伴う。

    Rank invariance hypothesis の反例

    小スケールで stem-heavy-mix が code-heavy-mix を STEM NLL で上回ったが、23B / 20T 規模では順位が逆転した。原因は stem-heavy-mix が高品質だが 多様性に乏しい 2 つのソース（重み 11.8% vs 0.3%）に依存していたことで、大規模では情報量が枯渇しスケールが鈍化したため。以降は ladder アプローチで scaling 特性まで含めて評価する方針に。

    1-6. Mid-training データミックス

    Pre-training と同じコーパスを高品質サブセットに再フィルタ／再 weighting し、長い系列長で再 pack。STEM/Math 35% + Code 55% + Background 10% に寄せて RL 推論のための土台を強化する。

    • STEM 推論データ：Bloom 認知レベル「Analyze 以上」のドキュメントを保持する Bloom Analyze heuristic でフィルタ。
    • コードデータ：repo を品質 3 ティアに分け、ティアごとに extension フィルタ。file-level と repo-level の 2 視点フォーマットを併用。
    • Memorization-aware epoch capping：pre-training で「NLL < 0.01 で予測されるトークン」割合が高い source は強く capping。
    • Long context データ：同じ重みで 64K → 256K に再 pack し、distribution shift を抑制。

    数値精度

    • weight/activation のデフォルトは BF16。Forward GEMM は FP8 E4M3、data-gradient は FP8 E5M2、weight-gradient は BF16 + FP32 accumulation。
    • FP8 は 1024-step delayed scaling。
    • 数値敏感領域（attention/MoE/output logits、residual stream、embedding、RMSNorm、router）は FP32。
    • Higher → lower precision の dropdown は stochastic rounding。

    Pre-training loss は早期にいくつかの spike が起きたが、いずれも自然に回復。spike は code dataset に偏り、dropless routing 下の expert imbalance に相関していた。training batch のスキップや手動介入は一切なし。

    1-8. 同時期モデルとの比較

    DeepSeek v3.2 / v4 Pro、Kimi-K2、Gemma4-31B、社内 23B モデルとの BPB（bits-per-byte）比較で、MAI-Base-1 は同等 active parameter のモデル群を 全 4 タスクで上回り、active 1.4×／total 1.6× の DeepSeek v4-Pro に次ぐ位置に。

    1-9. YOLO ― 自社分散学習フレームワーク

    YOLO（You Only Launch Once）は PyTorch 上に MAI が自社開発したフレームワークで、pre-training / mid-training / SFT / RL の learner を統合的に扱う。Megatron-Core・DeepSpeed・TorchTitan に学び、kernel から parallelism、scheduling まで end-to-end の制御を確保する。

    主要コンポーネント

    Kernel

    Triton / CUDA / CuteDSL / CUTLASS による FP8 GEMM、grouped GEMM、fused quantization、Blackwell の Cluster Launch Control（CLC）を活用。

    Tensor sharding

    JAX / DTensor 流の独自 sharding annotation。attention は TP、MLP/MoE は別 DP のような parallel folding も可能。

    Parallelism

    独自 ZeRO 1〜3、TP（column/row）、Ulysses-style CP、EP（dropless 含む）、PP。Pre-training は EP=64・TP=1。

    Activation checkpointing / offloading

    transformer-layer checkpointing と fine-grained recomputation を混合。pinned host memory + 専用 CUDA stream で非同期 offload。

    MoE 層

    dropless / capacity-capped 両対応。router replay や static-memory dropless mode をサポートし、imbalance による OOM を防止。

    Determinism

    同一 HW・SW・config 下で bitwise reproducibility。RMSNorm 逆伝播の 2 段 tiled reduction や stable sort top-k 等で順序固定、NVLink SHARP は無効化。

    Fault tolerance

    • Distributed Checkpoint（DCP）改造：metadata 作成・シリアライズを書き直して CPU メモリ・保存時間を 10× 以上削減、save plan は事前計算してキャッシュ。
    • Async checkpointing：device → host コピー後は別 process が書き込み。1 件 in-flight に制限する admission control。
    • Storage：Azure Blob を低レイテンシ／高帯域で使用。replicated state は 1 度読んで NCCL broadcast し hotspots を回避。
    • Rapid recovery：Ray actor + hot standby により pod 再生成を回避。GB200 では actor restart 時に OS cache を明示的にクリア。

    2. 強化学習 Climb（RL Climb）

    2-1. RL レシピ（GRPO 改良 + Self-Distillation）

    RL は GRPO（Shao et al., 2024）+ token-level policy gradient（Yu et al., 2025）をベースに、以下の 2 つの修正を加えた。

    Adaptive Entropy Control

    upper clip bound を policy entropy 観測値に基づいて自動調整する simple integral controller を導入。

    $$r^{tr}_{i,t}(\theta) = \mathrm{clip}\bigl(r_{i,t}(\theta),\; 1-\epsilon,\; (1-\epsilon)^{-1}+k\bigr)$$

    • 観測 entropy $\hat{H}(\pi_\theta)$ をトークンレベルで importance-weighted に推定。
    • target $H^* = 0.3$ を超えていれば $k$ を縮め、下回っていれば広げる（$\epsilon = 0.6$、$k_{\max} = 2.5$、step $\delta = 0.25$）。
    • 初期 $k = 0$ で clip 範囲は log 空間で対称、明示的な entropy bonus 項より良い。

    Outer Ratio Clip

    GRPO が意図的に unclipped にする 2 ケース（A<0 かつ r>1、A>0 かつ r<1）が gradient norm の壊滅的 spike を起こす場合があったため、外側に hard clip $r^{out}_{i,t}(\theta) = \mathrm{clip}(r_{i,t}(\theta), r_{\min}, r_{\max})$ を追加（実装では $r_{\max}=50$、$r_{\min}=0$）。

    Reward の分解

    $$R(q, y_i) = R_{\text{task}}(q, y_i) + w_{\text{lang}} \cdot R_{\text{lang}}(y_i) - w_{\text{len}} \cdot R_{\text{len}}(y_i)$$

    • Language consistency reward（$R_{\text{lang}}$）：英語以外の語数で penalize。RL の文脈延長で増える多言語混在 CoT による学習・推論ポリシー乖離を防ぐ。
    • Length penalty（$R_{\text{len}}$）：問題の pass rate $\rho_q$ で重み付けし、難問は緩く、易問は厳しく。

    Sampling 戦略

    • Problem sampling：早期に $G_{\text{early}}=16$ を生成して pass rate を判定、残り全 $G=128$ を生成。最終 pass rate $[\rho_{\min}, \rho_{\max}] = [0.1, 0.8]$ 内のみ訓練対象に。
    • Rollout sampling：top-p（$p=0.97$）。top-p 外のトークンの logit を −∞ に置換することで off-policy mismatch を抑制（rollout 時のマスクを訓練側に持ち込む）。
    • Length curriculum：8K → 16K → 32K → 64K → 128K と powers of two で出力長を拡張。

    Self-Distillation の役割と発見

    長期 RL の実用化に不可欠。midtrained checkpoint に対し RL 中に集めた rollout で SFT を行う。用途は (a) 生プロンプトから chat 形式への移行、(b) 数値崩壊からのリカバリ、(c) 新しい base policy への進歩継承、(d) reward hacking trace の filter-out。

    • O(1M) traces が teacher 並みに到達するには十分。多すぎるとポリシー分布が狭まり、その後の RL の探索余地を奪う。
    • incorrect trace を含めても、successful のみと結果はほぼ同等。最終的に successful のみを使用。
    • late checkpoint からの trace のみだと弱い。複数の strong checkpoint からの diverse な trace が最も良い。
    • 固定 token bugdet では prompt 多様性 > prompt あたり trace 数。random sampling が biased sampling を上回る。
    • Self-distillation 中に long-context 能力を忘れがちなので、mid-training data を mix する。

    主要ハイパラ

    • RL：AdamW（β1=β2=0.95、ε=10⁻¹⁵）、weight decay なし、constant LR 10⁻⁶（長 generation 時は 9×10⁻⁷）、global batch 7040（packed）、unpacked 最大 12000。
    • 5 gradient step ごとに inference を更新、最大 8 推論更新分の stale rollout は破棄（=40 gradient step）。
    • MoE load balancing 係数：RL 時 1×10⁻⁵、self-distillation 時 1×10⁻²（事前に expert 分布を整える）。

    SHA-256 → character n-gram MinHash-LSH → embedding コサイン類似度で、内部ベンチ・Olympiad・graduate STEM eval からも除外。

    2-3. Agentic Climb

    外部環境と相互作用するエージェントタスクを学習。ReAct 流の policy step ↔ environment step ループを Sandbox Execution Environment（SEE）上で実行。trajectory 全体を verifiable / judged な reward で採点し、token-level credit assignment を均一に適用。

    STEM と agentic を 同時に climb させると agentic 側が安定化し SWE / Tool 性能にも正の transfer が確認された（逆方向は中立）。

    2-3-1. Software Engineering（SWE）

    実リポジトリベースのコンテナ環境。tool は Bash と String Replace Editor の 2 つのみで完結。

    環境ビルドパイプライン（PR → 学習環境）
    1. 収集：public GitHub PR 1.02 億件から、main にマージ・15 ファイル未満・コード変更とテスト変更が両方含まれる PR を抽出 → issue リンクで絞り込み 487 万 PR。
    2. Agentic environment build：LLM agent が repo を読んで Dockerfile を生成、テスト suite を実行して dependency error を弾く。
    3. Reference grading 抽出：base commit に対し pre-fix（test diff のみ）と post-fix（test + code diff）の 2 段で実行。F2P（issue-resolution signal） と P2P（regression signal） を抽出。F2P が残らないものは破棄。
    4. 環境＆grader 検証：empty patch が落ち、golden patch が通ることを multiple trials で再確認。non-deterministic test も除外。
    5. 品質フィルタ + 書き換え：仕様の曖昧さ・leakage・実現性を agent が採点し、低品質なら問題文を書き換え。

    2-3-2. General Tool Use

    OpenAI 互換 function-calling schema をベースに、1 環境あたり 50 ツール超 も扱う stateful な mocked backend を用意。

    • 合成パイプライン：(i) tool 記述・関数実装・DB seed → (ii) tool 呼び出し trajectory のサンプリングと user request 生成 → (iii) 実行検証と過剰類似タスクの除去。150+ 環境、130K+ task。
    • Reward：environment-specific grader（最終 state、ツール使用パターン、最終回答）+ cross-environment grader（parallel tool calls、duplicate 回避、有効引数）。

    2-4. Helpfulness and Safety Climb

    Reward の組み合わせ

    Reward Model

    MAI-Base-1 を post-train した RM が k-way side-by-side 評価をテキスト出力。inference 時は サイクリックに k 回呼び出して permutation バイアスを除去。

    AI Judge

    rubric-guided の prompted judge。RM の再訓練を待たずに振る舞いを素早く変えるレバー。

    Verifiable Reward

    「single paragraph で答える」「10 語未満」のような制約を直接検証。reward hack や multi-epoch に強い。長さは 許容分位範囲を外れたら penalize。

    Reward の優先順位付け
    • Lexicographic shaping：上位 reward でタイの rollout 内でのみ下位 reward を発火させる。スケールに不変。
    • Gated application：上位 reward が最低水準に達しないと下位は計上されない。「unsafe な良文」は許さないのがその典型。

    Instruction Following / Steerability

    • 専門家が書いた指示と、constraint taxonomy + 40+ ドメインを seeds とした合成データを併用。System / Developer / User の階層に従う訓練を含む。
    • 難易度を pass-rate でコントロール、難しすぎず簡単すぎないものだけ残す。
    • Reward は rule-based verifier＋atomic rubric を判定する LLM judge＋RM を lexicographic 集約。

    Safety

    2 つの failure mode に対処：unsafe compliance（拒否すべきものを許可）と over-refusal（合法な要求を拒否）。

    • prompt は Harmful（refusal / partial refusal）と Borderline（do-not-refuse）に分類。
    • Harmful は Human red-teaming（vendor + 社内）と Automated（PyRIT、PAP、TAP）で収集、Borderline は do-not-refuse スライスと capability data から。
    • Reward は policy compliance / engagement / style の 3 軸で judge し、safety-gated aggregator（非 compliant なら他軸を問わず最低）。
    • 注意：scalar reward の単純加重では 87.8% の非 compliant 応答が RM ≥ 3 を獲得してしまうため、必ず gating が必要。

    Honesty

    既知の事実は正しく答え、知らないときは適切に hedge。data は (a) 既知の factual、(b) long-tail factual、(c) false-premise（誤前提）の 3 種。LLM judge が factuality と confidence の 2 軸を CONFIDENT_CORRECT / UNCONFIDENT_CORRECT / NOT_ATTEMPTED / UNCONFIDENT_INCORRECT / CONFIDENT_INCORRECT の 5 段階に分類し、信念付き正答に最高報酬・信念付き hallucination に最大 penalty。

    Style

    Style guide を策定し、emoji 使用・table/markdown・トーン・layout・density をルール化。grade は 0/1/2 の coarse な LLM judge（細かすぎる rubric は hack されやすい）。

    2-5. Consolidation（3 teacher → 1 model）

    3 つの specialist teacher を SFT で 1 モデルに統合する。STEM/Agentic teacher は 複数 checkpoint からの rolloutで多様性を確保、Helpfulness teacher は LLM judge + heuristic でフィルタ。

    2-6. RL インフラ ― Rocket

    Rocket は MAI 社内製の大規模非同期分散 RL フレームワーク。Learner は YOLO、Inference は SGLang + SGLang router。コントローラ、Problem Worker、Rollout Worker から成る。

    flowchart LR Q["Problem Set Sampler"] --> Ctrl["Async Controller"] Ctrl --> PW["Problem Workers"] PW --> RW["Rollout Workers"] RW --> IS["Inference Pool
    SGLang Router + Servers"] IS --> RW RW --> PW PW --> Ctrl Ctrl --> LR["Learners (YOLO)"] LR -.weight transfer.-> IS LR --> CK["Checkpointing"]

    Inference 設計

    • 大型 RL では inference : learner = 5 : 1 に達する。最大ジョブでは 4864 GB300 のうち 4096 が inference、768 が learner。
    • Single-turn workloads：KV cache が bottleneck → EP + DP、prefix cache off（sliding window 完全 evict）、DeepEP、EPLB。
    • Multi-turn workloads：prefill 主体 → prefix cache hit rate 97–98%。
    • システム安定性は 3 層 defense：(i) 各 SGLang サーバの self-watchdog、(ii) router の circuit breaker、(iii) job 全体の liveness monitor + step-progress watchdog。失敗した rollout は別 replica に retry。

    Learner と Inference の数値ギャップ

    • YOLO（学習）と SGLang（推論）の kernel / scheduling 違いから出る log-prob 乖離は importance-sampling 補正を不安定化する。
    • 両側で BF16 を使い、MoE routing replay と top-p mask replay を併用して mismatch を最小化。

    Weight Transfer

    • Learner と inference で異なる sharding を結ぶため、resharding / 通信 / per-tensor 変換を job 開始時に 1 度コンパイルした transfer planに集約。
    • plan は「1 learner ↔ 1 inference server」の idealized topology で記述し、runtime で replica の出入りに応じて transfer group を拡張するため replica の追加・脱落・置換に対しても plan が valid。
    • DP はパラメータを複製するだけなので、すべての DP rank を動員する必要はなく、36 サーバを 4 分割して 9 サーバ × 4 並列で transfer。

    4. Safety Red Teaming

    4-1. 内部 Red Teaming

    • 開発サイクル中、計 15 回の engagement で 2,170+ シナリオ × 25 ポリシーカテゴリを 5–10 ターンでテスト。
    • 主な攻撃パターンは 6 種：(i) 善意の口実下での multi-turn escalation、(ii) フィクションのフレーミング、(iii) 資格職人(credentialed persona) の偽装、(iv) 漸進的な再帰・format drift、(v) 文脈中の年齢指標バイパス、(vi) 権威ある文書の捏造。
    • これらは個別 prompt ではなく pattern が durable な攻撃面。以降のサイクルでは カバレッジ・チェックリストとして使う。
    • fiction-framing による self-harm が突出して目立ったため、多言語版含めた seed-expansion で SFT/RL の safety mix を強化。
    • 主要な改善幅：aggregate ASR ≈ −22%、jailbreak ≈ −44%、hate & fairness ≈ −43%、child safety ≈ −30%、mental health ≈ −20%。

    4-2. 独立 Red Teaming

    • Microsoft AI Red Team（AIRT）と外部ベンダーが、static eval が弱い領域（自動敵対攻撃、サイバー悪用、心理社会的・メンタルヘルス、多言語）を担当。
    • TAP（Tree of Attacks with Pruning）に脆弱性が見つかったため、シナリオ生成 → 攻撃変換 → TAP refinement の closed-loop pipeline を構築し、SOTA 並みに改善。
    • 低リソース言語（Yoruba / Telugu / Amharic / Burmese / Khmer / Malay）で英語では拒否される内容が出力される問題に対し、多言語 adversarial seed を追加して有意に縮小。long tail の言語は今後の継続課題。

    5. クラスタ環境と推論基盤

    5-1. Training Cluster

    MAI-Base-1 は単一サイトの 8K GB200 GPU クラスタで学習（NVL72 rack、cross-rack InfiniBand）。実験ノイズを抑えるため accelerator generation、rack health、scheduler、storage path はすべて同質に揃えた。

    5-2. Stability / Determinism / Goodput

    • Goodput = 理想 training duration / 実 wall-clock duration。これを 第一級の運用 KPIとして全 overhead カテゴリにオーナーと検出経路を割当。
    • visible failure（crashloop、node failure、link flap、OOM、checkpoint stall、manual requeue）と silent loss（MFU degradation、recomputation、長 startup、scheduler 遅延、network/memory 劣化）の双方を追跡。
    • MAI-Base-1 pre-training は 8K GPU で goodput 90.0%、total overhead 51 時間。Recomputation は 6.5 時間（overhead の 15%）、non-stepping time 14 時間（27%）まで圧縮。MFU drop overhead が最大残課題（18h, 35%）。

    5-3. 推論効率と MAIA-200 デプロイ

    MAI-Thinking-1 は Microsoft の自社 AI アクセラレータ MAIA-200上で実装され、同じ rack 電力予算下で GB200 ベース比 40% 以上のトークン生成スループットを達成。性能/Watt 改善により大規模 inference の効率的な提供が可能に。

    5-4. Sustainability ／ コミュニティ投資

    • 2025 年に Microsoft はグローバル年間電力消費の 100% を再エネ調達でマッチ。
    • MAI-Thinking-1 は主に Phoenix, AZの LEED Gold 認証データセンターで学習。バックアップ発電機は再生可能ディーゼル燃料。
    • Community-First AI Infrastructure として、5,000 万ドル超を Phoenix 地域の水資源（市営貯水施設、上下水道延伸、Gila River Water Storage、Nature Conservancy 連携など）に投資。
    • Estrella Mountain / Glendale CC で Datacenter Academy を運営し人材育成も推進。

    6. 結論と今後

    「MAI-Thinking-1 はゴールではなく出発点である」。Microsoft AI は、データ・インフラ・RL レシピ・評価のすべてを最適化する hill-climbing machine を、より多くのモダリティ・より大きなスケール・洗練された能力に拡張していく。

    AI の進歩は単一モデルではなく、「信頼できる形で改善できるパイプライン」の産物であるという思想が、レポート全体を貫いている。

    7. 付録 A：事前学習データパイプライン詳細

    A.1 Web HTML

    • 独自クローラで 約 1.2 兆ページ取得 → policy / adult / blocklist / UT1 list / AI 生成検出で 794B → 423B → 73.4B 英語 + 116.5B 非英語に削減。
    • Common Crawl は約 300B → 100B → 最終 24.2B。最後に proprietary と統合し URL/コンテンツの fuzzy dedup を追加。
    • 全 Web ページに Qwen3-Embedding-0.6B で embedding を計算し、用途別パイプラインへ。

    用途別パイプライン

    • 一般 Web：属性モデル（教育価値・事実性・writing 品質・情報密度・推論内容・知識性）+ quality モデルで 下位 70% を drop。Gopher-style heuristic 適用後、英語 4.6B（proprietary）+ 2.8B（CC）。さらに長文プロ品質コーパスを human-curated surface（雑誌・読み物・SNS 共有リンク）から構築。
    • STEM：topic（7 binary classifier）× educational value × educational level の embedding 分類。式・コードを保持するため MathML/LaTeX を正規化 → Markdown 変換 → セクション単位で LLM が keep/remove。OpenWebMath 流の後処理。英語 STEM 680B（うち数学 76B） + 多言語 760B（うち数学 58B）。
    • Code：STEM パイプライン流に CS topic 抽出。Qwen3-30B + GEPA/DSPy で最適化した judge を 2K 人手ラベルで微調整し品質スコアリング。約 233B トークン。
    • Key domain：Wikipedia など重要源は 包含寄りで抽出。

    A.2 Web PDFs

    • クロール約 100 億ドキュメント → ヒューリスティック分類で 6.2 億 → Azure Document Intelligenceで OCR。
    • OCR 後処理：式／表正規化、boilerplate 除去、glyph error 修正、line unwrapping、参考文献節除去。
    • SEO/スパム除去には PDF メタデータの "creator" / "producer" フィールドが高精度信号として有効。
    • Fuzzy dedup 後で 英語 1.8T + 多言語 1.85T tokens。educational/non-educational に分類し、Math / CS / STEM / non-STEM × 高校／学部／院 にラベル付。

    A.3 Books & Journals

    プロバイダごとに専用 ingestion pipeline で正規テキストへ。OCR 整形 → ヒューリスティック → exact / title / fuzzy dedup → LLM がトピックと品質ラベルを付与して mixing に活用。

    A.4 Public GitHub

    • 7.4T トークンを Files / Commits / PRs の 3 データセットに編成。
    • 共通処理：junk フォルダ除外、生成物（_pb2.py、.d.ts 等）の検出除外、非コード除外、30K 文字超過 reject、SHA-512 exact dedup、MinHashLSH fuzzy dedup、Qwen3-Embedding-0.6B 1024 次元 cosine による semantic dedup、推論訓練・評価との decontamination、品質 bin。
    • Files：StarCoder2 流フィルタ、repo の depth-first で連結 → 1.26T tokens。
    • Commits：repo ごとに最新 10K commit。pre-commit state を loss-mask prefix に、patch を git-diff or search-and-replace 形式で学習対象に → 4.5T tokens。
    • PRs：title / body / linked issues / reviews / threads / pre-commit state / constituent commits を結合。SWE-bench Verified の PR は除外。長すぎる prefix は線確率的に圧縮、256K 超は破棄 → 1.19T tokens。

    8. 付録 B：長文脈拡張（Long Context Extension）

    256K の context window を達成するために、16K で pre-train → 64K で mid-train → 256K に短く拡張する staged アプローチを採用。

    B.1 データ

    long-doc up-weighting / domain ratio 変更などを試したが、いずれも目立った改善はなく、結局 前段と同じ mixture を 256K で再 packするのが最良だった。

    B.2 評価設定

    • Code NLL：社内 repo の 256K チャンク。末尾 16K に対する prefix を 16K → 256K まで伸ばし、関連情報がどれだけ NLL を下げるかを測定。
    • Retrieval NLL：standup ノート 32K を A/B に分割、間に無関係な設計文書を挿入して 256K まで延長。距離が伸びても NLL がフラットであれば長文脈に強い。
    • Generative QA：内部リポジトリドキュメント上の QA。証拠位置と総文脈長で精度を測る。

    B.3 Progressive Context Length Scaling

    • 16K / 32K / 64K / 128K で 1T tokens の mid-training を別々に行うのと、32K で 1T tokens → 256K に約 100B tokens 追加で延長するのを比較。
    • 後者（progressive）が 128K context での NLL を 1T tokens の 128K mid-training と同等に達成。短文脈 quality も損なわない。
    • ＝ 「短い MFU-friendly な context で midtrain し、最後だけ短く延長」でよい。

    B.4 Adaptation Speed

    256K NLL の改善は extension iteration の最初の 1〜10% で大半が起きてプラトーに到達。新しい能力を獲得しているのではなく、位置・attention 機構の校正をしているだけ。

    B.5 Final Recipe

    最終モデルは 64K mid-training + 140B tokens の 256K 拡張。さらに長い窓（1M トークン）にも同じ staged アプローチが modest cost で適用可能と見込んでいる。

    9. 付録 C：推論トレースの進化（CoT Evolution）

    蒸留に頼らずゼロから climb したからこそ、CoT の振る舞いがどう進化するかを clean に観察できる。

    C.1 STEM CoT ― 「Weak guess、Strong work hard」

    AIME 2024 の $f(x) = \frac{(x-18)(x-72)(x-98)(x-k)}{x}$ が正実数で最小を取る k を求めよ という問題（正解 240）に対し：

    • 弱い model：最小化点を「18, 72, 98」から推測し、$k=40, 152, 512$ といった spurious な候補を生む。
    • 強い model：4 つの代数候補 $k \in \{8, 32, 200, 512\}$ を derive し、$x>0$ ドメイン条件で 512 を除外、$8+32+200=240$ に到達。

    違いは 最終 verification step での domain check にある。RL climb によってこのような 「最後まで丁寧に検証する」振る舞いが獲得される。

    C.2 Agentic CoT

    Agentic 側でも、climb が進むにつれ 計画立案 → 道具呼出 → 観察 → 再計画のループがより簡潔で目的指向に変化する例が観察される。

    10. 付録 D〜F：エージェント関連ツール／スキーマ／インフラ

    D. SWE Agent Tool Schema

    • Bash tool：bash コマンドを実行して stdout/stderr を返す。
    • String Replace Editor：ファイルを「exact 文字列の置換」で編集（bash の非エルゴノミックな編集を回避）。
    • これら 2 つだけで SWE エージェント環境を構成。

    E. Instruction Following 制約 Taxonomy

    論文では合成データ生成のシード taxonomy を、format（文字数・段落数・JSON など）・content（言語・トーン）・behavioral（指示優先順位、refusal）など多軸で整理。

    F. SWE 環境ビルド用インフラ

    • repository の build / test sandbox を SEE（Sandbox Execution Environment）上に大規模並列で展開。
    • network egress policy / CPU・メモリ制限 / 実行タイムアウトを統一して再現性を担保。
    • 環境のメタデータ・grading script・golden patch を中央 store に永続化し、後段で再検証可能に。

    12. 付録 K：クラスタ環境の詳細

    K.1 Hardware

    NVIDIA H100 / GB200 / GB300 を併用する heterogeneous fleet。Pre-training run は 単一サイトの GB200 クラスタ。GB200/GB300 は Microsoft first-party DC に NVL72 ラック単位でホスト、ラック内 72 GPU NVLink + ラック間 InfiniBand。H100 系は 8 GPU node。

    K.2 Cluster Readiness / Certification

    ノードは「provisioning されただけでは production プールに入れない」設計。3 段認証：

    1. Single-node diagnostics：GPU/CPU/HCA/NVLink/メモリを長時間 stress、ECC・throttle・clock 違反・link flap を検出。
    2. Rack-level multi-node NCCL collectiveで intra/inter node 通信を検証（健全ノードが 16 未満のラックは大規模ジョブから除外）。
    3. Cross-rack InfiniBandテストで spine / path diversity / RDMA 性能を確認（blast radius が大きいため on-demand）。

    ノードは Init → Certifying → Available → Impaired → Auto Remediating / Vendor Maintenance → Decommissioned という閉ループの lifecycle で管理。

    K.3 Scheduling / Orchestration / Control Plane

    • Kubernetes + Kueue（priority/quota/admission/preemption/topology-aware placement）+ MAI カスタムコントローラ + Ray の階層構造。
    • ラック断片化対策として soft rack reservation：キューに preferred rack を割当てつつ、必要時は reclaimWithinCohort で取り戻せる。
    • Ray actor を learner / inference server / rollout worker / router 用に明示的に配置・監視。

    K.4 Observability / Telemetry / Fleet Monitoring

    • GPU XID・ECC・thermal・power・NVLink BER・IB device state・PCIe error 等を Kubernetes node condition に変換し schedule / drain / remediate に直結。
    • Job 状態は Kueue / K8s / Ray / training log / 実験 metadata を横串で参照可能。namespace / pod / job / restart index で絞り込み可能。
    • Telemetry スタックは Datadog（near-real-time）、Azure Managed Prometheus、Azure Data Explorer（long retention）を時間軸別に分担。
    • 「ダッシュボードは制御ループの一部」という設計思想。

    本レポートは単に MAI-Thinking-1 という 1 つのモデルの紹介ではなく、「Microsoft AI が公開した最初の本格的 from-scratch 推論モデル」と「その背後にある hill-climbing machine」を、データ・アーキ・分散学習・RL・評価・安全・インフラまで含めて開示した technical disclosure である。
