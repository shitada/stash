    3 行サマリ

    • このセッションの中心メッセージは、AI 時代の開発者生産性を 活動量ではなく成果 で測るべきだということ。コード行数、PR 数、トークン消費量は行動変化を見る補助指標ではあるが、目標にすると簡単に歪む。
    • Microsoft の EngThrive は、開発体験を Speed（速さ） / Ease（楽さ） / Quality（品質） の 3 軸で捉え、「fast and easy to build great work」を目指す。個人を採点する仕組みではなく、エンジニアが動くシステムのボトルネックを測る枠組みである。
    • 事例として、Focus Time 改善では 8 週間で 1人あたり週 2.1 時間の集中時間を回復し、Time to First PR 改善では First Mate エージェントなどにより新メンバーの初回 PR 到達を 1 週間未満へ圧縮した。

    1. セッションの狙い

    冒頭で登壇者は、AI の価値提案には大きく 2 つあると整理する。ひとつは、AI を製品に組み込み、顧客体験を変えること。もうひとつは、AI によって開発者自身の働き方を変えること。このセッションは後者、つまり 「AI を使ってソフトウェア開発の仕事そのものをどう変えるか」 に焦点を当てている。

    登壇者は、AI によって仕事の進め方が大きく変わるなかで、組織が「エンジニアは何倍コードを書けるようになったのか」という問いに飛びつきやすいことを認める。しかし、それは不安に対する分かりやすい答えであって、変革を進めるための有用な答えではない。むしろ、活動量を目標化すると逆効果になる。

    2. なぜコード量では生産性を測れないのか

    セッションは、業界が長年かけて「コード行数」や「PR 数」を個人の生産性指標として使うべきではないと学んできたにもかかわらず、AI ブームのなかで同じ発想が戻ってきている、と警告する。AI によってコード生成量が急増するからこそ、コード量はさらに危うい指標になる。

    Microsoft の縦断研究では、成熟した組織における開発者の時間は、おおまかに次の 3 つへ分かれる。

    時間の分類おおよその割合説明
    Innovation約 40%新しい価値や機能を作る時間。
    Run the business / Keep the lights on約 45%既存システムの維持、運用、改善、移行、依存関係対応など。
    Organizational responsibilities約 15%会議、研修、組織市民として必要な活動。

    さらに深掘りすると、エンジニアが技術システムと直接やり取りする時間はその一部であり、実際に能動的にコードを書いている時間は週の 10〜15% 程度だという。つまり、コードを書く時間だけを速くしても、ソフトウェア開発全体のボトルネックを見誤る可能性が高い。

    3. AI による開発ループの変化

    登壇者は、AI と開発者の関係を 3 つの波として説明する。ChatGPT 的な対話から始まり、非同期タスクを任せるエージェント、そして複雑な目標を持つエージェントとの協働へ進んでいる。

    flowchart LR A["Wave 1
    質問と回答"] --> B["Wave 2
    非同期タスクとツール使用"] B --> C["Wave 3
    複雑な目標、記憶、権限、ツールを持つエージェント"] C --> D["開発者の仕事
    計画、検証、判断、設計、品質責任へ移動"]

    従来の SDLC は、計画、作成、検証、デプロイ、運用で構成される。平均的には、運用が 75〜80%、作成が 10〜15%、残りが計画・検証・デプロイに割かれる。しかし、AI エージェントを高いレベルで使うチームでは、時間とエネルギーが 計画と検証 に大きく移り、コード作成の時間は急速に縮小しているという。

    コードは、これまでシステムへの入力であり、真実の源泉だった。しかし今後は、コードがシステムの出力になっていく。

    ただし、AI の進歩は SDLC 全体で均等ではない。単一行のコード生成は非常に強くなっているが、システム設計、持続可能なアーキテクチャ、検証、運用、そして「taste（良い判断・審美眼）」はまだ人間の役割が大きい。特に taste は、優れたエンジニアが今後も持ち込む重要な価値だと強調された。

    4. EngThrive: fast and easy to build great work

    EngThrive は、Microsoft が AI 時代の開発者生産性を理解し、改善プログラムを設計するために使っているフレームワークである。そのミッションは 「fast and easy to build great work」。速く、楽に、優れた仕事を作れるようにすることだ。

    flowchart TD A["EngThrive"] --> B["Speed
    Idea to Customer"] A --> C["Ease
    Innovation Time / Focus Time"] A --> D["Quality
    Defect Escape / Resilience / Experience"] B --> E["価値を速く届ける"] C --> F["開発者が流れに乗れる"] D --> G["顧客体験と信頼を守る"]
    軸意味代表的な成果指標
    Speedアイデアが顧客価値になるまでの速さ。Idea to Customer、Time to First PR、Time to nth PR。
    Ease開発者がどれだけ摩擦なく仕事に集中できるか。Innovation Time、Focus Time、Bad Developer Day。
    Quality速さや楽さを犠牲にせず、製品品質と顧客体験を保つこと。Defect Escape Rate、Incidents per PR、Mean Time to Mitigate、顧客体験指標。

    3 軸はトレードオフとして扱わない。速さだけを上げて品質を落としたり、品質だけを強めて開発者体験を悪化させたりするのではなく、3 つを同時に引き上げることを目指す。

    5. 活動指標ではなく成果指標を見る

    セッションで最も重要な概念として語られたのが、activity vs outcome である。活動は「何をしたか」を表し、成果は「何を達成したか」を表す。活動は motion、成果は progress と言い換えられる。

    種類例使い方危険な使い方
    活動指標AI 利用率、トークン消費量、PR 数、agent-generated PR 数、コード行数。行動や働き方の変化を観察する補助情報として使う。目標値や評価指標にする。token maxing や意味のない PR 量産を誘発する。
    成果指標Idea to Customer、Innovation Time、Focus Time、Defect Escape、Time to First PR。システムのボトルネックを特定し、改善対象として扱う。単独で個人の人事評価に使う。文脈なしに比較する。

    活動指標を目標にすると、開発者は簡単にそれを最大化できる。例えば、無意味なコードを大量生成し、数百行ごとにコミットすれば、トークン消費とコード行数は増える。しかし価値は増えない。これが活動指標を目標にする危うさである。

    一方、成果指標を「攻略」すると、望んでいた成果が生まれる。例えば Time to First PR を短くするために、最初の PR が no-op だったとしても、開発環境、権限、CI、レビュー、デプロイ経路を通過できたこと自体に価値がある。

    6. 個人ではなくシステムを測る

    EngThrive は個人を採点するための仕組みではない。開発者が働く システム を測るための仕組みである。例えば、ビルドが遅い、環境構築が複雑、レビューの待ち時間が長い、会議が分断的に入るといった問題は、個人の能力ではなくシステムの摩擦である。

    登壇者は、従業員パフォーマンス評価はもっと複雑で別の話だと明確に切り分ける。生産性の測定は、チームや組織がより良く働けるように、システムを改善するためのものである。

    7. 代表的な成果指標

    Idea to Customer

    Speed の代表指標。アイデアが生まれてから、それが顧客の手元で価値を生むまでの暦日・時間を測る。アイデアは PRD、チケット、設計案などとして始まり、実験、canary、デプロイ、顧客利用までを含む。この全体を測ることで、どこにボトルネックがあるかを分解できる。

    Innovation Time

    Ease の代表指標。開発者が新しい価値を作るために使える時間の割合を測る。登壇者は、39% 程度からできるだけ高く、80〜90% に近づけることを理想として語った。AI により、これまで長年当然とされてきた運用・移行・雑務のボトルネックを壊せる可能性がある。

    Quality

    品質は比較的よく理解されている領域として、Defect Escape Rate、Incidents per PR、Mean Time to Mitigate、顧客体験や製品愛着を測る指標などが挙げられた。重要なのは、速度と楽さを上げるときにも品質を一緒に見ることである。

    8. EngThrive の運用原則

    指標を持つだけでは変革は進まない。登壇者は、EngThrive を実際の組織改善プログラムとして動かすための運用原則を説明した。

    1. ボトルネックに集中する

    SDLC のどこで価値提供が詰まっているかを見つけ、そこを改善する時間と責任を作る。

    2. リーダーに責任を持たせる

    開発者体験は現場任せではなく、最上位レベルのリーダーシップが説明責任を持つ。

    3. ベンチマーク達成ではなく継続改善

    組織変更やチーム差で数字は揺れる。罰するためではなく、継続的に改善するために見る。

    4. 共通言語と共通指標

    会社全体で同じ定義の指標を使うことで、比較と学習と再利用が可能になる。

    特に強調されたのは、company-wide consistent metrics の重要性である。チームごとに指標定義がばらばらだと、組織全体のボトルネックを理解できない。リアルタイムに近いダッシュボード、ビジネスレビューのリズム、成功パターンの共有が必要になる。

    9. ケーススタディ1: Focus Time 改善

    1つ目の事例は、開発者の Focus Time を改善するプログラムである。Focus Time は、会議などで分断されず、深く考え続けられる時間を指す。開発者が flow に入るための時間として、Microsoft では 120分の連続ブロック を重視している。

    8週間
    改善プログラムの期間
    +2.1h
    1人あたり週の Focus Time 回復量（P50）
    +13%
    PR velocity の増加（活動変化の補助指標）
    -25%
    Bad Developer Day telemetry の減少
    55,000h
    会議 hygiene 改善などで回復したエンジニアリング時間の換算

    この改善は、AI ツールそのものを導入したからではなく、AI による分析で得たデータを使い、組織運営を変えたことで実現された。流れはシンプルで、まずシニアリーダーが 1 つの指標を選び、各リーダーに明確な責任を割り当て、2週間ごとのレビューでデータ、仮説、施策、結果を確認した。

    また、Focus Time を奪っていた運用作業やシステム健全性タスクに AI を適用する動きもあり、これが SRE agent の起点になったと説明された。SRE が退屈な作業に追われるのではなく、より深いシステム理解や信頼性設計に集中できるようにする狙いである。

    10. ケーススタディ2: Time to First PR 改善

    2つ目の事例は、新入社員やチーム移動者が、そのチームで最初の PR を出すまでの時間を短縮する取り組みである。Microsoft 全体で多くのエンジニアに関わる指標であり、目標は 1週間未満 だった。

    施策内容狙い
    First Mate エージェントDay 0 から新メンバーに付き、所属チームや作業環境を理解したうえで、最初の PR から 10件目、30件目までの道筋を支援する。オンボーディングを個人の探索に任せず、文脈付き支援にする。
    PR ワークフロー文書の改善PR 作成、レビュー、CI、デプロイの手順を整備する。人間だけでなく、First Mate が参照できる質の高い grounding data にする。
    動画演習社内固有ツールやフローの視覚的な使い方を示す。新しい環境で迷いやすい内部ツールの理解を速める。
    マネージャーへの期待値設定全社リーダー向けの場で、新メンバーが早く開発体験に入れることの重要性を共有。オンボーディング速度を個別努力でなく組織責任にする。

    この指標は「no-op の PR で簡単に攻略できるのでは」という疑問を呼びやすい。しかし登壇者は、それでよいと答える。no-op であっても、ローカル環境が動き、権限があり、インフラにつながり、レビューと CI とデプロイ経路を通れたという事実には価値があるからだ。成果指標を攻略すると、狙った成果そのものが起きる。

    Time to First PR、Time to 10th PR、Time to 30th PR は、その後の開発活動、innovation time、idea to production の速さとも強く相関する。最初の摩擦を減らすことは、短期のオンボーディング改善にとどまらず、長期の成功軌道に影響する。

    11. Q&A の要点

    Teams メッセージやデイリースタンドアップは Focus Time を壊すのか

    Teams メッセージは、リモートチームにおける通常の会話や協働として機能するため、Focus Time を壊すものとしては単純にカウントしていない。ただし、通知設定やベストプラクティスにより、不要な割り込みを減らすことは推奨している。

    スタンドアップや一部の recurring meeting については、会議の性質、参加人数、リズム、タイトルなどから working meeting として検出できるものもある。重要なのは、会議をブロック化し、IC が午後以降を会議で細切れにされないようにすること、そして会議には agenda、purpose、outcome が必要だと明確にすることだった。これらが不明確な会議は断ることを期待として伝えている。

    Focus Time は何分必要か

    業界研究では、flow に入るための連続時間として 60分、90分、120分という考え方がある。登壇者は、過去には 60〜90分派だったが、Microsoft での研究と成果を見て、現在は 120分の連続ブロックを重視していると説明した。

    活動指標はまったく使わないのか

    使わないわけではない。AI 利用量、トークン消費、coding velocity などは、チームが AI 活用のどの波にいるか、行動がどう変化しているかを見る補助情報として有用である。ただし、それを目標や評価にした瞬間に、指標としての意味を失う。

    経営層の理解をどう得るか

    登壇者は、IC や第一線マネージャーにも想像以上に力があると答えた。現場のエンジニアは「正しいことをするのが難しすぎる」「speed、ease、quality を上げるにはこのボトルネックを直す必要がある」と具体的に声を上げられる。AI により、かつては大規模チームでなければ解けなかった問題を、小さなチームが短期間で改善できるようになっている。

    12. 実践するなら何から始めるか

    1. 活動量ではなく成果を選ぶ。 コード量やトークン量ではなく、Idea to Customer、Focus Time、Time to First PR、Defect Escape など、システム改善につながる指標を選ぶ。
    2. 最初は 1 つの指標に絞る。 組織全体に一気に多くの指標を投げず、開発者調査や現場の痛みから最も効きそうな指標を選ぶ。
    3. リーダーのレビューリズムを作る。 2週間ごとなど短い周期で、データ、仮説、施策、結果を確認する。
    4. 個人評価ではなくシステム改善として説明する。 数字を罰に使うと、組織は防御的になり、指標も歪む。
    5. AI は万能薬ではなく、古いボトルネックを壊す手段として使う。 文書理解、オンボーディング支援、運用作業削減、データ分析など、低摩擦で効果が大きい場所から始める。

    13. まとめ

    このセッションは、AI ツールの導入率やコード生成量を自慢する話ではない。むしろ、AI 時代だからこそ、開発者生産性をより厳密に、より人間的に、よりシステム的に捉える必要があるという話である。

    EngThrive の核は単純で、速く、楽に、良い仕事を作れるようにすることだ。そのためには、活動ではなく成果を測り、個人ではなくシステムを改善し、継続的にボトルネックを解く必要がある。AI はそのための強力な道具だが、測るべきものを間違えると、より速く間違った場所へ進むだけになる。
