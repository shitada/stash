    • Doc-to-LoRA（D2L）は、長い文書を「ハイパーネットワークで1回のフォワードパス・サブ秒」処理して LoRA アダプタに変換し、以後はその文書をコンテキストに載せずに質問へ回答できるようにする手法（＝近似的なコンテキスト蒸留の償却）。
    • 公式リファレンス実装が MIT で公開済み。model.internalize(doc) → model.generate() → model.reset() という3つの呼び出しだけで、最小構成なら学習済みモデルをダウンロードしてすぐ試せる。
    • 公開後わずか4ヶ月で Code2LoRA・Frames2LoRA・Skill-to-LoRA・Doc-to-Atom など 20本超の「X-to-LoRA／パラメトリックメモリ」派生研究が登場し、研究ジャンルとして急拡大している。

    1. Doc-to-LoRA とは ― 背景と位置づけ

    大規模言語モデル（LLM）の文脈内学習・文書理解・多段推論では「長い入力系列」が中心的な役割を果たす。しかし Transformer の注意機構は系列長に対して二乗（quadratic）のコストを持つため、長文を扱うほど推論はメモリを食い、遅くなる。

    この問題に対する既存の打ち手にはそれぞれ難点がある。コンテキスト蒸留（Context Distillation, CD）は文脈情報をモデルのパラメータへ転写できるが、プロンプトごとに蒸留する「per-prompt 蒸留」は学習コストとレイテンシの面で非現実的だった。

    Doc-to-LoRA（D2L）は、この限界を「コスト償却（cost amortization）」で乗り越える。すなわち、メタ学習で「文書 → LoRA」を生成するハイパーネットワークを一度だけ訓練しておけば、本番では重い学習パイプラインの代わりに1回の安価なフォワードパスで、その文書専用の LoRA を即時生成できる。著者は Text-to-LoRA（自然言語のタスク記述から LoRA を生成）と同じグループで、D2L はその「文書版」にあたる。

    「我々はメタ学習のコストを一度だけ払い、タスクや文書ごとの LoRA をオンデマンドで生成できるハイパーネットワークを訓練する。これにより、かつては重いエンジニアリングのパイプラインだったものが、単一の安価なフォワードパスになる。」 — Sakana AI 公式ブログ（2026-02-27、要約引用）

    RAG・ICL・CD との関係

    D2L は「文脈をどこに置くか」という観点で、既存手法と次のように整理できる。重要なのは、D2L が RAG や ICL を置き換えるというより、「重み空間に文脈を持つ」第三の軸として相補的に位置づけられる点である（この相補性は後述の派生研究でも繰り返し主張される）。

    手法文脈の置き場所推論時コスト更新の容易さ
    ICL（文脈内学習）プロンプト（注意）系列長に応じて増大（二乗）即時（だが毎回再消費）
    RAG（検索拡張）検索→プロンプト取得文の分だけ増大索引更新で容易
    従来の Context Distillationパラメータ（蒸留）推論時は軽いプロンプト毎に蒸留＝高コスト
    Doc-to-LoRAパラメータ（LoRA）推論時は軽い・文書は窓外1フォワードパスで即時生成

    2. 基本的な仕組み

    D2L の核心は「ハイパーネットワークが、対象 LLM 向けの LoRA を文書から直接予測する」点にある。通常の LoRA ファインチューニングが勾配で反復的に重みを更新するのに対し、D2L は勾配反復なしに重み（LoRA）を1パスで生成する。

    2-1. 「内在化（internalize）」の流れ

    使う側から見た動作は3段階に集約される。公式リファレンス実装の API もこの3つに対応している。

    flowchart LR Doc[長い文書 / 文脈] --> HN[ハイパーネットワーク
    D2L] HN -->|1パスで生成| LoRA[文書専用 LoRA 重み] LoRA --> Base[凍結ベース LLM に適用] Q[ユーザの質問] --> Base Base --> Ans[回答
    文書はコンテキストに不要]
    • internalize（内在化）：文書をハイパーネットワークに通し、LoRA を生成してベースモデルへ適用。以後、その知識は「重み」に宿る。
    • generate（生成）：質問だけを与えれば、内在化済みの知識に基づいて回答する。元文書をプロンプトに載せ直す必要がない＝KV キャッシュとレイテンシを節約。
    • reset（解除）：内在化を取り消すと、ベースモデルは元の状態に戻る（内在化前は文書について幻覚する）。

    2-2. なぜ訓練できるのか ― 近似コンテキスト蒸留のメタ学習

    D2L は要旨で「1回のフォワードパスで近似的なコンテキスト蒸留を行うことをメタ学習する」と説明される。概念的には、次のような「教師／生徒」の蒸留をメタ学習していると理解すると分かりやすい。

    flowchart TB subgraph Train[メタ学習（オフラインで一度だけ）] D[多数の 文書・QA ペア
    ＝自己生成の合成データ] --> T[教師: 文書を文脈に入れた LLM] D --> S[生徒: 文書なし＋生成 LoRA の LLM] T -->|振る舞いを一致させる| Loss[蒸留ロス] S --> Loss Loss -->|更新| HNW[ハイパーネットワークの重み] end HNW --> Use[本番: 未知文書も1パスで LoRA 化]

    つまり「文書を文脈に入れたときの振る舞い」を、「文書なしで LoRA だけを当てたモデル」が再現できるよう、ハイパーネットワークを訓練する。一度訓練すれば、未知の文書に対してもゼロショットで LoRA を生成できる。リファレンス実装では、この訓練に自己生成（self-generated）の合成データを用いており、データ生成は一度行えばよい設計になっている。

    2-3. 主な実験結果

    4〜5倍
    NIAH でほぼ完璧に解けた、ベース文脈窓に対する系列長
    サブ秒
    文書 → LoRA 生成のレイテンシ
    CD超え
    実 QA で標準コンテキスト蒸留を上回りつつ省メモリ・低遅延
    • Needle-in-a-Haystack（NIAH）：文書中の「針」情報を LoRA に格納する写像を学習し、ベースモデルのネイティブ文脈窓を4倍（論文）〜5倍（ブログ）超える系列長でも、ほぼ完璧なゼロショット精度を達成。
    • 実世界 QA（計算制約下）：標準のコンテキスト蒸留を上回り、ピークメモリ消費と更新レイテンシを大幅に削減。
    • クロスモーダル転送：視覚言語モデル（VLM）の視覚情報を text-only LLM へ転送し、画像を「内在化した重みだけ」で分類することにも成功。

    狙い：著者は D2L を「頻繁な知識更新」や「個人化されたチャット挙動」を可能にする、LLM の高速適応への一歩と位置づけている。

    3. 自分で実現する最小構成の手順

    D2L は公式リファレンス実装（Python 93.8%、MIT ライセンス、ICML 2026 採択版）が公開されている。ここでは「(A) 学習済みモデルを動かす」最小ルートを中心に、「(B) 自分で学習する」「(C) 概念を最小に理解する」を順に示す。

    3-A. 学習済みモデルを動かす（最短ルート）

    パッケージ名は ctx_to_lora（context-to-LoRA）。uv で環境を作り、学習済みハイパーネットワークを取得して、3呼び出しで内在化を体験できる。

    ① インストール

    curl -LsSf https://astral.sh/uv/install.sh | sh
./install.sh

    ② 学習済みモデルの取得（Hugging Face の SakanaAI/doc-to-lora）

    uv run huggingface-cli login
uv run huggingface-cli download SakanaAI/doc-to-lora \
    --local-dir trained_d2l --include "*/"

    ③ Python API で内在化 → 生成（非バッチ入力向けの最小例）

    import torch
from ctx_to_lora.model_loading import get_tokenizer
from ctx_to_lora.modeling.hypernet import ModulatedPretrainedModel

# 学習済みハイパーネットワーク（Gemma デモ）をロード
checkpoint_path = "trained_d2l/gemma_demo/checkpoint-80000/pytorch_model.bin"
state_dict = torch.load(checkpoint_path, weights_only=False)
model = ModulatedPretrainedModel.from_state_dict(
    state_dict, train=False, use_sequence_packing=False
)
model.reset()
tokenizer = get_tokenizer(model.base_model.name_or_path)

# 文書と質問を用意
doc = open("data/sakana_wiki.txt").read()
chat = [{"role": "user", "content": "Tell me about Sakana AI."}]
chat_ids = tokenizer.apply_chat_template(
    chat, add_special_tokens=False,
    add_generation_prompt=True, return_tensors="pt",
).to(model.device)

# ★文書を「重み」に内在化（1フォワードパス）
model.internalize(doc)

# 文書をプロンプトに載せずに回答できる
outputs = model.generate(input_ids=chat_ids, max_new_tokens=512)
print(tokenizer.decode(outputs[0]))

# 内在化を解除（reset すると元のベースモデルに戻る＝文書について幻覚する）
# model.reset()

    GUI で試したい場合はインタラクティブデモも同梱されている。

    uv run demo/app.py

    3-C. 概念を最小に理解する（自作する場合の構成要素）

    仮にフルスクラッチで最小再現を考えるなら、必要な部品は次の4つに集約される。これは 2-2 の蒸留メタ学習をそのまま部品化したもの。

    ① 凍結ベース LLM

    重みは更新しない。LoRA を当てる対象（Gemma / Mistral 等）。

    ② ハイパーネットワーク

    文書（文脈）を読み、対象モジュールの LoRA 重み（ΔW = BA）を出力する。

    ③ 合成 文書・QA データ

    多数の（文書, 質問, 望ましい回答）。自己生成で用意。

    ④ 蒸留ロス

    「文書ありの教師」の振る舞いを「文書なし＋生成LoRAの生徒」が再現するよう最小化。

    最小の心得：まず 3-A で学習済みモデルの internalize → generate → reset を体感し、次に 3-B の NIAH スクリプトで小さく訓練を回すのが、つまずきの少ない順序。フルスクラッチ（3-C）は最後でよい。

    4. X-to-LoRA 派生研究 ― 公開後4ヶ月の広がり

    D2L の「文脈 → LoRA を1パス生成」という発想は、公開後すぐに多方向へ展開された。被引用論文を辿ると、20本超の派生研究が確認でき、「X-to-LoRA／パラメトリックメモリ」という研究ジャンルが立ち上がりつつある。以下、方向性ごとに整理する。

    4-1. 直接の改良・堅牢化

    研究arXiv何を進化させたか
    Doc-to-Atom（Doc2Atom）2606.12400D2L の「単一モノリシック adapter」が抱える無関係クエリ干渉・合成的想起の弱さ・長文スケール性を解決。文書を意味型付きの「知識アトム」に分解し各々を micro-LoRA 化＋検索キー付与、クエリ時に関連アトムだけを動的合成。6つの QA で D2L 超え・省メモリ。
    The Override Gap2604.23750D2L の弱点を定量分析：文書が事前学習知識と矛盾すると深い事実で精度が 46.4% へ崩壊。原因を「振幅（margin）不足」と特定し、訓練不要の Selective Layer Boosting / Conflict-Aware Internalization で 46.4%→71.0%（Gemma-2B）・53.6%→72.5%（Mistral-7B）に改善。KID-Bench（489問）公開。
    SHINE2602.06358凍結 LLM 自身のパラメータを in-context ハイパーネットに再利用し、少パラメータで高品質な context→LoRA 写像を実現。

    4-2. 別モダリティ・別ドメインへの展開

    研究arXiv内容
    Code2LoRA2606.06492リポジトリ知識を LoRA に注入し推論時のトークン増をゼロに。Static（スナップショット）＋ Evo（コミット差分ごとに GRU で adapter 更新）。RepoPeftBench（604リポジトリ）で per-repo LoRA 上限に匹敵、Evo は cross-repo +5.2pp。
    Frames2LoRA / Video2LoRA2606.04351動画をパラメータに内在化。Perceiver ハイパーネットが VLM の層ごとの中間表現を読み LoRA を1パス生成。クエリ時の視覚トークンを最大1,500分の1、TTFT を6〜80倍改善。非重複セグメントの adapter がランク空間で合成可能。
    PRISMR2606.12942多モーダル listwise ランキングで、軽量ハイパーネットが項目別 LoRA を生成し instance 固有 adapter に合成。長文脈で起きる「parse collapse（候補の取りこぼし）」を抑制。

    4-3. エージェント／スキルの内在化（SKILL.md 運用に直結）

    特に注目したいのがこの方向。エージェントのスキル文書（SKILL.md など）を、毎回コンテキストへ注入する代わりに LoRA 化する研究で、トークン削減・秘匿性・合成性で利点がある。スキルを APM で配布する運用とも親和性が高い。

    研究arXiv内容
    Skill-to-LoRA（S2L）2606.16769SKILL.md をランタイムに毎回注入する代わりに、スキル専用 LoRA に置換。スキル文が誘発する「行動変化」をモデル化（文書圧縮ではない）。Qwen3.6-27B / SWE-Skills-Bench（21スキル）で pass 率 +2.9〜+5.2pp、ステップあたりトークン −6.6%。Wrong-LoRA/Shared-LoRA は性能低下＝スキル固有 adapter の整合が効いている証拠。
    LatentSkill2606.06087テキストスキルをプラグ＆プレイ LoRA に変換。スキルを文脈空間でなく重み空間に格納。ALFWorld success +21.4/+13.4pt・prefill トークン −64.1%、Search-QA exact match +3.0pt・skill トークン −72.2%。生成スキル LoRA は構造化された意味幾何を成し、パラメータ空間の算術で合成可能。

    5. 課題と限界

    導入を検討する際は、派生研究が明らかにした以下の制約を押さえておきたい。

    • 知識衝突に弱い：文書が事前学習知識と矛盾すると精度が崩れる（深い事実で 46.4%）。Override Gap が振幅増幅で部分的に解決。
    • 単一 adapter の干渉：1文書につき1つのモノリシック adapter だと、無関係クエリで干渉し合成的想起が弱い。Doc-to-Atom が「知識アトム」分解で対処。
    • 多反復での能力崩壊：継続的に経験を内在化し続けると劣化しうる。原則レベル経験・step-wise 注入が緩和。
    • 基盤研究の途上：容量・合成性の限界は解明途上（Parametric Memory Law、Understanding LoRA as Memory が定量化に着手）。

    6. まとめ ― 実務への示唆

    Doc-to-LoRA は「長文をコンテキストに置き続ける」前提を覆し、文脈を重み空間へ即時に移すという選択肢を実用ラインに引き上げた。RAG・ICL を置き換えるのではなく、第三のメモリ軸として相補的に使うのが現実的な見立てである。

    まず試すなら

    3-A の internalize → generate → reset。学習済みモデルを落として体感するのが最短。

    自社ドメインへ

    NIAH スクリプトで小さく訓練 → メイン実験へ。データ生成は1回でよい。

    エージェント運用

    Skill-to-LoRA / LatentSkill は SKILL.md の重み化。トークン削減と秘匿性に効く。

    長文コスト削減

    QA・コード・動画で「文脈を窓外へ」。Code2LoRA・Frames2LoRA が好例。

    とりわけ、スキルを APM で配布・選択導入する運用とは Skill-to-LoRA / LatentSkill の相性がよく、「テキストとして読み込むスキル」と「重みとして当てるスキル」を将来併用する設計余地がある。
