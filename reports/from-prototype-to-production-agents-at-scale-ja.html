    1. Agent 時代の 3 つの変化

    Tina Sharkey は、AI agents の新しい時代に入ったとして 3 つの変化を提示する。第一に、building はもはや難所ではない。GitHub Copilot、GitHub Copilot CLI、Claude Code などにより、強力な agent prototype は数分で立ち上がる。問いは「作れるか」から「enterprise scale で信頼して動かせるか」へ移った。

    第二に、agent capability は爆発的に広がった。静的 router が固定 tool へ request を振り分けるだけではなく、agent は新しい agent を spawn し、compute environment にアクセスし、新しい skill や memory を作り、最初にプログラムされていなかった仕事を進められる。

    第三に、agent は tool ではなく teammate になりつつある。chatbot ではなく、business outcome を担い、人だけでなく他の agent とも会話しながら長時間動く operator として扱う必要がある。

    2. Foundry が目指す production learning loop

    Microsoft は Foundry、Azure、GitHub、Microsoft 365、Microsoft Security を、agent 変革のための holistic system として位置付けている。重要なのは、AI systems は「完成して終わり」ではなく、production learning loop で進化し続けるという考え方だ。

    flowchart LR A["GitHub / Copilot CLI
    agentを作る"] --> B["Evals
    goodを定義"] B --> C["Foundry
    production workloadで実行"] C --> D["Traces / cost / action signals"] D --> E["Evaluation + Optimizer"] E --> F["prompt / skills / tools / model / memory を改善"] F --> C

    これは単一 model の fine-tuning ではなく、model、harness、context、memory、tools、skills を含む system 全体を改善する考え方である。変更の最終判断は developer が握り、agent は実行されるほど賢く、安全で、安くなるように調整されていく。

    3. デモシナリオ: fiber outage response agent

    デモの題材は、Microsoft の cloud infrastructure に近い scenario である。data center 近くで fiber cable が切断された場合、sensor が outage を検知し、agent が起動する。agent は必要な情報を探し、field rep を派遣し、D365 に ticket を作り、Teams に status を流す。

    flowchart TD A["Sensor detects fiber outage"] --> B["Agent wakes up"] B --> C["Foundry Toolbox
    tools / policies"] B --> D["Fabric IQ
    site reliability / location"] B --> E["Work IQ
    supplier conversations"] B --> F["Document Intelligence
    work orders / supplier agreements"] C --> G["Dispatch field rep"] D --> G E --> G F --> G G --> H["D365 ticket"] G --> I["Teams status update"]

    セッション全体は、この agent を build / deploy / operate の 3 フェーズで成長させる構成になっている。

    4. Build: VS Code から agent を作る

    Jeff Hollan は、Foundry Toolkit for VS Code を使って agent を作る流れを実演する。開発者は Claude Code、GitHub Copilot CLI、Cursor、Visual Studio など好きな tool を使えるが、デモでは VS Code + GitHub Copilot + Foundry Toolkit を使った。

    構成要素役割
    Foundry Toolkit for VS Codeagent 作成、model、tracing、evaluation を editor 内に統合。Foundry best practices を coding agent skill として組み込む。
    Microsoft Agent FrameworkPython で agent を構築。multi-step / multi-agent workflow、skills、memory、middleware、harness を扱う。
    Agent Framework harnessagent が shell command、file read/write、code execution を安全な harness 内で実行できるようにする。
    GitHub Copilot SDK / Claude Agent SDK integration追加 harness として plug-in 可能。

    デモでは、「field operations team が fiber optic line を修理するための情報を扱う agent」を Copilot に生成させる。agent は field technician が現場で必要な仕様、supplier contract、site reliability data、Teams / Outlook conversation などへアクセスする。

    5. Tools: Foundry Toolbox と context の統合

    Jeff が特に覚えてほしい feature として挙げたのが Foundry Toolbox である。Toolbox は、agent が使う tools を 1 つの managed endpoint にまとめ、authentication、governance、policy、guardrails を集中管理する。

    Tool / Contextデモでの使い方
    Foundry IQsupplier contract や agreement documents を indexed knowledge として参照。
    Fabric IQsite reliability data、uptime、location information を参照。
    Work IQTeams / Outlook の supplier conversation history を参照。
    Web IQ / web search必要に応じて web context を tool として提供。
    Content UnderstandingPDF contract や specification から tables、markdown、figures、raw JSON を取り出し、agent-readable format に変換。

    Toolbox には tool search もある。agent の context window を無駄にしないよう、MCP-compatible endpoint に問い合わせた時、現在の task に relevant な tools だけを返す。これにより、agent はその瞬間に必要な tools に集中できる。

    6. Local debug と voice-enabled agent

    VS Code 内では F5 debug で agent を local host 上に起動できる。breakpoint を置けば request / response、streaming events、tool invocation を追跡できる。デモでは、Quincy North Site の B-side panel に必要な connector type を field technician が質問し、agent が toolbox 経由で specification を参照して回答した。

    さらに Foundry Agent Service に deploy した agent は、voice mode を有効化するだけで voice-enabled にできる。Foundry が industry-leading voice models を wrap し、WebSocket 経由で real-time に response を stream する。現場作業員が手袋をしていて chat UI を使いづらい場面でも、音声で agent とやり取りできる。

    Build phase の発表: Microsoft Agent Framework 1.0 production-ready、Foundry Toolkit for VS Code GA、Toolboxes in Foundry GA soon、Skills の first-class toolbox integration、Voice Live integration は prompt agents で GA / hosted agents で public preview、Hosted Agents in Foundry Agent Service GA soon。

    7. Deploy: Hosted Agents、Routines、secure session

    deploy フェーズでは、Fiby という network operations coordinator agent が紹介された。Fiby は telemetry を監視し、routine によって定期的に wake up し、anomaly があれば investigation log、skills、subcontractor dispatch、human approval などを実行する。

    Hosted Agents の重要な特徴は、conversation や routine ごとに isolated session を作れることだ。各 session は専用 workspace を持ち、file system state を保持しつつ、他の session と隔離される。これにより、subcontractor A の context が subcontractor B の session に漏れるような問題を防ぐ。

    Routines

    agent を reactive から proactive に変える。cron 的に、または event 的に起動し、Foundry が queue / execute / track する。

    Sandbox isolation

    session ごとに isolated workspace を持ち、file system や execution state を隔離する。

    Scale to zero

    agent が idle の時は compute cost を払わず、再開時に state を復元する。

    Human-in-the-loop

    approval 待ちの間は durable task が状態を持ち、承認後に session を resume する。

    8. Teams / Microsoft 365 Copilot へ公開する

    Foundry agent は、Teams や Microsoft 365 Copilot に公開できる。デモでは Fiby が Teams 上の autonomous agent として動き、自分の identity、email address、Teams presence を持つ様子が紹介された。ユーザーは Teams で「今対応中の active incidents は何か」と聞き、同じ Foundry 上の agent session が resume または start して回答する。

    機能状態・意味
    Publish to Microsoft 365 Teams and CopilotGA soon。Foundry agents を Teams / Copilot へ公開し、identity、policy、permissions を引き継ぐ。
    Autopilot agents in TeamsPublic preview。Azure IDs、email address、Teams presence を持ち、会話開始や follow-up を行える。
    Agent 365 governanceautopilot agents を end-to-end で統治する。

    9. Operate: Trace replay と observability

    operate フェーズでは、agent を本番で信頼できるものにするための visibility が示された。Foundry では conversations 一覧から個別 request を開き、trace replay view で agent の trajectory を確認できる。

    trace replay では、最初に model reasoning が何秒かかり、次にどの tool を呼び、どの input / output が流れたかを時間軸で追える。さらに token perspective へ切り替えれば、どの段階で token が消費されたかも見える。再生ボタンにより、ユーザーが見た応答の流れを 8x speed などで追跡できる。

    sequenceDiagram participant U as User participant A as Hosted Agent participant M as Model participant T as Toolbox participant O as Observability U->>A: Ask question A->>M: Reason A->>T: Tool call T-->>A: Tool output A-->>U: Answer stream A-->>O: Trace / tokens / tool I/O

    10. Evals と custom rubric

    Jeff は、voice-enabled field operations agent の回答が bullet list になり、音声で聞くと robotic だったという実例から optimization を始める。まず `azd ai agent eval init` で agent project を evaluation 用に初期化する。

    多くのチームは「eval が大事」と言いながら eval dataset を持っていない。Foundry は historic traces や agent の signal から、LLM を使って初期 eval dataset を提案できる。また、tool selection、tool output、retrieval、fluency などの built-in evaluator に加え、scenario-specific な custom rubric を生成できる。

    rubric 項目例意味
    Correct tool usesite uptime なら Fabric IQ など、適切な tool を選べているか。
    Safety warningfield operations で安全注意を落としていないか。
    Voice optimized conciseness音声で聞きやすく、冗長な bullet list になっていないか。
    Weighted scoring重要項目に高い weight を与え、agent の「good」を明示する。

    11. Agent Optimizer と continuous improvement

    `azd ai agent optimize` は、system prompt、skills、tool descriptions、tool configuration、target model などを候補変数として扱い、evaluation dataset と custom rubric に対して候補を探索する。デモでは、voice conciseness を強く評価する rubric に合わせ、system prompt などを調整した candidate が 11% の改善を示した。

    flowchart TD A["Production traces"] --> B["Eval dataset generation"] B --> C["Custom rubric\nweighted criteria"] C --> D["azd ai agent optimize"] D --> E["Candidate variants\nprompt / tools / skills / model"] E --> F["Compare quality / cost / latency"] F --> G["Developer chooses winner"] G --> H["Promote or rollback"]

    Tina は、Agent Optimizer for Foundry Agent Service が private preview であり、production traces と evals から ranked candidates を作り、quality、cost、latency で side-by-side に比較できると説明する。platform が候補を見つけ、developer が最終的な winning variant を選ぶ。

    12. Procedural memory と導入事例

    Procedural memory in Foundry Agent Service は public preview で、agent が session をまたいで playbook を学べるようにする。毎回ゼロから学ぶのではなく、過去の run で得た手順や改善を使い、agent がより賢く、安全で、安くなる。

    Foundry は既に 80,000 以上の customers が利用していると紹介された。Iberdrola は 14か国で mission-critical energy operations を scale し、identity、memory、governance、observability を Foundry Agent Service で使っている。Twilio は Twilio Agent Connect を hosted agents on Foundry で展開し、KPMG は global KPMG Workbench platform を hosted agents 上に構築し、Foundry Toolbox の tools / skills で client interaction を強化している。

    13. 実践上の持ち帰り

    フェーズ実践ポイント

    DeployHosted Agents で session isolation、scale-to-zero、durable state、人間承認を設計する。
    ReachTeams / Microsoft 365 Copilot へ公開し、ユーザーが既に働いている場所で agent を使えるようにする。
    Operatetrace replay、token view、tool I/O、OpenTelemetry で「なぜそう答えたか」を追えるようにする。
    Improveeval dataset、custom rubric、Agent Optimizer、procedural memory で production learning loop を回す。

    このセッションが示したのは、agent production は「agent を書く」ことではなく、agent が企業の中で安全に動き、必要な context と tools を使い、長時間仕事を続け、観測され、評価され、改善され続ける system を作ることだという点である。
