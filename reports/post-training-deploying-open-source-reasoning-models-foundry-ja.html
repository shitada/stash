    3 行サマリ

    • エージェントはトークン大食いでスケールするほどコストが膨らむ。フロンティアモデルの代わりに、本番トレースから学んだ小型オープンソースモデルをポストトレーニングすれば、品質を保ち（または向上させ）つつコストを大幅削減できる。
    • 鍵は「ループを閉じる」こと：本番投入 → トレース収集 → データセット化 → SFT（模倣）＋ RFT（検証可能タスクの強化学習）→ 評価（カスタム eval で hill climb）→ 再デプロイ、を継続的に回す。デモでは小型モデルが 76% 未満から目標の 90%+ へ。
    • Foundry は コードファースト（Ray クラスタ + slime/verl/TRL）と ローレベル API（コード名 "loom"、ノート PC の CPU から GPU ジョブを起動）の2経路を提供。トレーニング済みモデルは Foundry Managed Compute でサーバーレスにデプロイ・自動スケールできる。

    1. なぜポストトレーニングか — エージェントのコスト問題

    Chris Lauren は、モデルを本番投入することは「始まりにすぎない」と説く。あらゆるソフトウェア同様、エッジケースでもスケール時に高品質を保つには継続的な学習と改善が必要になる。特にエージェントはトークンを大量消費する。チャットボットと違い、エージェントは複数の反復的なツール呼び出しを行い、企業システムやデータベースから取得したデータを推論し、ポリシーを検証し、次に何をすべきか「考える」ためだ。スケールすればコスト（請求額）もスケールする。

    解決策がポストトレーニング（ファインチューニング）。これによりモデルに「自社の望むやり方で考え・行動する」ことを教えられる。ビジネスドメインだけでなく、ツールの使い方やシステムとの連携も教えられ、コストのかかる誤った対話を排除できる。

    フロンティアモデルは賢いが、多数のツールを前に「どれをどう使うか」を推論・軌道修正する過程そのものがコストになる。ファインチューニングで最適なツールを正しい順序で選ばせれば、最良の結果を最低コストで得られる。

    Foundry はモデル選定・ツール整備・eval の取り込みを一体で支援する。「より速く・より良く・より安く」を、継続的に学習するエージェントとして実現する。

    2. モデル選定と評価（eval）— 科学的手法

    最初のステップは「どのモデルがワークロードに適切か」を見極めること。新モデルが次々登場し複雑化するが、Foundry はあらゆるモデル（オープンソース／プロプライエタリ）を継続的に追加している。正しいモデル選定の鍵は科学的手法 = guess and check（仮説と検証）であり、その検証が評価（evaluation）である。

    デモは小売カスタマーサービスのエージェント（顧客が最近購入した商品を返品したいシナリオ）。フレンドリーな対話、ポリシー遵守、注文照会などが必要。これを GPT-5 系のフロンティアモデルでも、小型のオープンソースモデル Qwen3 14B でも実行できる。Foundry は本番監視に必要な全テレメトリを収集し、個々のトレースの詳細（ツール呼び出しの往復、推論ループ、各ステップで増えていくコスト）まで掘り下げられる。

    eval は「プロダクト仕様（product spec）」。本番投入の前に成功基準を定義する。品質バーを満たすだけでなく、ポリシー遵守・正しいツール呼び出しなどを含む多次元最適化。Foundry には組み込み evaluator とカスタム evaluator がある。

    発想の転換 — 「監視」から「定義」へ

    従来は「本番投入後にうまくいっているか監視する」発想だった。これをひっくり返し、先に『うまくいっている状態』を定義する。そうすれば継続的に hill climb（坂を登るように改善）して高品質な出力に到達できる。

    ベンチマークは「誰かが公開すると決めた情報」にすぎない。出発点としては良いが、自分のシナリオ・自分のデータでテストしなければ本番の成功基準は満たせない。カスタム evaluator こそが、自分にとって意味のある形で hill climb を可能にする。

    デモでは複数バージョンのエージェントをカスタム eval でテストし、統一指標「retail quality」を定義。初期バージョンは「モデル品質自体は良くても、このエージェント文脈・自分の品質定義では振るわない」ことが判明した。

    4. ポストトレーニングの位置づけ — 3つの学習段階

    「ポストトレーニング」は OpenAI や Anthropic などのラボが広めた言葉で、ファインチューニングや RFT を包含する傘的な用語。どんなモデルも3つの独立した段階を経て準備される。3段階は連続ではなく、どこからでも開始・継続でき、各段階で「何をするか」と「コスト」が重要になる。変えられる要素はデータ・計算（compute）・モデルのパラメータ（アーキテクチャ）の3つ。

    段階内容データ / 計算 / パラメータ
    事前学習（pre-training）世界について学び「知的」になる。全重みが変化。桁違いに多いデータ・計算。巨大なパラメータ。
    （中間段階の）特化学習言語・コーディングなど専門知識を注入。特化した知識の付与。
    ポストトレーニング（post-training）必要な「仕事」（例: エージェンティックワークフロー）を上手にこなせるようにする。本セッションの主題。自社のルール・手順をモデルに教える。

    ベースモデルがいくら良くなっても、あなたが関わらせたい固有のルール・手順は知らない（サポートチケット処理、製造・医療の特定シーケンスなど）。ラボと同じ技術を自分の環境で適用してこれを教える。

    5. ポストトレーニングの構造 — データ × スキル × レシピ

    ポストトレーニングは「データソース」と「獲得したいスキル」を組み合わせ、レシピを選んで一度に1つずつ最適化していく。

    flowchart LR subgraph D[データソース] D1["本番エージェントの
    トレース（蒸留）"] D2["人間の選好
    （RLHF）"] D3["合成データ
    （synthetic）"] end subgraph S[獲得したいスキル / 目標] S1["安全性
    （safer）"] S2["特定タスクの精度
    （coding / agentic）"] S3["コスト削減
    （cheaper）"] end D --> R["レシピを選択"] --> S

    本セッションのレシピは トレースからの蒸留（distillation）→ SFT → RFT の組み合わせ。目的は2つ：モデルを正確・タスク完遂的にしつつ、安価にすること。大型モデルは優秀だが「正解を探索する」ためにパラメータも計算も巨大でコストがかかり、CLI やコーディングのタスクで知らぬ間に $100万〜$500万を浪費する例も出ている。ポストトレーニングはこれを削減する手段。

    SFT（教師ありファインチューニング）— 模倣

    専門家（人間、または専門家が作ったデータ）のパターンを真似させる。例えばカスタマー応答エージェントに「トーンはこう、フォーマットはこう」と教える。仕組みは「実演（demonstration）」を与え、モデルの答えが正解の専門家の答えにどれだけ近いかを損失関数（loss）で採点し、完全一致／部分一致の双方に報酬を与えてモデルの重みを正しい挙動へシフトさせる。「何をすべきか（模倣）」を教える。

    RFT（強化ファインチューニング）— 経路探索

    SFT より桁違いに複雑だが「最も費用対効果が高い」。コードや数学のような検証可能（verifiable）なタスクに有効。例えば「返金を成立させる」には多数のポリシー・シーケンスを正しい順序で踏む必要がある。RFT は各ステップに報酬を与えることで、モデルが単に模倣するのではなく正解に至る経路（path）を自力でナビゲートすることを学ばせる。本デモは蒸留付き SFT と RFT を組み合わせる。

    6. ループを閉じる — トレースからデータセットへ

    Chris の説明した複数エージェントのうち、強力な GPT-5.2 を使うエージェントが最良の性能を出している。「最良モデルのやり方から学ぶ」ため、その本番トレースからデータセットを作成し、蒸留（高価な大型モデルから蒸留）して SFT / RFT を起動できる。

    flowchart LR A["エージェントを
    本番投入"] --> B["本番トレースを
    収集・観察"] B --> C["トレースを蒸留・
    フィルタしデータセット化"] C --> D["SFT で模倣を学習"] D --> E["RFT で経路探索を学習"] E --> F["監視・評価
    （カスタム eval）"] F -->|hill climb| A

    これが「ループを閉じる（closing the loop）」継続改善。本番に置き、挙動を観察し、トレースを取り、ファインチューニングする、を繰り返す。

    7. 経路①コードファースト — Workbench / Ray クラスタ

    Foundry はトレーニングジョブの起動に2つの経路を提供する。1つ目がコードファーストのカスタムジョブ。

    • クラウド上に Workbench（compute インスタンス）を確保。マシン・ワークベンチが設定済みで、依存関係の pip インストール等を気にせず「connect」すると VS Code が開きすぐコードを反復できる。
    • レシピ：ベースモデル＋エージェントのトレースから開始 → SFT → RFT → 監視・評価。トレースは本番から全件取得し、必要なものへフィルタ・蒸留してデータセット化（既定の自動処理も、自前処理も可）。
    • RFT では各プロンプトについて複数サンプルをロールアウトし、モデルの応答に応じて1つ選び、軌道（trajectory）を辿って報酬を与える。
    • 構成要素：呼び出すビジネス API、選ぶグレーダー（graders）、GPU クラスタ（ノード数＝ジョブの速さ）。SDK を入れ、ジョブを設定して起動。

    Ray による分散と可視化

    RFT ジョブは認証済みの Ray エンドポイント（Ray クラスタ）で実行され、Ray が Python コードを多数のノードに分散する。Ray ダッシュボードで全ノードの稼働・CPU/GPU リソース使用、各アクター・各メソッド/関数の挙動まで完全に可視化できる。「fire and forget の API ではなく」、データサイエンティストやデベロッパーが各ノード・各クラスタで何が起きているかを精密に追える。

    ロールアウトとグレーディングの観察

    報酬が増えていく様子や、各ロールアウトの採点理由を確認できる。例：あるサンプル（prompt 8）は必要な関数の呼び出しを取りこぼしたため報酬が低い。良いランとツール呼び出しの並びをサイドバイサイド比較でき（例: rollout 16 で「キャンセル理由を探す」ツールを正しく呼べたものと呼べなかったもの）、なぜ高く／低く報酬されたかを理解できる。

    評価で改善を確認

    学習したモデルを eval にかけると、各サンプルの pass rate（品質）が分かる。デモは「GPT-5.2 がベースモデルとしては 14B より良い」一方、はるかに安く・速い小型モデルに蒸留＋SFT＋RFT のポストトレーニングを施すとスコアが段階的に上昇することを示した。ポストトレーニング済みモデルは未処理モデルより pass rate も報酬も高い。各段階は個別にも、ループでも実行でき、監視しながら目標へ近づける。

    8. 経路②ローレベル API — コード名 "loom"

    コードファーストはクラスタのプロビジョニング、Ray の管理、コード持ち込み、インフラ・ネットワークトポロジ・データストリーミング性能のデバッグなど専門的で重い作業を伴う。誰もが GPU を持つわけではない。そこでより簡単なローレベル GPU 構成の API（コード名 "loom"、正式名称未定）を発表。

    • ノート PC の CPU 上に自分の RL ロジックを書いてジョブを起動すると、サーバー側で実行され、GPU クラスタ・イメージ・環境・ネットワーク等すべてを loom が管理する。
    • ユーザーが扱うのはコアな GPU プリミティブ（forward pass / backward pass による損失最適化）だけ。クラスタの仕組みを気にする必要はない。左側（CPU）でクライアント、右側（GPU クラスタ）で実行され、エクスポート・管理すべきは4つの呼び出し。
    • 制御を持つ利点の例：モデルに数学を教えるなら、いきなり最難問ではなく加減算から徐々に線形代数・微積へ hill climb させる、といったカリキュラムをクライアント側で設計できる。
    • クラスタ不要・高い制御性。ノート PC のノートブックにコードを持ち込むだけで SFT も RFT も実行可能。

    Foundry 上での見え方

    経路Foundry 上の表示位置づけ
    コードファースト（トレーニングハーネス/canvas）Train 配下（カスタムトレーニング）事前学習・特化学習も含め非常に強力

    いずれもログ・チェックポイント（学習中の全チェックポイントを後でデプロイ可能）を確認でき、ダッシュボードでランを比較できる。報酬関数の上昇、エントロピーの低下、group composition の上昇など、コードファーストと同等のリッチなテレメトリを API ファーストでも得られる。コードファーストとローレベル API はともにプライベートプレビューで、サインアップ用リンクが提供された。slime・verl・TRL などの RL フレームワークを利用できる。

    9. デプロイ — Foundry Managed Compute

    「トレーニングしてもデプロイしなければ意味がない」。トレーニング済みモデルをエージェントに組み込むには、API として動くようデプロイする必要がある。Build で発表された Foundry Managed Compute は、サーバーレスでスケーラブルなデプロイを実現する。

    • サーバーレス学習モデル、ノート PC や他プラットフォームで学習したモデルの重みを持ち込んでデプロイ可能。Foundry が自動最適化・スケールアウト。
    • カスタムコンテナで投機的デコーディング（speculative decoding）やドラフトモデルなど高度に最適化されたランタイムを持ち込み、推論を高速化することもできる。

    デプロイの実際（Manoj Bableshwar）

    モデルカタログからベースモデルもファインチューニング済みモデルもデプロイする。ベースモデルはまず eval のためにデプロイして性能を見る。

    デプロイ方式特徴
    Provisioned Throughput Unit（PTU）Fireworks 系モデル等向けのマネージド提供。保証スループット SLA で予測可能な性能。
    Managed ComputeFoundry マネージド GPU にデプロイ。GPU 課金のため運用・支払い方法を選べる。
    • 例: 32B モデルを Managed Compute で。2 GPUならコンテキスト長を最大化する十分なキャッシュ付き、1 GPUなら最小フットプリント（約 40K コンテキスト長）。
    • ファインチューニング済みモデルはジョブ完了時に自動登録される。別環境で学習したモデルはプロジェクトにインポート可能。その際ベースモデル（アーキテクチャ）の指定が最重要で、正しいコンテナ・デプロイ構成の特定に使われる。モデル名・バージョン・フルウェイト/アダプタの別を指定するとコマンドが生成され、コピペ実行でインポートできる。
    • ベースモデルとファインチューニング済みモデルのデプロイ手順は同一（global managed compute 選択、アクセラレータ選択など）。

    エージェント統合と観測性

    デプロイ済みモデルは Playground で試せ、コードからも呼べる。chat completions API はステートレスで状態は自前管理が必要だが、モデルをエージェントに追加すると状態管理やツール呼び出しなどの機能が得られる。デモではファインチューニング済みモデルをエージェントに追加し、「今日の NBA の試合」などリアルタイム情報のツール呼び出しに応答。responses API 互換のため、チャット履歴を渡さず直前のレスポンスを渡すだけで「好きな色」を記憶して回答できた。

    デプロイ後は観測性・監視・コスト管理が重要になる。Managed Compute ではコスト管理が箱から出してすぐ使える。総消費トークンなどのメトリクスを追跡してダッシュボードに追加でき、モデル（例: Qwen）でフィルタしてコストを監視できる。

    10. まとめ

    Chris のクロージング。本セッションの学び。

    豊富なモデル

    Foundry にオープンソース／プロプライエタリの全モデル。エージェントに追加し、自分のワークロードで評価する。

    足りなければ鍛える

    価格対性能が十分ならデプロイ。不足なら SFT/RFT で継続的に反復・改善。

    2つの RL 経路

    ローレベル API のフル制御コード（slime/verl/TRL）と Ray による分散・監視。ロールアウトとサンプルの品質を精査できる。

    コスト効率デプロイ

    サーバーレス Foundry Managed Compute で 1〜多数 GPU、コンテキスト長別オートスケール。

    そして最も重要なのは、Foundry が全テレメトリを自動収集すること。コスト監視にも、本番から学ぶ新データセット作成にも使え、デプロイ後もモデルを継続改善できる。サンプルリポジトリは公開済みで「今夜から始められる」。スライドは他セッション同様の場所から後で入手できる。
