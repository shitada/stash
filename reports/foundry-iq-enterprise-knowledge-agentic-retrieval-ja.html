    • Foundry IQ は、エージェントを「あらゆる場所の知識」に接続する Microsoft のコンテキストエンジニアリング基盤。統合層 / 知識検索層 / コア検索エンジンの3層に整理され、どの層からでも使え、後から上下に移動できる。

    1. Foundry IQ とは — 3つの設計目標

    Pablo Castro は冒頭、Foundry IQ を「エージェントを知識に接続するためのもの」と定義する。AI モデルやエージェントを、アプリケーション・会社・必要なデータへつなぎ、仕事を完遂させるための基盤である。設計時に重視した要素は3つ。

    ① 簡単に始められる

    立ち上げが「朝飯前（piece of cake）」であること。ファイルをドラッグ&ドロップして質問する程度の手軽さ。

    ② 多用途（versatility）

    アプリが高度化し、要求が増え、問題が難しくなっても、同じプラットフォーム内に必要なものが揃っている。

    ③ 最高品質

    AI 駆動の全コンポーネントで高品質。特に検索ではランキング・関連性・コンテンツ理解に優れる。

    2. 最速デモ — ファイルから1分で知識ベース＆MCP サーバー

    「簡単さ」を実証するため、Pablo は Foundry の build > knowledge から新しい知識ベースを作成。Wikipedia から取った映画記事ファイルを「Movies wiki」という名前でドラッグ&ドロップし、エージェントが用途を判断できるよう説明（「映画に関するフル記事」）を付与して保存する。

    • 作成した知識ベースは特定の Foundry IQ インスタンス（デモでは demo3）内に存在し、Azure AI Search がバックエンド。
    • 各知識ベースは 追加作業ゼロで MCP サーバーとして機能する。API で呼ぶことも、MCP として使うこともできる。
    • デモでは新規エージェントを書く代わりに GitHub Copilot をエージェントとして利用。auth プロキシで MCP サーバー（demo3.search.windows.net + 知識ベース名 + API バージョン）を接続。
    • 「Neo はどの薬を飲んだ？」と質問 → エージェントが知識ベースにグラウンディングし「Neo は赤い薬（red pill）を飲んだ、これは The Matrix から」と回答。
    ゼロから、エージェントが MCP / API で接続できる動く知識ベースまで、わずか1分。

    3. 層構造アーキテクチャ

    Foundry IQ は「選択肢を隠さず、使いたい層を選べる」よう層状に構築されている。

    4. 新発表 — Serverless Foundry IQ（パブリックプレビュー）

    これまで Foundry IQ と基盤の Azure AI Search は 専有キャパシティ（dedicated capacity）の作成を必要としていた。専有は安定ワークロードに最適で予測可能・隔離されているが、必要量を事前に見積もる必要がある。これを容易にするのが Serverless 版。

    10〜20秒
    サービス作成にかかる時間
    ゼロ
    未使用時はゼロにスケール（ストレージのみ課金）
    本番対応
    そのまま production へ

    Serverless は使用量ベースで、使うまで課金されない。動的ワークロード（エージェントがサービスを作成する、アプリが動的にサービスを作成・削除する等）に最適。既存 Foundry IQ スタックと同じ SOTA 技術（SOTA ランキング、即席 RAG スタック）が基盤。

    観点ServerlessDedicated（専有）
    課金モデル使用量ベース。使わなければ無料高い予測可能性
    得意なパターン多数の中小インデックス、開発ワークフロー、ユニットテスト、エージェントによる動的作成隔離キャパシティ、安定した大規模ワークロード
    特記その場で作成・削除する用途に強い機密コンピューティング（confidential computing）を上から下まで実行可能

    デモ：エージェントがサービスを自作する

    Pablo は IMDb の映画タイトルデータ（ファイル）を指し示し、エージェントに「そのファイルを見て、新しい Foundry IQ サービスとインデックスを作り、最初の100行をロードして」と一言で指示。並行して Foundry 上の GUI からも、対応リージョン（West Central US 等）で「Serverless」を選んで新規サービスを作成（10〜20秒）。Azure ポータルでも同じものが作れる。

    5. 知識ベースと知識ソース

    知識ベース（knowledge base）は「アクセス方法ではなく知識の範囲（スコープ）」であり、エージェントからは内部が隠蔽され関心が分離される。定義するのは「知識のドメイン」と「その情報の利用ポリシー」。

    • 検索の労力（quick に返すか、反復検索するか）を指定できる。
    • モデルの挙動に影響を与える steering instructions（操舵指示）を与えられる。
    • 回答合成（answer synthesis）を行うかどうかを選べる。

    その下に知識ソース（knowledge sources）がぶら下がる。多くは「知識の場所を指し示すと、ベクトル/字句インデックスを作成」する方式だが、オンラインアクセスが必要なソースへのフェデレーションも可能。実行時の表現に関わらず、シナリオごとに使う知識ソースを選んでクエリできる。

    データの所在に応じた選択肢（スペクトラム）

    知識ソース内容・特徴
    ファイル知識ソースコンテンツをアップロード。ファイルと同じ手軽さだが、背後に検索エンジンがありインデックスを作成→高速検索。「100万 PDF を blob に持っている」ような規模ではローカルファイルアクセスでは捌けない（毎秒10〜20クエリは不可能）ため有効。
    オブジェクトストア（Blob / OneLake / DB）アプリが内部管理するコンテンツの着地点。指し示すだけで既定のインジェストパイプライン（チャンキング・ベクトル化）が走り高品質な結果。各ステップのカスタマイズ・独自ステップ注入も可能。マルチメディアにも対応。
    Microsoft IQ プラットフォームWork IQ / Fabric IQ / Web IQ（後述）。
    MCP サーバー（拡張）組み込みオプションで足りないとき。データだけでなくアプリを「データとして」接続する普遍的な統合点。

    継続的インジェストと Azure Content Understanding

    Blob ストレージ等を指定すると、全コンテンツ処理（コンテンツ抽出、画像処理、ベクトル化）を実行。さらに 一度きりではなく継続的に動作し変更を追跡、ソースに変更があると差分をインデックスへ増分反映するため、常に新鮮なインデックスが自動で保たれる。

    PDF などの中身を理解するため、Foundry IQ は Azure Content Understanding と統合。OCR、ドキュメントレイアウト解析、読み順（reading order）の保持、ページをまたぐ表・疎な表の処理といった難所を高品質に抽出する。デモでは、手書きを含む表やチェックボックスまで markdown に抽出され、2カラムのビジネス文書のレイアウト・読み順・画像も正しく扱われた。構造化コンテンツにはスキーマを与えて特定要素だけ抽出し、フィルタ等に使える。

    6. Microsoft IQ プラットフォームとの統合 — Work / Fabric / Web

    組織のデータは孤立して存在しない。Word 文書、カレンダー、メールは M365 テナントに、分析データはデータレイク/データウェアハウスにある。Foundry IQ は IQ プラットフォーム全体に箱から出してすぐ接続できる。

    IQ コンポーネント役割設定のポイント
    Work IQM365 テナント（Word/カレンダー/メール等）への直接アクセステナントを指すだけで設定不要。ただしオフィスデータのためテナント管理者の同意が必要。
    Fabric IQ分析資産（データレイク、データウェアハウス）。自然言語→SQL、複数データソースのナビゲートOneLake カタログ全体と統合。レイクハウス、データエージェント、オントロジー等のオブジェクトが見える。
    Web IQWeb へのアクセス（前日に発表）MCP サーバー経由で web search ツールとして接続。

    各 IQ は単独でも動作するため、データが1か所にあるならそれだけ使えばよい。一方、自社アプリのデータ＋M365＋分析データを横断するエージェントも一般的なので、Foundry IQ は IQ プラットフォーム全体への統合をサポートする。

    デモ：1つの知識ベースに複数ソースを束ねる

    • Work IQ：テナントを指定（管理者同意後すぐ動く）。
    • Fabric IQ：OneLake のレイクハウスにある parquet（映画の評価などの統計）に対し、簡単なデータエージェントを作成。自然言語の質問を Fabric が SQL に変換して実行。レイクハウス選択時はインデックス化オプションが出るが、データエージェントは検索時に質問を統合するためインデックス不要。
    • Web IQ（MCP 経由）：「browse for more」から MCP サーバーを追加し、Web IQ のエンドポイントで認証、web search ツールを選択。

    結果、1つの知識ベースが、Wikipedia 記事（自前コンテンツ）・Web・Fabric データエージェントを必要に応じて使い分けて回答できるようになる。これを消費するエージェントは、裏で追加された可動部品を一切知る必要がない。

    MCP をデータ統合点に

    「エージェント版の ODBC があればよいが存在しない」。MCP はデータ統合専用ではないがどこにでもあるため優れた統合点になる。Foundry IQ は MCP の出力をヒューリスティックで項目指向のリストに変換し、自前のランキングを上から適用して関連性を高める。例：GitHub MCP サーバーで Issue を1つのテーブルとして取り込む、といった使い方が「ただ動く」。

    7. ドキュメントレベルのセキュリティ

    RAG 黎明期は「全文書を入れてチャット UI を被せる」novelty が先行した。だが現実の企業では全員が全文書を見られるわけではない。ドキュメントレベルのセキュリティは後付けで間に合わせるものではなく、検索システム自体が元データと同じセキュリティモデルを支える必要がある。

    • アクセス制御の伝播：SharePoint や階層型名前空間付き Blob（ACL）などアクセス制御をサポートするソースから、構築するインデックスへ ACL を伝播。Entra と統合されグループメンバーシップ等も機能。
    • クエリ時の権限適用：呼び出しユーザーの委任トークン（delegated token）を提示すると、そのユーザーが見られるサブセットだけが検索対象に見える。セキュリティはスタックの最下層で強制され、自前で上に重ねる必要がない。
    • Purview 統合：機密度ラベルで暗号化された文書も、Purview 統合により復号→インデックス化→ACL で保護。誰が見られるかを制御し、機密度ラベルを構築中の UI に伝播できる（「アプリが Office のように振る舞う」）。完全なサンプルも公開。

    なお先に依頼していたエージェントは完了しており、Azure ポータルで確認すると Serverless サービス・100ドキュメントのインデックスが自動生成され、映画情報が格納されていた。「一言の指示で動くインデックスができた」。

    8. 検索（Retrieval）— 第2世代エージェンティックリトリーバル

    検索は「すべての作業が価値に変わる瞬間」。検索エンジンは上位3〜4件がエージェントの求めるグラウンディングそのものになった瞬間に有用になる。Foundry IQ にはエージェンティック手法を用いた検索パイプラインが組み込まれている（生インデックスだけ使うことも可能）。

    flowchart LR Q["エージェントからの
    質問・データ要求"] --> P["クエリプランニング
    （対象ソース選択・分解）"] P --> S["複数検索を実行"] S --> E{"求める結果が
    得られたか？"} E -->|No・設定した検索努力に応じ反復| P E -->|Yes| R["最終結果を
    エージェントへ提示"]

    パイプラインはまずクエリプランニングで「どのデータソースを見るか」「クエリをどう分解するか」を決め、いくつか検索を実行→結果を見て十分か判断→不足なら設定した検索努力（retrieval effort）に応じて反復→最終結果を生成する。現在は第2世代。昨年10月から現在までにモデルがツール呼び出しループ等で大きく向上したため、最新モデルの実力を活かすべく検索スタックを大幅刷新した。

    評価結果 — 全指標で改善

    比較指標は recall、回答の正確性、回答の完全性、そして「答えが元データにあるのにモデルが『分からない』と言う頻度」。比較対象は字句インデックス（BM25）、ハイブリッド検索（ベクトル＋字句）、そしてエージェンティック検索オフの「minimal モード」。結果、全指標で実質的な改善。

    トレードオフはレイテンシ。検索インデックスは瞬時（レイテンシ予算に計上不要なほど高速）だが、retrieve 呼び出しは数秒、複数回反復すると10秒程度になることもある。

    改善の要因

    要因内容
    言語モデルの向上今日のモデルの作法に合わせて検索ワークフローを大幅更新。大きな差を生んだ。
    新しいセマンティックランカー長年ランカーを訓練しており、さらに境界を押し広げた。
    回答合成（answer synthesis）大幅な改善。
    MCP パラメータバインディング等モデルごとの差が大きいため、使用モデルに応じてプロンプトを調整し正確性を高めた。

    科学チームが評価方法の詳細なライトアップを公開しており、情報検索に関心があれば画面のデータを含む長文解説を参照できる（これで 3年連続の評価公開）。

    トークン効率 — Pareto フロンティア

    近年重要なトークン効率について、「検索にかかるトークン」ではなく「正解を得るのにかかるトークン」で評価。誰でも数トークンで検索はできるが、問題は正解したか。グラフは「最少トークンで到達できる最良点（Pareto 線）」を示し、ハイブリッド検索（外側エージェントが追加オーケストレーションを担う）と、Foundry IQ がオーケストレーションを担う完全エージェンティック検索を自社比較。

    • より少ないトークンでより良い回答を生成。
    • プロンプト構築・自身のコンテキスト管理をトークンキャッシュを意識して設計。プレフィックスキャッシュにヒットして高速、かつキャッシュトークンは常に安いため低コスト。

    9. retrieve API のデモと内部の可視化

    Pablo は retrieve API を呼ぶ簡単なクライアントを用意し、medium の検索努力で内部を見せる。複数データソースを持ち一部がオンラインのため、知識ベースにはオーケストレーション用モデルの指定が必要（デモは GPT-4.5 mini、既定の推論努力、回答合成の有無、両者への steering instructions も任意指定可）。

    質問は「評価（ratings）」と「興行成績（box office）」を含む複合的なもの。

    • ratings は Fabric のデータエージェントが parquet を SQL で問い合わせて取得。
    • box office は自前データになく、Web グラウンディング（MCP サーバー）で上位作品を取得。
    • プランナーが走り、まずデータエージェント、次に Web 用 MCP サーバーへ。一部コンテンツを破棄し、合成ステップで回答を構築して推論を終了。
    • 各結果には参照（references）が付き、どの知識ソース由来か、元ファイルへのリンク等を提示。通常 UX には出さないが、エージェントへの供給やデバッグに有用。
    ランキング・関連性・検索を改善する全技術と科学は、たった1つの API（retrieve 呼び出しまたは MCP サーバー呼び出し）の裏に隠蔽され、残りはすべて内部で起こる。

    10. 層を上下する柔軟性 と スケール・速度

    システムは「層をきれいに積層し、どの層をいつ使うか選べる」設計。非自明だった工夫は、1つの層を選んでもそこに固定されないこと。統合層から始めても、知識ベースでも、コア検索エンジン（Azure AI Search 上）に直行してもよく、後から上下に移動・方針変更できる。

    Azure ポータルで内部を覗くと、demo3 サービス内に作成した知識ベース、その知識ソース（リモート2つ、インデックス化2つ）、データを取り込むindexer（データパイプライン）、インデックスのスキーマまで確認できる。例として、ベクトル検索の圧縮は既定で8ビット量子化だが、量子化ポリシーやリランキングポリシーを細部まで制御できる。実際のチャンク（チャンキング結果のスニペット）やベクトルも閲覧可能。これによりレイテンシのトレードオフ（完全エージェンティック検索＝数秒で高品質 / ファイル的に大量検索＝超高速）を選べる。
