    1. 前回の復習とLLMの定義 01:02

    冒頭では、受講者が授業中に注釈できるよう、今後は各回のスライドを木曜夜までにcourse siteへ事前公開する方針を案内する。

    講義はTransformer系モデルを3つに再整理する。Encoder–DecoderはT5のようなtext-to-text変換、Encoder-onlyはBERTのような表現学習・分類、Decoder-onlyはGPT系の自己回帰生成に向く。Lecture 3はこのうちDecoder-onlyを中心に扱う。

    系統代表主な出力この講義での位置づけ
    Encoder–DecoderT5入力textを別のtextへ変換前回の復習
    Encoder-onlyBERT文・tokenのcontextual embedding分類や検索表現の例
    Decoder-onlyGPT、LLaMA、Gemma、DeepSeek、Mistral、Qwen左から右への次token生成本講義の中心

    講義ではLLMを「token列へ確率を割り当て、次tokenを予測するlanguage model」で、parameter・学習token・計算資源が大規模なものと説明する。目安として十億parameter以上、数千億〜数兆token規模を挙げ、現代の大半はDecoder-onlyだとする。

    用語の精密化：LLMに公的な閾値はない

    「10億parameter以上」「textを生成するDecoder-only」はこの講義の実用的な整理であり、普遍的な定義ではない。large に厳密なparameter境界はなく、文献によってはBERT系やEncoder–Decoderもlarge language modelに含める。重要なのは、parameter数だけでなく学習data量・compute・用途を併せて見ることだ。

    前提となる講義

    Self-attentionとTransformer本体はLecture 1要約、位置表現・BERT・GQAなどの派生はLecture 2要約で詳しく整理している。

    2. Mixture of Experts：総容量とactive computeを分ける 07:37

    Dense modelは全parameterを各forward passで使う。MoE（Mixture of Experts）は入力xをrouter（gate）へ通し、複数expert Ei のどれを使うか決める。Transformerでは通常、各blockの計算量が大きいFFNを複数expertへ置き換え、routingは文全体ではなくtokenごと、layerごとに行う。

    g(x) = softmax(Wgx)
    y = Σi∈Sk(x) g̃i(x) Ei(x) Wg: routerの学習weight、Sk(x): score上位k expert、g̃: 選択後に使う重み。正規化方法などの細部はarchitectureごとに異なる。
    flowchart LR X["token表現 x
    Self-attention後"] --> G["Router / Gate
    softmax(Wg x)"] G -->|"top-k"| E1["FFN Expert 1"] G -->|"top-k"| E2["FFN Expert 2"] G -. 非選択 .-> EN["FFN Expert N"] E1 --> Y["重み付き和 y"] E2 --> Y

    図をクリックすると拡大・ドラッグ・ズームできます。

    方式計算するexpert特徴目的
    Dense MoE全expert全出力の重み付き平均複数expertの表現を混ぜるが計算節約は小さい
    Sparse MoEtop-kのみ典型的にk=1または2総parameterを増やしつつactive parameterとFLOPsを抑える
    FLOPとFLOP/s

    講義中の “FLOPS” はforward passに必要な浮動小数点演算数という意味で使われている。厳密には、FLOPまたはFLOPsは演算量、FLOP/sは1秒あたりの処理性能である。MoEが直接減らすのは主にtokenあたりのactive FLOPsであり、expert間通信やrouterのoverheadまで自動的に消えるわけではない。

    expertは人間が付けた専門分野ではない

    数学者・歴史家の比喩はroutingの直観には有用だが、expertは通常、共同学習されたFFNである。特定domainや品詞へ分化する場合はあっても、各expertが明確で安定した人間可読の役割を必ず持つとは限らない。

    3. Routing collapseとload balancing 20:31

    routerとexpertはtask lossからend-to-endで同時学習できるが、少数expertばかり選ばれるrouting collapseが起き得る。選ばれないexpertは更新機会を失い、人気expertにはtokenが集中する。Switch Transformerは、hardな割当量とsoftなrouter確率を両方使う補助lossを加える。

    fi = (1/T) Σx∈B 1[argmax p(x) = i]
    Pi = (1/T) Σx∈B pi(x)
    Laux = αN Σi=1N fiPi T: batch内token数、N: expert数、fi: expert iへhard routingされた割合、Pi: expert iの平均soft probability、α: 補助lossの係数。
    講義中に保留された「fiをどう微分するか」

    fiはargmax由来で微分不能なので、Switch Transformerではstop-gradientして定数として扱う。gradientは微分可能なPiを通ってrouterへ流れる。均等ならfi≈Pi≈1/Nとなり、集中時よりlossが小さい。ただし完全な均等利用を保証する式ではなく、capacity制限やtoken droppingなど実装上の対策も別に必要になる。

    一次資料: Switch Transformers §2.3

    講義はさらに、router scoreへnoiseを足して別expertにも選択機会を与えるnoisy gatingを紹介する。Dropoutと同じく探索・regularizationの発想だが、noise、補助loss、expert capacityの設計はモデルごとに異なる。

    1.6T
    Switch-Cの総parameter。2,048 experts、top-1 routing。
    4×
    Switch-CがT5-XXLと同compute budgetで固定perplexityへ達した速度。論文内条件。
    46.7B
    Mixtral 8×7Bの総parameter。
    12.9B
    Mixtralでtokenごとに使うactive parameter。8 experts中top-2。

    MoEは「parameter総数を増やしても全parameterを毎回使わない」ため、capacityを拡張しやすい。一方、modelを置くmemory、expert parallelismの通信、負荷偏りは残る。総parameter、active parameter、実測latencyは別々に読む必要がある。

    4. 自己回帰生成・greedy・beam search 36:35

    Decoder-only LMはprompt xの後を1 tokenずつ生成する。全出力列の確率は条件付き確率の積で、実装ではunderflowを避けてlog probabilityの和として扱う。

    P(y | x) = Πt=1m P(yt | x, y<t)
    log P(y | x) = Σt=1m log P(yt | x, y<t) y<t: それまでに生成済みのtoken。BOSから開始し、EOSまたは上限で停止する。
    方式次tokenの選び方長所主な弱点
    Greedy decoding毎stepでargmax単純・高速・sampling自体は決定的局所最適であり、列全体の最尤を保証しない。多様性がない。
    Beam search上位b本の部分列を保持・展開greedyより広く探索計算・memoryが増える。bが有限なら大域最適ではなく、短い列を好むbiasもある。
    Sampling確率分布から乱択多様な生成低確率tokenによる破綻を制御する必要がある。

    長さbiasとlength normalization 43:56

    各token確率は1以下なので、log probabilityを足すほどscoreは負方向へ下がり、未補正のbeam searchは短い列を優先しやすい。講義は「token数の冪で補正する」と説明する。GNMTで使われた代表形は次の通り。

    lp(y) = ((5 + |y|) / 6)α
    score(y) = log P(y | x) / lp(y) |y|: 出力長、α: 長さpenalty。これは探索上のheuristicで、taskに応じた調整が必要。
    beam searchは「より大域的」だが大域最適ではない

    beam幅を有限にした時点で途中候補を捨てるため、最終的な最尤列を保証しない。また生成taskではbeamを広げるほど品質が上がるとは限らず、翻訳など候補の確からしさを重視するtaskで使われることが多い。

    5. Top-k・Top-p・temperature 46:36

    語彙全体からそのままsampleすると、ごく低確率の不自然なtokenも選ばれ得る。そこで候補集合を切り詰め、残った確率を再正規化してsampleする。

    方式候補集合性質
    Top-k確率上位k token集合sizeは固定。分布が平坦でも尖っていてもk個。
    Top-p / nucleus累積確率がp以上になる最小の上位集合分布に応じて候補数が変わる。

    最終hidden stateを語彙sizeへ線形射影して得る値がlogit zで、Softmaxが確率へ変換する。temperature TはSoftmax前にlogitの差を拡大・縮小する。

    pi(T) = exp(zi/T) / Σj exp(zj/T) T > 0。低温ほどargmax付近へ集中し、高温ほど平坦になる。
    pi(T) = exp((zi−zk)/T) / Σj exp((zj−zk)/T),   k = argmax z 最大logitを引く同値変形。講義の極限説明に加え、exp overflowを防ぐ数値安定化としても使われる。
    • T → 0+：最大logitが一意ならone-hotへ近づく。APIのT=0は通常、0除算ではなくgreedyとして特別処理される。
    • T → ∞：有限語彙なら一様分布へ近づく。
    • 高温は多様性を増やすが、事実性や品質を保証しない。低温も正答を保証せず、単に高logit候補へ集中させる。
    なぜtemperature 0でもAPI出力が変わり得るか

    Transformerの数式とgreedy選択が決定的でも、serving時のdynamic batchingでkernel形状やreduction順が変わると、浮動小数点の非結合則 (a+b)+c ≠ a+(b+c) による微差が生じ、近接logitの順位が反転し得る。Thinking Machines Labは原因を単なる「GPU threadの終了順」ではなく、batchに依存するkernel選択・演算順として分析し、batch-invariant kernelを提案している。

    公式技術記事: Defeating Nondeterminism in LLM Inference

    6. Guided decoding：生成中に形式を守らせる 1:04:53

    「JSONで答えて」とpromptして失敗時にretryする代わりに、現在までの出力とschema・grammarから次に許されるtokenだけを残す。講義はfinite-state machine（FSM）やcontext-free grammar（CFG）を入口として挙げる。

    z′i = zi   if token i is valid,   otherwise −∞
    p = softmax(z′) 禁止tokenの確率を0にしてから、許可集合内で再正規化・選択する概念図。

    FSM

    正規表現や有限状態で表せる制約を追跡し、現在状態から有効なtokenだけを許す。

    CFG

    括弧の入れ子など、有限状態だけでは表しにくい文法構造をparser状態として追跡する。

    JSON Schema

    型、必須field、enumなどをschemaへ落とし、対応するgrammarで出力を拘束する。

    保証されるのは主に構文・schemaで、内容の正しさではない

    valid JSONでも値が事実とは限らない。tokenizerでは1 tokenが複数文字を含むため、単純な文字単位filterより実装は複雑で、対応schemaにも制限があり得る。OpenAIのStructured OutputsもJSON modeとschema準拠を区別している。

    公式資料: OpenAI Structured Outputs

    7. Context lengthと「Context Rot」 1:07:07

    講義はcontext length、context size、window sizeを、モデルが一度に扱えるtoken数として導入する。現代モデルには数万〜百万token級のものがあるが、上限に収まることと、その全情報を同じ精度で利用できることは別問題である。

    Context Rotの一次資料

    Chromaの2025年technical reportは18モデルを評価し、taskの難しさをなるべく固定して入力長だけを増やしても性能が非一様に低下すると報告した。単純なNeedle in a Haystackだけでなく、needleと質問のsemantic similarity、類似distractor、haystackの構造を変え、長いcontextが「解決済み」とは言えないことを示す。

    公式technical report: Context Rot / 再現code

    実務上の含意は「入るだけ詰め込む」より、検索・reranking・chunk選択で関連contextを絞ること。distractorは単なる無関係textより悪影響を与える場合があり、contextの量だけでなく関連性・配置・構造が重要になる。

    context windowの数え方はAPIごとに確認する

    多くの生成APIではsystem/user prompt、tool結果、会話履歴、生成tokenの合計がwindowを共有する。一方で「最大input」と「最大output」を別枠で公開する製品もある。context lengthを単純に「入力だけの上限」と決めつけず、対象modelの公式仕様を確認する必要がある。

    Promptの4要素

    要素役割例
    Context役割・状況・前提を設定「疲れたteddy bear向けの話を作る」
    Instructions実行するtaskを指定「bedtime storyを生成」
    Inputtask固有のdata場所、対象文書、質問
    Constraints形式・安全・長さなどの制約対象年齢、JSON schema、禁止事項

    これは厳密な理論ではなく、promptを点検するためのmental modelである。4要素の境界はsystem/user message設計やapplicationによって重なる。

    8. In-context learning：weight更新なしでtaskを示す 1:14:09

    In-context learning（ICL）は、prompt内のinstructionや例からtask patternを読み取り、model weightを更新せずに応答を変えることを指す。ここでの“learning”はtraining時のparameter更新とは異なる。

    方式Prompt長所費用・risk
    Zero-shotinstructionと対象inputのみ短く、例の収集不要instruction品質とmodel能力へ強く依存
    Few-shotinput/output例を数件追加形式・task意図を具体的に示せるcontext、compute、latencyが増え、偏った例へ過適合し得る

    講義はfew-shotが一般に有効だが常に勝つとは限らないと強調する。例が有限で偏っていると未知分布でgeneralizeしにくく、能力の高いmodelでは、手順を明確にしたzero-shot instructionが同等以上になる場合がある。

    Plan-and-Solve

    講義で名前だけ紹介された手法。Zero-shot CoTの「まず計画し、その計画に沿って解く」を明示し、計算誤り、step欠落、semantic misunderstandingを減らすことを狙う。特定benchmarkでの改善を報告したprompting手法であり、全taskへの保証ではない。

    一次資料: Plan-and-Solve Prompting

    9. Chain-of-Thoughtとself-consistency 1:18:35

    Chain-of-Thought（CoT）は中間推論を生成してから最終回答を出すprompting。元論文は、十分大きいmodelの算術・常識・symbolic reasoningでfew-shot CoTが精度を改善すると報告した。長い推論を生成するため、token costとlatencyは増える。

    Self-consistencyは同じ問題から複数のCoT pathをstochastic samplingし、最終回答を抽出して多数決する。

    a* = argmaxa Σr=1R 1[ar = a] R: sample数、ar: reasoning path rの最終回答。離散的に比較できる回答で使いやすい。
    観点CoTSelf-consistency
    狙い中間stepを外化して問題を分解異なる推論pathを集約して単発の誤りを減らす
    追加cost長い出力複数回の生成＋回答抽出
    並列化1 path内は自己回帰各pathは独立なので並列化可能
    注意もっともらしい説明が因果的に忠実とは限らない並列資源がなければwall-clock latencyも増え、total computeは必ず増える
    CoTはdebugの手掛かりだが、内部判断の保証された説明ではない

    講義はCoTをinterpretabilityやroot-cause分析に使えると説明する。実務上、誤った日付などを見つける手掛かりにはなるが、Turpinらはbiasing featureに影響されたmodelが、その影響を述べずにもっともらしい別理由を生成する例を示した。出力されたrationaleをmodel内部の忠実な因果traceと同一視してはいけない。

    10. 推論最適化の全体像とKV cache 1:25:00

    講義は推論最適化を、同じ計算結果を保つ“exact”な工夫と、architecture・表現・token予測を変える近似的な工夫に分ける。その後、Attention周辺と出力token周辺の順に見る。

    課題講義の手法主なtrade-off
    過去K/Vの再計算KV cache計算をmemoryへ置き換える
    K/V head数GQA / MQAcache削減と表現力の折衷
    cacheの断片化PagedAttentionblock table・kernel実装の複雑さ
    K/V vectorの大きさMLAarchitectureとtrainingを変更
    1 tokenずつのtarget model呼出しSpeculative decodingdraft modelと検証overhead
    次tokenだけを学習Multi-token prediction複数headを含むtraining objectiveへ変更

    KV cache

    新tokenのQueryは毎step新しく必要だが、過去tokenのKey/Valueは変わらない。そこで全layerの過去K/Vを保存し、decode時に再利用する。prompt全体を並列処理する段階はprefill、1 tokenずつ続ける段階はdecodeと呼ばれる。

    KV bytes ≈ B × 2 × L × S × Hkv × dh × b B: batch/sequence数、2: KとV、L: layer数、S: 保存token数、Hkv: KV head数、dh: head次元、b: 1要素のbyte数。allocator等のoverheadは除く。

    例としてL=32、S=32,768、Hkv=8、dh=128、BF16（2 bytes）なら、1 sequenceだけで約4 GiBになる。contextと同じく線形に増え、同時request数を制約する。

    なぜtrainingでは通常cacheしないか

    teacher forcingでは正解token列を一度に与え、causal mask付きで全positionを並列計算できるため、token-by-token decodeの再計算問題が生じない。KV cacheは主に自己回帰inferenceの最適化である。

    GQAはQuery headを多く保ちながらK/V headをgroup共有し、式のHkvを減らす。MHA・GQA・MQAの詳しい比較はLecture 2要約を参照。

    11. PagedAttentionとMulti-head Latent Attention 1:33:09

    PagedAttention：連続した最大領域を先取りしない

    各requestが最大context分の連続memoryを予約すると、実際の生成長との差がinternal fragmentationになり、空き領域が小さく分断されるexternal fragmentationも生じる。vLLMのPagedAttentionはKV cacheを固定size blockへ分け、logical blockを任意のphysical blockへ対応付ける。

    flowchart LR R["Requestのlogical token列"] --> BT["Block table"] BT --> P7["Physical block 7"] BT --> P1["Physical block 1"] BT --> P9["Physical block 9"] S["共通promptを持つ別sequence"] --> BT2["別のblock table"] BT2 --> P7 BT2 --> P1 BT2 --> COW["分岐時だけCopy-on-Write"]

    非連続block、共有、Copy-on-Writeの関係。クリックで拡大できます。

    必要に応じてblockを割り当てるため、未使用予約と断片化を減らせる。共通prompt、parallel sampling、beam候補はphysical blockを共有し、書き換え時だけCopy-on-Writeする。論文は評価したserving条件でKV memory wasteを4%未満に抑え、既存system比2〜4倍のthroughputを報告するが、これはあらゆるhardware・workloadへの保証ではない。

    MLA：K/Vを低rank latentへ圧縮

    DeepSeek-V2のMLA（Multi-head Latent Attention）はtoken表現htを低次元latentへ圧縮し、必要なK/Vをそこから復元する。

    ctKV = WDKVht
    ktC = WUKctKV,   vtC = WUVctKV WDKV: down-projection、WUK/WUV: K/Vのup-projection。headごとの大きなK/Vそのものではなく共有latentをcacheする。
    講義図の「1 tokenにつき1表現」を精密化

    DeepSeek-V2の実際のMLAはjoint KV latentだけで完結せず、RoPEと両立するためのdecoupled rotary key成分もcacheする。したがって「K/V全headを1つのlatentへ大幅圧縮する」が正確で、「保存物が文字通り1 vectorだけ」とは限らない。

    DeepSeek-V2の総parameter / tokenあたりactive parameter。
    128K
    DeepSeek-V2がsupportするcontext length。
    −93.3%
    DeepSeek 67B比のKV cache削減。論文報告値。
    5.76×

    12. Speculative decodingとMulti-token Prediction 1:41:34

    Speculative decoding

    小さいdraft modelがk tokenを先に提案し、大きいtarget modelがその列を1回のparallel forward passで採点する。target modelの分布pとdraft分布qから、各候補token xを次の確率で受理する。

    A(x) = min(1, p(x) / q(x)) targetの確率がdraft以上なら必ず受理。低い場合は比率でaccept/rejectする。

    reject時は、単にtargetから引き直すのではなく、正規化した正の残差分布[p−q]+からsampleして終了する。これにより、hardware数値精度の範囲で最終sample分布をtarget modelと同じに保つ。全候補を受理した場合は、targetの同じpassで得たk+1番目の分布からさらに1 token進める。

    flowchart LR C["現在のcontext"] --> D["Draft model
    k tokenを逐次提案"] D --> T["Target model
    候補列を一括採点"] T --> A{"各tokenを
    min(1,p/q)で受理"} A -->|"全て受理"| N["k token以上進む"] A -->|"途中でreject"| R["[p-q]+から補正sample
    その位置で終了"]

    modified rejection samplingにより速度とtarget分布の一致を両立する。

    なぜ一括検証が速いのか

    小batchのautoregressive decodeは、演算器よりmodel weightを読むmemory bandwidthがbottleneckになることが多い。大modelをtokenごとに何度も呼ぶ代わりに候補列を一括採点すると、1回のweight読込で複数positionを処理できる。ただしdraft精度、受理率、batch size、hardwareに依存し、decodeが常にmemory-boundとは限らない。元論文はChinchilla 70Bの評価で2〜2.5倍のspeedupを報告した。

    Multi-token Prediction（MTP） 1:46:41

    MTPは外部draft modelを用意せず、共有trunkの同じhidden stateからt+1 ... t+nを予測する独立headを学習する。training objectiveそのものを次token予測から複数未来tokenの同時予測へ拡張する。

    LMTP = −Σt Σj=1n log p(j)(xt+j | x≤t) p(j): j token先を予測するhead。shared trunkの上にn個のoutput headを置く。
    観点Speculative decodingMTP
    提案元別の小さいdraft model同じmodel内の複数future-token head
    Training変更不要複数token予測lossが必要
    検証target分布を保つaccept/reject講義の紹介paperでは主headによるgreedy verification
    追加効果主にdecode高速化sample efficiencyやcode能力の改善も報告

    Gloeckleらの論文は、13B modelでnext-token baselineよりHumanEvalを12%、MBPPを17%多く解き、4-token prediction modelで最大3倍のinference speedを報告する。いずれも実験条件下の結果で、head数を増やせば無条件に同じ改善が得られるわけではない。

    13. 講義全体から得られる設計原則

    Capacity ≠ active compute

    MoEでは総parameterとtokenごとのactive parameterを分けて評価する。

    Model ≠ decoder

    同じnext-token分布でもgreedy、beam、sampling、制約で出力挙動は変わる。

    Long window ≠ reliable use

    上限token数だけでなく、retrieval精度、distractor、context構造を測る。

    Computeをmemoryへ移す

    KV cacheは再計算を減らす代わりに大きなmemoryを必要とする。

    Memory layoutもalgorithm

    PagedAttentionはmodelを変えず、serving systemの配置でthroughputを上げる。

    複数tokenを1 passで進める

    speculative decodingとMTPは別経路で自己回帰の逐次性を緩和する。

    Lecture 3の中心メッセージは、LLMの性能と効率はTransformer本体だけで決まらないということだ。routing、decoding、prompt、cache、memory allocator、verificationまで含めたsystemとして設計し、それぞれの品質・compute・memory・latencyのtrade-offを測る必要がある。

    14. 用語集

    Active Parameters

    1 tokenのforward passで実際に使われるparameter。Sparse MoEでは総parameterより小さい。

    出典: Mistral AI

    Router / Gate

    token表現からexpert scoreを計算し、top-k routingを決める学習可能network。

    出典: Sparsely-Gated MoE

    Perplexity

    平均negative log-likelihoodの指数。language modelが正解tokenへ高い確率を置くほど低い。

    出典: Switch Transformers

    Logit

    Softmaxへ入る正規化前score。temperatureやmaskは通常、この段階へ作用する。

    出典: Google Cloud

    Context Window

    modelが1回の処理で参照できるtoken範囲。最大長と実際の情報利用精度は別。

    出典: Context Rot

    Teacher Forcing

    training時に直前のmodel出力ではなく正解prefixを与え、全positionのlossを並列計算する方法。

    出典: Scheduled Sampling

    Prefill / Decode

    prompt全体を処理してcacheを作る段階 / cacheを使い1 tokenずつ生成する段階。

    出典: PagedAttention

    Memory-bound

    演算器の能力よりmemory bandwidthやdata移動が処理時間を制約する状態。

    出典: Speculative Sampling

    Internal / External Fragmentation

    割当block内の未使用領域 / 空きmemoryが非連続に分断されて使いにくい状態。

    出典: PagedAttention

    Low-rank Compression

    大きな表現を低次元latentへ射影し、必要時に展開する。MLAはKV cache削減へ使う。

    出典: DeepSeek-V2

    Rejection Sampling

    proposal分布のsampleを確率的に受理・棄却し、目的分布からのsampleを得る方法。

    出典: Speculative Sampling

    Sample Efficiency

    同じdata量・training stepでどれだけ高い性能へ到達するか。推論時samplingの意味とは別。

    出典: Multi-token Prediction
