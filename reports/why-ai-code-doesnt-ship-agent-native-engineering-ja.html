    2. 中心テーゼ ― マクロ委任・マイクロ操舵と「SDLC の終わり」

    Mario は、エージェントの使い方が質的に変わったと指摘する。以前は「マクロ委任」しても「これが間違っている、あれも間違っている」と常に細かく訂正し続ける必要があった。だが昨年12月から1〜2月にかけて、マクロに委任し、あとは少し方向を正す（マイクロ操舵）だけで済むようになった。

    以前現在（1月以降）
    委任の仕方マクロ委任しても常時訂正が必要マクロ委任＋マイクロ操舵（「この issue を直して」と委任できる）
    開発者の時間細かな訂正に追われる創造性・判断（judgement）に集中、委任を増やせる
    コードレビュー—Copilot からのフィードバックが継続的に改善
    本番—適切なガードレール付きで本番デプロイ、本番障害のデバッグまで

    その結果、エージェントは従来「SDLC（ソフトウェア開発ライフサイクル）」と呼ばれた領域の全体に入り込む。Mario は「SDLC はもう概ね死んでいる」と述べ、それが製品開発ライフサイクルと融合し、本質は「人間とエージェントの協働」になったと整理した。鍵になる問いは「これだけ作業しているのに、出荷（ship）を妨げているものは何か」である。

    3. エージェントネイティブ・エンジニアリングシステムの6つの柱

    この変化は「システム」を必要とする。GitHub は本来「人間と人間の協働」のために作られたが、それを「人間とエージェントの協働」へ進化させる。そのために必要なプリミティブ（基本要素）を、GitHub は次の6つの柱に整理した。これを総称して Agent-Native Engineering System（エージェントネイティブ・エンジニアリングシステム） と呼ぶ。

    graph TD subgraph AN[Agent-Native Engineering System] S[1. Surfaces 面
    IDE・CLI・Copilotアプリ・スマホ] R[2. Runtime 実行環境
    ローカル/クラウドの安全な実行] A[3. Automation 自動化
    予測可能性とスケール] Q[4. Quality 品質
    意図→出荷物の検証] M[5. Memory 記憶
    自己改善する継続的知能] E[6. Enterprise 信頼と規模] end S --> R --> A --> Q --> M --> E
    柱役割セッションでの言及
    ① Surfaces（面）開発者が作業する場所IDE、コマンドライン、新しい GitHub Copilot アプリ、スマホ
    ② Runtime（実行環境）エージェントが安全に実行できる場ローカル／クラウドで信頼できる実行。基調講演の MXC に言及
    ③ Automation（自動化）システムを予測可能にしスケールさせる「3Dプリンタ」の比喩。CI/CD・DevOps・DevSecOps はすべて自動化の運動だった
    ④ Quality（品質）意図（CAD図）が出荷物（成果）になったか検証検証システムが必要
    ⑤ Memory（記憶）自己改善する継続的知能後述の Chronicle に対応
    ⑥ Enterprise企業レベルの信頼とスケール面から企業まで一貫

    Mario は自動化を「3Dプリンタ」に例える。CAD図（意図）を入れると物体（ソフトウェア）が出てくるが、最後に「欲しかったものか」を検証し、欠陥があれば捨てて作り直す制御が要る。自動化こそがシステム全体を予測可能にし、スケールを可能にする——これは CI/CD や DevSecOps と同じ「自動化の運動」の延長線上にある、というのが主張だ。エージェントネイティブ化には、新しい API・新しい UX（エージェント体験）・予測可能性を伴う新しい自動化が必要になる。

    4. GitHub Copilot アプリ ― 「より多くやる」でなく「より多く完了させる」

    ここから Evan Boyle がデモを担当。まず強調したのは、Copilot アプリは「単なるセッションマネージャ」ではないという点だ。多くのツールは大量の並行作業を「始める」のを助けるが、Copilot アプリは作業を「完了させる（finish）」ために作られている。面白いのは実装そのものではなく、ユーザーに価値を届けるためのメタ作業——issue のトリアージ、CI を通してグリーンに保つ、レビュー指摘への対応、マージ、顧客へのデプロイ——だという。

    「エージェントの約束を『より多くやる』から『より多く完了させる』へと転換すべきだ。集中を取り戻すこと。活動（activity）ではなく成果（outcome）に焦点を当てること。」 — Evan Boyle（09:13 付近・要約）

    受信箱と複数セッションの一括起動

    ホーム画面の「受信箱」には、公開リポジトリに寄せられた大量のフィードバックが集まる。Evan は「terminal（ターミナル）」関連の改善 issue を絞り込み、関連する複数 issue を選択して Actions →「new individual sessions」を実行。すると左ナビに3つのセッションが即座に起動し、それぞれ隔離されたワークツリー（work tree）が作られ、issue のコンテキストが自動で添付される。関連作業をまとめて起動・レビューできる点が利点だ。

    5. Canvases ― エージェントが駆動する自作UI

    続いて、Evan とチームが作った Canvases（キャンバス）。エージェントがアプリ右側の面を駆動でき、ユーザーとの間で双方向（bi-directional）にやり取りする仕組みだ。

    組み込みキャンバス（ターミナル）

    音声指示で「ターミナルキャンバスを開いて hello canvas を表示する Web サーバーを起動し、ブラウザでそのページを開いて」と依頼。エージェントがターミナルを開き、コマンドを送り、出力を読み、ブラウザを開く（Python の Web サーバーを起動）。自動テストも可能。

    トリアージキャンバス

    チームが .github リポジトリにチェックインしたトリアージ用キャンバス。最近の issue・最も「いいね」の多い issue・共通テーマを表示し、ボタン（analyze sample）でエージェントに「5〜8個のフィードバックバケットを作って」と指示できる。これをチームは毎日のスタンドアップで使い、顧客の要望をレビューしている。キャンバスから直接セッションを起動でき、サブセッションが入れ子（nesting）になる。背後ではエージェントが「create session ツール」を呼んでおり、ユーザーがアプリでできる操作はすべてエージェントも駆動できる。

    キャンバスを作る／配布する

    • create canvas スキル：音声で「描けるホワイトボードキャンバス、JPEG 書き出しと gist 公開のボタン付き」を依頼すると、.github フォルダに拡張が動的に生成される。チェックインすればチーム全員が同じキャンバスを入手（リポジトリスコープ）。デモでは実際にワンショットで生成し、gist 公開まで成功させた。
    • 拡張エコシステム：gist や URL から拡張をインストール可能。デモでは「agentic Kanban board（カンバン）」を導入し、タスク起票やテーマ変更のホットリロードを実演。
    • マーケットプレイス：個人向けの小さな拡張だけでなく、ファースト／サードパーティの公式キャンバス拡張を扱うマーケットプレイスを立ち上げ予定。Miro と協業し、付箋やデザインアーキテクチャをエージェント駆動で操作するキャンバスを披露した。

    狙い：チームや個人のワークフローに合わせたハイパーパーソナライズなソフトウェアを、エージェントと共に作れるようにすること。

    6. Copilot --cloud と GitHub Sandboxes

    「エージェント不安（agent anxiety）」——退勤間際の16:55、プランを仕上げてエージェントを走らせてから帰りたいのに時間切れになる感覚。Evan は「ノートPCを閉じてそのまま帰れる」ようにしたいとして、copilot --cloud を紹介した。

    • GitHub Sandboxes 上に構築。数秒で起動する高速マイクロVM。
    • ステートフルかつ隔離。PR を安全な環境でテストでき、ローカル環境と分離。認証情報は厳格にスコープされる。
    • タスクをクラウドで一晩走らせ、翌朝戻って続きを再開できる。ノートPCはインターネット接続不要。

    Mario は後半、この Sandbox を「YOLO（一回きり）」を多用するデモの裏で支える信頼とセキュリティの要だと補足。現在ローカル Sandbox（パブリックプレビュー）とクラウド Sandbox の両方があり、Copilot アプリからも委任できるようになる予定だとした。

    7. Chronicle ― 作業の継続性と SQL API

    Copilot は dotcom、クラウドエージェント、CCR（Copilot Code Review）、Visual Studio、VS Code など多数の面に存在する。どこで何を作業していたか思い出しにくい——この課題に対する答えが Chronicle だ。

    • Chronicle search：すべてのセッションをユーザー単位・プライベートに保持し、面・マシン・Copilot 製品をまたいで継続性を提供。「すべてのセッション、すべてのコード行、すべてのプロンプト」を記憶し、過去の文脈（例：以前直した Rust のバグ）を探してセッションに読み込む。
    • Chronicle stand-up：直近24時間の作業を集約し、スタンドアップ用にまとめる。重要なのはエージェントが SQL を扱える点で、直接の SQL API がエージェントに公開されている。全文検索と構造化 SQL クエリを、全セッションを収めたデータレイクに対して組み合わせて実行する。
    「Chronicle はあなたがこれまでにやったことを全部覚えている。すべてのセッション、すべてのコード行、すべてのプロンプトを。」 — Evan Boyle（23:41 付近）

    8. Agent Merge ― 自律的な PR マージ

    作業を「線の向こう側へ運ぶ（get work over the line）」ための機能が Agent Merge。PR に付く「agent merge」ボタンを押すと、エージェントが PR を定期的に点検し、グリーンでマージ可能になるまで自律的に対応する。

    flowchart TD Start[agent merge ボタンを押す] --> Check{定期チェック} Check --> C1[マージ衝突はないか] Check --> C2[CI はグリーンか] Check --> C3[エージェント/同僚の
    レビュー指摘は残っていないか] C1 --> Fix[必要な変更を実施] C2 --> Fix C3 --> Fix Fix --> Check Check -->|すべて解決| Ready[グリーン・マージ可能] Ready --> Notify[このタイミングで初めて
    レビュアへ通知＝集中を保護]

    最初の Agent Merge PR の例

    Agent Merge を作った Jeremy は、CLI のヘルスチェックを追加する PR で試した。CCR（Copilot Code Review）が概要と5件のコメントを返し、Jeremy の代理エージェントが「このコミットで対応した。理由はこれ、対応しなかった点はこれ」と応答。Jeremy は Agent Merge を設定して昼食に行き、その間に CCR の指摘がすべて処理された。準備が整って初めて Mario に PR が送られる——レビュアの注意と集中を守る設計だ。

    さらに Agent Merge は盲従しない。Mario が「ドキュメントのリンクを足すべきでは」と指摘すると「2行下に既にあります」と返し、「1,000行を超えたから大規模リファクタを」と無理を振ると「もっともな懸念だが、この変更の一部としてはやりません」と断ってマージした。

    「もっともな懸念です。ですが、それはこの変更の一部としては行いません。」 — Agent Merge（28:53 付近、Mario の無茶振りへの返答）

    問い：あなたは「活動（activity）」に集中しているか、「成果（output）」に集中しているか。Copilot アプリの狙いはすべて成果——50個を同時にやるのではなく、本当に重要な5〜10個を高品質にやり切ることにある。

    9. Automations ― モーニングブリーフと自動リプロ

    「エージェントは時間を節約するはずが、多くのチームでは逆にノイズの放水ホースを生んでいる」。Mario は、エージェントで集中を守る使い方として自動化を示した。

    モーニングブリーフ（ローカル・自分専用）

    毎朝6時にローカルで実行される自分専用の自動化。「重要なもの」を指定しておくと、トップ10 issue（リンク付き）、進行中の PR、未レビューの重要 PR、新規の重大 issue をまとめてくれる。出力はチームに共有されず、コーヒー片手に集中して一日を始められる。

    クラウド自動化（issue リプロ）

    自動化はクラウドにも作れる。「issue 作成」をトリガに、プライベートな GitHub アプリ用リポジトリでバグ報告の再現を試み、失敗するテストケースを書き、ラベルを付け、再現手順とテストケースをコメントする自動化を作成。これにより、チームが見る頃には再現が済んでおり、修正は多くの場合容易になる。オープンソース経験者なら「丁寧な再現付きの報告」のありがたさが分かる、と Mario。

    10. Rubber Duck とマルチモデル・セキュリティレビュー

    Rubber Duck（ラバーダック）

    GitHub はマルチモーダル・マルチエージェント・マルチサーフェスになった。これにより、同僚に「これどう思う?」と相談する（ラバーダッキング）感覚を、エージェントとの間でも得られる。Rubber Duck にはアドバイザーと批評家（critic）の2モードがあり、Copilot CLI と GitHub アプリの両方で使える。Mario は主に批評モードを常用し、トークン節約時はアシスタントモードを使うという。

    マルチモデルのコード／セキュリティレビュー

    最後のデモは「品質」。Mario は同僚 Enrique の PR（アプリのクラウド対応、入れ忘れていた機能）をレビュー。Command-K で URL を貼ると Copilot がレビュー用セッションを生成する。ここでの会話はgithub.com に送られないプライベートな対話で、エージェントとコードの共通理解を作る。/review（コードレビュー）と /security review（セキュリティ）を組み合わせ、複数モデルで実行できる。

    「複数のモデルは互いに同意しない。訓練分布が違うからだ。だからこそ複数モデルを使うと、最良の結果が得られる。」 — Evan Boyle（40:51 付近・要約）

    Mario は次のオーケストレーションを1つの指示で実行した。

    sequenceDiagram participant P as 親セッション participant R as セッション1: コードレビュー participant S as セッション2: セキュリティレビュー participant D as Rubber Duck P->>R: マルチモデルでコードレビュー
    (Gemini / GPT-5.5 / Sonnet 4.6) P->>S: マルチモデルのサブエージェントで
    セキュリティレビュー R-->>P: 結果を親へ返信 S-->>P: 結果を親へ返信 P->>D: 返ってきた出力を批判的に吟味 D-->>P: 最終レコメンド（必要なら PR にコメント）

    セッション1はコード品質、セッション2はGitHub Advanced Security と協働して作ったセキュリティ・サブエージェントによる検査に専念。両者の結果が親セッションに戻ると、Rubber Duck が最後の批評を加えてから人間に提示する。全モデルとラバーダックが同意し、セキュリティレビューが問題を検出——この一連で、高品質な検証を高速に回せる。デモ終盤では Gemini を使ったセキュリティ・サブエージェントが3つ並列で稼働する様子が示された。

    パートナーと信頼

    信頼の観点で多くのパートナーと協業している。Sonar（コード品質）、Endor（セキュリティ）、Miro（キャンバス）、LaunchDarkly（安全な本番デプロイ）などが挙げられた。

    11. まとめ ― アイデアから本番まで

    セッションは、冒頭のエージェントネイティブ・エンジニアリングシステムに立ち返って締めくくられた。Surface・Runtime・Automation・Quality・Memory・Enterprise の柱を一貫して agent-native に作り直すことで、アイデア（意図）を、人間とエージェントの協働で本番まで運べるようになる——これが「出荷を妨げるものを取り除く」という本セッションの主題への答えだ。

    働き方

    マクロ委任＋マイクロ操舵。人間は創造性と判断に集中。

    完了に寄せる

    「より多くやる」でなく「より多く完了させる」。活動でなく成果。

    集中を守る

    自動化・Agent Merge・モーニングブリーフでノイズを抑える。

    品質と信頼

    マルチモデル・レビュー、Sandbox、Advanced Security、パートナー連携。

    なお本セッションでは、6つの柱のうち Memory（記憶） と Enterprise の踏み込んだ実演は時間の都合で割愛され、今後さらに紹介されるとされた。
