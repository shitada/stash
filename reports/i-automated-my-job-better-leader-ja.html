    GitHub Copilot アプリ オートメーション 生産性 アクセシビリティ AuDHD リーダーシップ

    3 行サマリ

    • シニアリーダーの本当の負荷は個々のタスクではなく、15 か所に散らばる仕事を「自分の脳」だけがつないでいること。GitHub Copilot アプリの「オートメーション（スケジュール実行されるプロンプト）」が、この不可視労働を肩代わりする。
    • 著者は約 40 個のオートメーションを運用。朝のブリーフ、自社ローンチの把握、日々の成果記録、約束のフォロー、依存関係更新やカンファレンス調整までを自動化し、1 日 1 時間規模の時間と大量の認知的負荷を取り戻した。
    • ただし称賛・気づき・人とのつながりは自動化しない。「オートメーションは足場を担い、人間の仕事は自分でやる」。AuDHD の著者にとっては実行機能のムラを埋めるアクセシビリティツールでもあり、AI を「生成」から「不可視労働の拡張」へと捉え直す。

    1. はじめに ― 脳が唯一の「接続システム」になる問題

    シニアリーダーの仕事について誰も警告してくれないことがある、と著者は言う。仕事が難しいのは、ひとつひとつのタスクが難しいからではない。仕事が 15 もの別々の場所に存在し、それらをつなぐ唯一のシステムが自分の脳だから難しいのだ。

    ミーティングは次々と重なり合う。決定は自分のいないスレッドの中で下される。誰かが計画会議で自分の名前を出し、見たこともないドキュメントの中にアクションアイテムが生まれている。それに気づくのは 2 週間後、誰かが何気なく「進捗は?」と尋ねてきたときだ。

    昨年、著者のチームはパフォーマンスレビューの締め切りを危うく逃しかけた。締め切りが、誰も見ていないチャンネルで告知されていたからだ。ある人は 10 分間 Slack を探しても見つけられず、別の人はまったく無関係なチャンネルでその日付を発見した。著者は結局「Slack でのフォローアップを怠った、それは私の責任だ」と投稿する羽目になった。脳が唯一の接続システムであり続ける限り、こうしたことは起き続ける。

    コンテキストスイッチにあまりに多くのエネルギーを使い、本来この役割が必要とする「考える・つなぐ・創る」── しかも自分が実際に好きな仕事 ── に回す余力が残っていなかった。

    そんなとき GitHub Copilot アプリのオートメーションを使い始め、ワークフロー全体が変わった ── というのがこの記事の主題である。

    2. オートメーションとは何か

    GitHub Copilot アプリ ― エージェントと「協働する」ための場

    GitHub Copilot アプリは、macOS・Windows・Linux 向けのスタンドアロンなデスクトップアプリで、エージェントと「ただ対話する」ためではなく「協働する」ために作られている。リポジトリをまたいで並列セッションを走らせることができ、各セッションはそれぞれ独自のブランチとワークツリーを持つ。エージェントが何をしているかは Canvases（キャンバス）を通じてリアルタイムに見える ── キャンバスは双方向の作業面で、人とエージェントが同じプラン・ターミナル・ブラウザセッションの上で同時に操作する。進捗はチャット履歴に埋もれず、可視で、かつ操舵できる。

    オートメーション ― 常設の指示書を持つエージェント

    オートメーションとは、あなたの実際の作業コンテキストに対して走る、スケジュールされたプロンプトだ。カレンダー、メール、メッセージ、GitHub リポジトリ。これらに MCP サーバーと連携を通じて接続し、仕事が散らばるあらゆる場所で何が起きているかを見る。そして「本当に自分の注意が必要なもの」を教えてくれる ── だから、それ以外は無視できる。

    常設の brief（指示書）を持つエージェントだと考えればいい。何を気にかけ、どう考え、いつ走るかを伝える。あとは……ただやってくれる。毎日。あなたが頼むのを覚えていなくても。なぜなら、あなたは覚えていないから。

    3. 実際の使い方 ― 40 個のオートメーション

    著者は GitHub のシニアディレクターで、開発者リレーションを率いている。スコープは広く、カレンダーは埋まっている。そして自分の脳は多くの人が想定するのとは違う動き方をする ── AuDHD（自閉スペクトラム＋ADHD）だ。パターン認識と深い集中は得意だが、3 日前にフォローすると約束したスレッドを覚えているのは本当に苦手だという。

    最初から 40 個を作るつもりだったわけではない。オートメーションのタブが気になり、アプリに「何ができるの?」と尋ねたところ、自分では思いつかなかったことを提案してくれた。最初に立ち上げたときのプロンプトは、おおよそこうだ。

    「私のすべての作業面、カレンダー、メール、メッセージを横断して、どこでボールを落としているか、どこで助けが要りそうかを見つけ、役立つオートメーションを提案して。」

    アプリは即座に 6 個ほどを提案した。最初のドラフトは完璧ではない ── それでいい。洗練し、声（voice）を与え、自分の考え方を教えていく。可能性が見えてからは続け、いまは約 40 個になった。

    すべては紹介しないが、最も重要なカテゴリと、それぞれのハイライトは次の表のとおり。固有名のオートメーションは原文の名称（英語）を残している。

    カテゴリオートメーション何をするか
    朝のブリーフMeeting Prep（ミーティング準備）カレンダーを引き、すべての会議のコンテキストを構築。1on1・大規模な同期・社外コールで異なるフォーマットを用意。着席した時点で、各会議が何の話で、自分が何を持ち込むべきかが分かっている。
    Pre-Meeting Access Check（事前アクセス確認）招待状で参照されるドキュメントやリンクに、実際に自分がアクセスできるかを検証。会議に出てからアジェンダ用ドキュメントがロックされていると気づく、あの種のパニックを防ぐ。
    Daily Triage Digest（日次トリアージ）GitHub・メール・メッセージを横断的にさらい、自分の注意が必要なものをすべて収集する。
    最新情報の把握Ship Decoder直近 24 時間に GitHub が出荷したものをすべて見つけ、平易な言葉で説明。会話でそのまま使える実際のコンテキストになる。
    Launch Radar週次で実行。自分のチームの領域に触れる今後のローンチを浮かび上がらせ、不意打ちを食らわないようにする。
    キャリアの設計Daily Wins Recap（日次の成果まとめ）毎晩実行し、その日実際に達成したことを要約。走る記録を残すことで、パフォーマンスレビュー期に「8 か月分を思い出せず白紙の前でパニック」になるのを防ぐ。
    チームと人Commitments and Follow-Up Tracker（約束とフォローの追跡）自分のメッセージから「やる」と言ったことを探し、まだ対応していないものを指摘する。「調べておく」と言って忘れることは信頼の問題 ── これがその信頼を守る。
    メンテナンスとロジDependabot PR Triage複数リポジトリ横断で、安全な依存関係の更新を毎日見つけてマージ。
    Stale Work Finder（停滞作業の発見）開いたまま忘れた PR、静かになった issue、ほこりをかぶったブランチを浮かび上がらせる。
    Travel Logistics Tracker（出張ロジ追跡）カンファレンス関連のスレッドを監視し、ロジスティクスを単一のブリーフに集約。カオスになりがちなカンファレンスシーズンを助ける。

    朝のブリーフがもたらした変化

    何かを開く前に、すでに複数のオートメーションが走り終えている。その累積効果として、著者の朝は「アジェンダを読んだふりをしながら 12 個のタブを慌てて開く」状態から、「コーヒー片手にいくつかの要約を読む」状態へ変わった。まるで別の人生だという。Ship Decoder と Launch Radar の 2 つだけでも、チャンネルをスクロールして何が起きたか繋ぎ合わせる作業を、1 日およそ 1 時間節約している。

    キャリアの設計 ― 自分を認識するためのオートメーション

    最も意外だったのがこのカテゴリだ。著者のデフォルトモードは「チェックを入れたら即、次へ」。立ち止まらず、認識せず、ただ進み続ける。だからレビューの季節が来ると、自分のインパクトを言語化できず、何か月分もの仕事を思い出せないまま白紙を見つめる。Daily Wins Recap はその記録を代わりに保ち続ける ── タスクリストではなく、実データに裏打ちされた「感謝の実践」のように。インポスター症候群が大きい日に、事実で言い返してくれる存在だ。

    「ロボットは、私が自分を信じられないときでさえ、私を信じてくれる。妙に心を動かされる? 分からない。でも、効くのだ。」

    チームと人 ― 自動化しないものの線引き

    ここで著者は強く念を押す。読者はこう思うかもしれない ──「彼女は仕事の人間的な部分まで自動化しているのでは?」。答えは ノー。そして、この区別こそ、この投稿の何よりも著者にとって重要だという。

    • Commitments and Follow-Up Tracker は謙虚にさせられるが不可欠。約束を忘れることは信頼の問題であり、オートメーションはその信頼を守る「ため」に働く。
    • 自分が書く kudos（称賛）は依然として自分のもの。気づくことも自分のもの。オートメーションは、自分の脳がそれを「ふさわしい人」から奪わないようにするだけ。
    • これらのオートメーションはつながりを置き換えない。可能にする。人にちゃんと向き合うための余白を返してくれる。以前は気を散らし、疲れ切って会話に入っていた。いまは 1on1 で本当に「そこにいる」し、称賛は具体的で本物になった。

    「オートメーションは足場（scaffolding）を担う。人間の仕事は私がやる。それが取り決めだ。」

    4. 実際のオートメーションはどんな見た目か ― Stale Work Finder

    著者のセットアップから実際の 1 つ、Stale Work Finder のプロンプトを示す。これが「オートメーション」の中身の実例だ。

    Find all my stale work across GitHub using the gh CLI. Things that are falling through the cracks.

Check for:

- PRs I opened that haven't received a review in 7+ days
- PRs I'm assigned to review that I haven't reviewed yet (older than 3 days)
- Issues assigned to me that have had no activity in 14+ days
- Draft PRs I own that have been drafts for 2+ weeks
- For each item show: repo, title, link, how long it's been stale, and who's involved.

Format as:

  1. 🔴 Embarrassingly stale (3+ weeks)
  2. 🟡 Getting dusty (1-3 weeks)
  3. 🟢 Just needs a nudge (under a week)

    これだけだ。プロンプトを書き、スケジュールを設定すれば、エージェントが代わりに実行する。詳しく書いても、緩く書いてもいい ── アプリが接続済みのツールからコンテキストを補完してくれる。著者の場合は毎週月曜に走り、結果はいつも少し目を見開かされるものになる。それでも、知らないより知っているほうがいい。

    5. アクセシビリティとしてのオートメーション ― AuDHD の視点

    著者ははっきり言う ── 自分にとって、オートメーションはアクセシビリティのツールだと。

    AuDHD は、実行機能とワーキングメモリが極端に不安定になることを意味する。そして、その不安定さ自体が、人に説明するうえで一番難しい部分だ。ある日は 17 のスレッドを頭の中に保持できる。別の日は 10 分後に会議があることを忘れる。その中間がない。かつてはこの「日による差」が怖かった。脳がその火曜日にどう動こうと、チームは一貫したリーダーシップに値するからだ。

    これらのオートメーションは、その差を縮める。私を一貫させる。私の実行機能が今日現れようと現れまいと、チームは同じ質の注意を受け取れる。私にとってそれは、繁栄と、じわじわ燃え尽きることの分かれ目だ。燃え尽きる側はもう経験した。星ゼロ、おすすめしない。

    6. どう始めるか（本気で）

    同じようなものを作ろうと考えているなら、著者の助言はこうだ ── 一度に全部を自動化しようとしないこと。最も摩擦の大きい「ひとつ」から始める。

    著者にとってそれはミーティング準備だった。準備には 4 つの異なるツールを回り、統合する帯域もない情報を統合する必要があり、いつも準備不足のまま会議に入っていた。それを 1 つのオートメーションが解決した。そしてその安堵を感じてから、続け、続け、続けた。

    コツはこうだ ── GitHub Copilot アプリでチャットを開き、自分の作業面を監査させる。どこでボールを落としているか? 繰り返しのパターンはどこか? ずっとやろうと思いつつ手をつけられないことは何か? そこから始める。最初のドラフトは完璧ではない。会話の中で洗練し、自分の声・優先順位・「良い」の基準を教え、そして走らせる。

    40 個もあれば統合できるものもあるはず、と著者も認める。著者は具体性を好むが、いくつかを 1 つの大きなオートメーションにまとめるスタイルでもいい。まずは 1 つから。感触を確かめて、次を作る。そしてまた次を。気づけば 40 個になり、それについてのブログを書いている ── というわけだ。

    7. より大きな視点 ― 生成から「不可視労働の拡張」へ

    いま、人が職場で AI とどう関わるかについて、興味深い局面にある、と著者は見る。初期の議論は主に「生成」だった ──「これを作って」「このコードを書いて」。だが現実は、少なくとも著者にとっては、もっと「不可視労働の拡張（augmentation of invisible labor）」に近い。燃え尽きさせるのに成果物には決して現れない仕事。パフォーマンスレビューでは誰も認めないのに、皆が埋もれているメタワークだ。

    初期の捉え方：生成

    「物を作って」「コードを書いて」。アウトプットとして目に見える成果を AI に生成させる。

    著者の実感：不可視労働の拡張

    コンテキスト切り替え、約束の追跡、情報の同期 ── 成果物に現れないのに人を消耗させる「メタワーク」を肩代わりさせる。

    著者が知るリーダーは皆、コンテキストに圧倒されている。著者が知る神経多様性の専門家は皆、神経定型の人が考えもせずにこなすシステムに、膨大なエネルギーを注いでいる。オートメーションは組織の機能不全も、まずいマネジメントも、理不尽な仕事量も直さない。だが、本来やるべき仕事をするための余白は返してくれる。そしてそれで十分だ ── いや、それはとても大きい。

    最後に著者はこう添える。これは GitHub の製品なので、依存関係の更新、issue のトリアージ、リポジトリ横断のセキュリティスイープも当然こなす。開発者向けワークフローは期待どおり。著者はたまたま、それを誰も話題にしない仕事の部分に使っているだけだ、と。

    8. 著者について
