    1. まず結論: Windows ML は何のためのものか

    Windows ML は、Windows 上で AI モデルをローカル実行するための統一推論フレームワークである。公式ドキュメントでは、 ONNX Runtime を基盤に、NPU / GPU / CPU 上の推論を Windows が管理する Execution Provider（EP）で加速する仕組み と説明されている。 旧来の「WinML」という高レベル API だけを指すより広い意味で、2026 年時点の Windows ML は、ONNX Runtime、DirectML、ベンダー EP、Windows App SDK、モデル配布、開発ツールを含む「Windows 向けローカル AI の実行基盤」と考えると理解しやすい。

    セッションの合言葉は Unmetered Intelligence。つまり、すべてをクラウド API に送るのではなく、ユーザーの PC 上の GPU / NPU / CPU を使って、 トークン課金、ネットワーク依存、データ持ち出しを減らすという発想である。登壇者は「1 billion Windows devices」を対象に、日常的な Windows PC を AI-capable にしていくと説明した。

    プライバシー

    音声、画像、文書、企業データを端末内で処理し、クラウドに送らない設計を取りやすい。

    低レイテンシ

    クラウド往復が不要。音声変換、映像処理、入力補助などリアルタイム用途に向く。

    オフライン

    ネットワークが弱い現場でも、端末内モデルなら推論を継続できる。

    コスト

    すべての処理をクラウドトークン課金に乗せず、端末リソースを使える。

    2. Microsoft Foundry on Windows の 3 層

    セッションでは、ローカル AI を始める方法を Windows AI APIs / Foundry Local / Windows ML の 3 段階に整理していた。これは公式の Foundry on Windows 概要ページの意思決定ツリーとも一致している。

    選択肢向いている場面できること主な対象
    Windows AI APIs 既製の AI 機能を数行のコードで使いたい。モデル選定や最適化を自分で持ちたくない。 Phi Silica、要約、構造化 JSON、OCR、画像説明、背景抽出、オブジェクト消去、画像 / 動画スーパー解像、音声認識など。 主に Copilot+ PC と Windows App SDK 対応アプリ。一部 API は CPU / GPU へ拡大中。
    Foundry Local Qwen、Phi、Mistral、DeepSeek、Whisper など、一般的な OSS モデルを端末内で使いたい。 モデル取得、キャッシュ、ハードウェア加速、OpenAI 互換 API、SDK からの in-process 推論、任意のローカルサーバー。 Windows / macOS Apple silicon / Linux。クライアントアプリ向けの単一ユーザー推論が中心。
    Windows ML 自分で選んだ、変換した、学習した ONNX モデルを Windows PC の CPU / GPU / NPU で直接走らせたい。 ONNX Runtime API、EP カタログ、モデルカタログ、Windows ML CLI、Foundry Toolkit、WebNN 経由のブラウザ推論。 Windows 10 以降、Windows Server 2019 以降、Windows 365。高度なカスタム推論向け。

    重要なのは、これらが競合ではなく積み重なっている点である。Windows AI APIs や Foundry Local は「すぐ使える」上位レイヤーで、 Windows ML はそれらを支える基盤であり、さらに独自モデルを持つ開発者が直接触る低レベル寄りの選択肢でもある。

    4. Windows AI APIs で何ができるか

    Windows AI APIs は、モデル探し・配布・最適化をほぼ意識せず、Windows アプリにローカル AI 機能を組み込むための API 群である。公式ドキュメントでは「Windows ML によって支えられた hardware-abstracted AI APIs」と説明されている。

    カテゴリ機能例用途
    言語Phi Silica、会話要約、テキスト要約、Text to Table、構造化出力、LoRA fine-tuning、Aion previewチャット、要約、業務入力の構造化、自然言語 UI、補助エージェント。
    音声・動画Speech Recognition API、Video Super Resolution音声入力、会議 / 通話の文字起こし、動画編集、低品質映像の改善。
    画像OCR、Image Super Resolution、Image Object Extractor、Image Description、Object Erase文書読み取り、画像説明、背景抽出、不要物除去、アクセシビリティ。
    検索・RAGApp Content Search / Semantic Search、RAG、コンテンツ安全性アプリ内ドキュメント検索、意味検索、ローカル知識ベース。

    Build 2026 で特に強調されたのは、当初 NPU / Copilot+ PC 中心だった API の一部を CPU と GPU にも拡大 していることだ。これにより、NPU 搭載 PC だけでなく、より広い Windows PC にローカル AI 体験を届けられる。ただし API ごとの対応状況、Windows App SDK のバージョン、地域制限は確認が必要である。

    5. Foundry Local で何ができるか

    Foundry Local は、端末上で OSS モデルを動かすためのエンドツーエンドのローカル AI ソリューションで、Build 2026 セッションでは GA と紹介された。公式ドキュメントでは、C#、JavaScript、Rust、Python SDK、最適化済みモデルカタログ、自動ハードウェア加速、OpenAI 互換 API を提供すると説明されている。

    モデル取得とキャッシュ

    初回利用時にモデルを取得し、端末内にキャッシュ。以後の起動を速くし、モデル管理をアプリから隠す。

    ハードウェア自動選択

    GPU / NPU があれば加速し、なければ CPU にフォールバック。開発者がデバイス判定コードを持たなくてよい。

    OpenAI 互換 API

    既存の OpenAI SDK をローカル endpoint に向け替えやすい。Responses API 形式にも対応。

    軽量ランタイム

    公式資料では runtime はアプリパッケージに約 20 MB 追加と説明される。

    セッションの在庫管理デモでは、Qwen 3.5 の vision-language model を使って、配送品の写真からアイテムを識別・説明した。コード上は、モデルを取得し、ダウンロードし、メモリにロードし、responses client で画像とプロンプトを渡すという流れで、複雑な EP 選定やモデル配置は Foundry Local 側が引き受ける。

    注意点: Foundry Local は「本番クライアントアプリに埋め込む軽量な単一ユーザー推論」に最適化されている。公式 FAQ でも、vLLM や Triton のような多ユーザー・サーバー推論基盤の置き換えではないと説明されている。

    6. Windows ML で何ができるか

    Windows ML を直接使う場面は、既製 API や Foundry Local のモデルカタログに収まらないときである。たとえば、自社で学習したモデル、Hugging Face から取得して fine-tune したモデル、PyTorch / TensorFlow / scikit-learn 由来のモデルを ONNX に変換して Windows 上で配布したい場合だ。

    6-1. 実行できるモデル

    • Windows ML は ONNX Runtime を基盤とするため、基本的には ONNX 形式 のモデルを扱う。
    • Hugging Face の ONNX モデル、Foundry Toolkit で ready-to-optimize なモデル、PyTorch / TensorFlow などから変換したモデル、自社で学習したモデルが候補になる。
    • モデルは CPU では追加設定なしに動き、必要に応じて NPU / GPU 向けの Execution Provider を追加する。
    • モデル互換性や性能は、ONNX Runtime バージョン、対応オペレーター、対象ハードウェア、量子化方式に左右される。

    6-2. ハードウェア加速

    シリコン主な EP向いている用途
    NPUOpenVINO（Intel）、QNN（Qualcomm）、VitisAI（AMD）省電力で継続的な端末内推論。Copilot+ PC の常時 AI 体験。
    GPUMIGraphX（AMD）、NvTensorRtRtx（NVIDIA）、OpenVINO（Intel）、QNN（Qualcomm）、DirectML（included / legacy）画像、動画、生成 AI、高スループット推論。
    CPUORT CPU EP（included）、OpenVINO（Intel）全端末へのフォールバック、小型モデル、低セットアップの初期導入。

    EP は「ハードウェアごとの最適化を ONNX Runtime に接続する部品」で、グラフ分割、カーネル登録、オペレーター実行を担う。Windows ML の特徴は、これらの EP をアプリが全部バンドルするのではなく、 Windows ML の ExecutionProviderCatalog API を通じて、Windows 認証済み EP を動的取得できる 点にある。これにより、デバイスごとのビルド分岐や大量のネイティブ依存関係を減らせる。

    6-3. 配布とアプリサイズ

    方式特徴向いているケース
    Framework-dependentWindows App SDK Runtime に依存。Windows ML バイナリをシステム側で共有し、アプリを小さくしやすい。Microsoft 推奨。Microsoft Store 配布、一般ユーザー向けアプリ、更新を OS / SDK 側に任せたい場合。
    Self-containedWindows ML バイナリをアプリに同梱。公式資料では onnxruntime.dll、DirectML.dll など合計約 41 MB。厳密なバージョン固定、管理端末、外部ランタイム依存を避けたい場合。

    6-4. モデルカタログ

    Windows ML には、巨大なモデルファイルをアプリに同梱せず、オンライン catalog から端末へダウンロードして共有する Model Catalog API もある。複数アプリが同じ SHA256 のモデルを必要とする場合は、端末内で重複ダウンロードを避けて共有できる。さらに、現在のハードウェアと EP に合うモデルだけを絞り込める。

    7. Windows ML CLI とモデル最適化

    Build 2026 の新発表として、Windows ML CLI が preview で紹介された。これは、モデルを Windows ML 向けに変換、解析、最適化、ベンチマークするためのワンストップのツールチェーンで、GitHub Copilot などのエージェントワークフローにも組み込みやすいよう agent skills を持つ。

    動画では Cardiff NLP の感情分析モデルを例に、winml catalog で候補を見つけ、inspect で適性を確認し、export で ONNX へ変換し、analyze で Qualcomm などターゲット EP の operator 対応状況を見ていた。緑は完全対応、黄は部分対応または CPU fallback、赤は非対応を示す。続いて optimize でグラフを書き換え、operator fusion を行い、再度 analyze で緑に近づけ、最後に perf で NPU 利用率や throughput を確認していた。

    重要な含意は、Windows ML が EP 配布を楽にしても、モデル最適化そのものは自動で完全解決されるわけではない という点である。公式ドキュメントも「Windows ML handles execution provider distribution, not model optimization」と明記している。CLI と Foundry Toolkit は、その最適化作業を人間とエージェントが扱いやすくする道具である。

    8. WebNN: Web アプリでも Windows ML を使う

    Windows ML はネイティブ Windows アプリだけの話ではない。セッションでは、Chromium 系ブラウザの WebNN preview を使い、ONNX Runtime Web から Windows ML ベースのハードウェア加速に乗るデモが行われた。

    WebNN は、Web アプリから GPU、CPU、NPU などの端末内 AI アクセラレータを利用するための Web 標準 API である。ONNX Runtime Web では execution provider に webnn を指定し、deviceType として cpu、gpu、npu を選ぶ。Windows ではその下に Windows ML が位置づく。

    要素役割
    ONNX Runtime WebJavaScript から ONNX モデルを実行する framework API。
    WebNN EPブラウザ内で WebNN を使う ONNX Runtime の execution provider。
    WebNN APIブラウザが端末内 AI アクセラレータを使う標準 API。
    Windows MLWindows 上で WebNN の下にあるネイティブ推論・加速基盤。
    現状: セッション時点では WebNN は preview で、Edge / Chrome の experimental flags が必要と説明された。ONNX Runtime Web の WebNN EP も活発に開発中で、operator 対応や deviceType の挙動はモデルとブラウザ実装に依存する。

    9. 過去情報: WinML / DirectML から 2026 年の Windows ML へ

    Windows ML の名前は以前から存在していたが、2026 年の文脈では役割がかなり再編されている。歴史を短く整理すると、次のようになる。

    時期・技術位置づけ現在の見方
    Windows Machine Learning / WinML ONNX モデルを Windows アプリに組み込む高レベル API。Load / Bind / Evaluate に近いモデル中心ワークフロー。 旧 docs は残るが、新しい Windows ML docs では ONNX Runtime ベースの統一基盤へ説明が移っている。
    DirectML DirectX 12 スタイルの低レベル ML API。GPU 上で operator や graph を実行し、ゲームやエンジンに深く統合できる。 公式 docs では「sustained engineering」。新規 feature 開発は Windows ベース ONNX Runtime deployment 向けの Windows ML に移動。
    ONNX Runtime + DirectML EP DirectML を直接書かず、ONNX Runtime から DirectML を EP として使う経路。 互換性とスケールの観点で、多くの開発者にはこの経路の方が現実的。
    Windows ML 2.0 Windows が保守する ORT、EP カタログ、Windows App SDK ML、モデルカタログ、CLI / Tooling を含む新しい中心軸。 2026 年時点の推奨理解。CPU / GPU / NPU にまたがる本番ローカル AI 配布の基盤。

    DirectML は今でも低レベル制御が必要なゲーム、エンジン、レンダリングパイプラインでは価値がある。一方で、一般的なアプリ開発者にとっては、DirectML を直接扱うより、ONNX Runtime / Windows ML / Windows AI APIs / Foundry Local を通じて使う方が、配布、互換性、更新、ハードウェア差分の扱いが楽になる。

    12. 制約と注意点

    • ローカル AI は万能ではない。 大規模推論、重いマルチユーザー処理、最新の最大モデルが必要な処理はクラウドや専用サーバーの方が向く。
    • モデル最適化は別問題。 Windows ML は EP 配布や実行基盤を助けるが、モデル変換、量子化、operator 対応、性能調整は開発者側の作業として残る。
    • NPU 最適化には条件がある。 ハードウェア最適化 EP の動的取得は Windows 11 24H2 以降や特定ドライバーを要求する場合がある。
    • API ごとに availability が違う。 Windows AI APIs は Windows App SDK バージョン、Copilot+ PC、地域、limited access feature の条件を確認する必要がある。
    • WebNN は preview。 ブラウザ flags、operator 対応、fallback、deviceType の挙動が変わりうる。
    • EP のライセンス確認が必要。 QNN、OpenVINO、NvTensorRtRtx、MIGraphX、VitisAI などは各ベンダーのライセンスとサポート条件を確認する。
