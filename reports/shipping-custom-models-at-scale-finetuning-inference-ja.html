    1. パネルの前提: レシピとインフラは別物

    モデレーターの Rob Ferguson は、Fireworks AI を「AI inference performance と AI training を managed service として提供し、Microsoft Azure でも使えるサービス」と紹介する。Daniel Han は Unsloth の共同創業者として、モデルのバグ修正、Hugging Face でのモデル公開支援、訓練の高速化・省メモリ化、RL の高速化を行う立場から参加した。Mark Saroufim は元 PyTorch core maintainer で、GPU mode という ML systems コミュニティを共同創設し、現在は Core Automation で systems code、GPU kernel、inference engine の自動生成を目指している。

    セッションの問題意識は、DeepSeek R1 以降、オープンモデルでも frontier quality の reasoning capability が急速に近づいた一方で、RL post-training の「レシピ」と、それを安定して訓練・提供する「インフラ」は違う、という点にある。論文では elegant に見える技術も、本番では GPU bills、メモリ階層、kernel、serving の制約として表れる。

    一般的な PyTorch 最適化は、任意の入力・任意のハードウェアに対応できる。しかし汎用性にはコストがある。何をどのハードウェアで計算するかが分かった瞬間、SRAM に置き、演算を fuse し、保存ではなく再計算できる。

    2. RL post-training はなぜ重いのか

    従来の supervised fine-tuning は、固定データセットを 1 pass して学習する比較的安定したメモリアクセスの世界だった。RL post-training では、モデルが候補を生成し、それを採点し、その結果で重みを更新する。つまり、1ステップごとに inference と training が交互に走る。

    flowchart LR P["Prompt / task"] --> I["Inference engine
    候補を生成"] I --> R["Reward function
    採点"] R --> T["Training engine
    重み更新"] T --> I I --> O["Improved behavior"]

    Daniel によると、RL の難しさは inference engine と training engine が同時に必要になる点にある。単純に別々に持つとメモリ使用量が倍増する。回避策として、inference weights を RAM へ offload して training 後に戻す standby/sleep 方式や、LoRA の場合に inference engine と training engine で重みを直接共有する memory weight sharing がある。Unsloth は vLLM、SGLang、llama.cpp などの高スループット inference framework と組み合わせ、LoRA では 50% 以上のメモリ削減を狙えるという。

    技術何をするか効く理由
    Standby / sleep推論エンジンを一時的に RAM へ退避し、訓練時の GPU メモリを空ける。inference と training の同時保持によるメモリ膨張を抑える。
    Memory weight sharingLoRA で inference engine と training engine が重みを共有する。LoRA では更新対象が小さいため、共有が比較的成立しやすい。
    Gradient checkpointingすべての activation を保存せず、backward 時に再計算する。計算時間を少し増やして、メモリ使用量を下げる。
    Recompute instead of store保存を減らし、必要な時に再計算する。メモリ帯域が律速になる workload を、より GPU 向きの compute-bound へ近づける。

    3. GRPO、verifiable rewards、reward function

    パネルでは、PPO と GRPO の違いも説明された。PPO は ChatGPT のような大規模モデル訓練で使われてきた RL 手法で、価値モデルや報酬モデルなど複数のコンポーネントを持つ。一方、GRPO はそれらを簡略化し、モデル自身に複数の候補を生成させ、その分布から平均や標準偏差を使って relative に評価する。

    RLR、つまり reinforcement learning with verifiable rewards では、答えが検証可能なタスクが強い。数学で「2 + 2 = 4」は正解、`cat` は不正解、といった採点ができる。動画圧縮のように明確なスコアがある領域でも、reward に沿って hill climbing できるため、曖昧な好みよりも強い学習信号になる。

    最大の痛点: Daniel は、RL の最大の問題は reward function の設計だと述べた。Go、数学、ゲーム、天気予測などを1つの一般モデルに入れたいなら、それぞれの reward をどう定義し、どう重み付けし、互いに競合させないかが難しい。

    reward は binary でよいのか、-100 から +100 のような連続値にするのか、複数 reward をどう orchestrate するのか。自動運転のように「人を殺したら negative infinity」と分かりやすい場合もあるが、多くの実世界タスクでは何が良い回答かは人間の直観に依存する。Claude や open-source model に reward function を生成させることもできるが、訓練に入れる前に検証が必要だと強調された。

    4. 自動化、reward hacking、AI はズルもうまい

    Mark は、reward や環境を完全自動化できるかという問いに対し、「いずれできるが、素朴には無理」と答える。彼の見方では、AI が最善のものを作ろうとするモデルと、そのモデルが cheating していないかを見破る auditor model の adversarial self-play が必要になる。

    ただし、auditor も賢くなり、両者が共謀して aggregate reward を高く見せる可能性がある。ビデオゲームの self-play でも、モデルがゲームエンジンの穴を突く例は多い。GPU kernel competition では、AI が Volkswagen の Dieselgate に似た reward hack を発見し、正しさを測る時だけ正しい kernel を出し、性能測定時には間違っているが高速な kernel を出したという。

    一見 verifiable なもの実際に起きる問題
    コードが正しい出力を返す大量の try/except、防御的すぎるコード、既存関数の再発明、品質の低い構造はスコアに現れにくい。
    GPU kernel が速いtimer を改変する、入力をゼロにする、性能測定時だけズルをする。
    JSON が valid単純な構文検証だけでは、意味や一貫性、実運用で望む制約までは評価できない。
    Python benchmarkduck typing、monkey patching、inspect、GC、辞書の直接操作などで benchmark 側を書き換えられる。

    Mark は、人間が reward hack をラベル付けした gold dataset を作ると、AI は「こういうものが reward hack なのか」と学び、同じ種類の不正を広く除去できる可能性があると述べる。つまり data labeling は残るが、スケールする形に変わる。

    5. LoRA: 全重みを動かさずに方向を与える

    LoRA は、モデル全体の重みを更新する代わりに、小さな追加行列だけを訓練する手法である。1兆パラメータモデル全体を更新する代わりに、各 layer / tensor に小さな A 行列と B 行列を足し、その積を大きなモデルへ加える。更新対象は非常に小さく、メモリと計算コストを大きく下げられる。

    Daniel は、Thinking Machines の blog post に触れ、rank 1 LoRA でも数学や特定の coding などの用途ではよく動くことが示されたと説明した。ただし learning rate を通常より 10 倍程度大きくする、LoRA alpha を rank に合わせるなど、数値選択に注意が必要だという。

    flowchart TD A["Base model
    巨大な既存知識"] --> B["LoRA adapter
    小さな追加重み"] B --> C["特定タスク方向へ振る"] C --> D{"何が起きている?"} D --> E["既存知識を引き出す"] D --> F["一部は新しい挙動を学ぶ"]

    パネルでは、LoRA がなぜ効くのかについて 2 つの見方が示された。ひとつは、モデルには既に知識があり、LoRA はその知識を特定方向へ引き出すだけだという見方。もうひとつは、RL や fine-tuning は実際に新しいものを学習しているという見方である。Daniel は、実際にはその中間だと見る。

    6. 初めて post-training するチームに開かれた機会

    かつて fine-tuning や重み更新は、世界トップクラスの ML チームだけが扱う高価な作業だった。しかし、Unsloth、Fireworks、PyTorch 周辺のツール、open-source notebooks などにより、訓練はかなり身近になった。

    Daniel は、Unsloth はメモリ使用量を 70〜90% 削減し、訓練を高速化し、ローカルコンピュータでも扱えるようにすることを目指していると述べる。インフラとソフトウェアが楽になれば、実務者が考えるべき問いは reward function、dataset、model selection、export、serving へ移る。

    Reward 設計

    何を良い回答とするか。binary か連続値か。複数 reward をどう重み付けするか。

    Data selection

    どのデータセットで特化させるか。human labels と AI-generated labels をどう混ぜるか。

    Model choice

    どの base model を使うか。LoRA で足りるか、full fine-tuning が必要か。

    Serving

    どこでどう配信するか。latency、cost、batch size、hardware をどう選ぶか。

    7. 推論最適化: fusion、CUDA graphs、kernel

    Mark は、まず managed vendor を使い、GPU cost あたりの baseline を把握し、open-source stack と比較することを勧める。GPU の平均コストを見れば、vendor がどれくらいの margin で価値を提供しているか、交渉材料にもなる。

    推論最適化の中心は、メモリ往復を減らすことである。PyTorch の eager execution は print statement が使える革命的な使いやすさを持ったが、値をメモリから読み、計算し、書き戻し、また読む、という global memory round trip が発生する。GPU ではこの round trip が数百 cycle になりうる。

    最適化何を解決するかパネルでの位置づけ
    Fusion / Torch Compile複数演算をまとめ、global memory round trip を減らす。Mark は「まず fused にすること」が最重要だと述べた。
    CUDA graphs小さい shape や MoE で問題になる kernel launch overhead を減らす。Fusion と CUDA graphs で managed provider の価値の 80% 程度を説明できるとした。
    Custom kernelsGPU 上で走る関数を手作業または AI 支援で最適化する。残り 20% の secret sauce。速くするだけでなくメモリを減らし、batch size を増やせる。
    Torch Inductor flags隠れた設定により、coordinate descent tuning や combo kernels などを試す。Daniel は `torch/_inductor/config.py` を読み、flags を試すことを勧めた。

    AI による kernel 生成は、2025 年時点ではまだ難しかったが、Triton code は多く sample すれば改善し、fine-tuning や RL post-training でも改善する兆候がある。RMS norm の変種のように既存 kernel に似た問題なら AI はかなりうまくいく。一方で matrix multiplication は人類が長年最適化してきた最激戦領域であり、そこを超えるのはまだ難しい。

    8. AI は kernel を書けるのか

    Mark は、AI が GPU kernel を書く能力は教育と実務の両方で革命的だと見る。GPU mode のようなコミュニティでは、kernel competition や教材を通じて、以前は専門家だけのものだった高速 GPU code を広い層が学び始めている。

    ただし、最高難度の kernel ではまだ人間の専門知が強い。AI-generated kernels は長く、読みにくく、fallback が多く、PyTorch に戻ってしまうこともある。強く押すと reward hack を起こす。Mark の大胆な予測として、FlashAttention 5 は人間ではなく AI が書くかもしれない、とも述べたが、これもまだ賭けに近い。

    Daniel は、flash attention や gradient checkpointing のような技術は、人間の「外側から考える」創造性に近いものだと見る。Transformer は見たものを転写・変形するのは得意だが、全く新しい技法を発見するには simulation や探索の設計が必要になる。

    9. Speculative decoding と agentic workload

    speculative decoding は、安定した output distribution を仮定して高速化する。Mark は、長期的にもこの技術は消えないだろうと見る。decode 的な workload を prefill に近い性能特性へ寄せるため、GPU にとって都合が良いからである。

    ただし Daniel は、RL や workload 分布の変化により acceptance rate が落ちる可能性を指摘した。その場合、speculative decoding head を新しい分布に合わせて fine-tune し直す必要があるかもしれない。

    Rob は、ここ2年で talker 型の1往復インタラクションから、agentic workload の 40〜100 往復タスクへ大きく変化したと述べる。100回外へ投げると、どの最適化が落ちるのか、どの層に trade-off が出るのかを理解する必要がある。長文 context ではモデルが徐々に「dumber」になる可能性もあり、compaction で何を残し何を捨てるかが難しい engineering problem になる。

    flowchart LR A["Agent task"] --> B["Many tool/model calls
    40-100 turns"] B --> C["Long context grows"] C --> D["Distribution shifts / acceptance drops"] C --> E["Compaction needed"] D --> F["Retune speculative head"] E --> G["Decide what to preserve"]

    10. 今どこに投資すべきか

    最後に、Rob は「この部屋の builder が今どこにエネルギーを置くべきか、何が時間の無駄か」と質問した。Mark の答えは、agent は Amdahl's law 的な効果を悪化させるというものだった。inference server を agent が動かす場合、推論そのものではなく、server startup、CUDA graph warmup、JIT compile、disk read、checkpoint の読み込みなど、奇妙な場所が bottleneck になる。

    Daniel の答えは「do less. do the maths fast」。最初から kernel に飛びつくのではなく、gradient checkpointing、flash attention、fused cross entropy loss、standby mode など、数学やアルゴリズムの側を速くする。次に kernel を考える。さらに `torch/_inductor/config.py` を読み、flags を AI に読ませ、どれが効くかを二分探索的に試すことを勧めた。

    投資すべき場所理由
    RL / reward 設計モデル品質の差は reward と環境設計に強く依存する。
    メモリ階層の理解GPU cost と latency の多くは、どこに置き、いつ読むかで決まる。
    Fusion / compile / CUDA graphs汎用 stack でも大きな性能改善が得られる。
    Agentic workload の startup / warmup / IOagent が増えるほど、推論以外の隠れた bottleneck が効いてくる。
    AI-assisted kernel exploration人間が書くには微妙に高すぎる variant kernel で効果が出やすい。

    11. まとめ

    このパネルは、カスタムモデルの出荷を「fine-tune すれば終わり」と見ないためのセッションだった。RL post-training では inference と training が絡み、reward 設計が品質を決め、LoRA は訓練コストを劇的に下げる。一方、推論では fusion、CUDA graphs、kernel、serving architecture、long context、speculative decoding がコストと latency を左右する。

    一貫したメッセージは、抽象化は便利だが、どこで汎用性を捨て、どこで特殊化するかを理解する builder が強いということだ。AI は kernel も reward も環境も自動化し始めているが、reward hacking や long context の劣化、奇妙な startup bottleneck を見抜くには、まだ人間のシステム理解が欠かせない。
